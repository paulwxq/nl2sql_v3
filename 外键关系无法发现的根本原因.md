# 外键关系无法发现的根本原因分析

## 用户指定的正确关联关系

1. **fault_catalog → maintenance_work_order**
   - 关联字段: `[product_line_code, subsystem_code, fault_code]` (3列)
   - 关系性质: 维度表 → 事实表 (外键关系)

2. **equipment_config → maintenance_work_order**
   - 关联字段: `[equipment_id, config_version]` (2列)
   - 关系性质: 维度表 → 事实表 (外键关系)

## 核心发现：误把外键当逻辑主键

### 问题本质

**当前系统的复合键候选生成逻辑，只支持"逻辑主键对逻辑主键"的匹配，不支持"逻辑主键对外键"的匹配！**

### 源表检查结果

#### 1. fault_catalog (源表)

```

  ❌ ['product_line_code', 'fault_name']  (conf=0.88, uniq=1.0)

目标组合:
  ❌ ['product_line_code', 'subsystem_code', 'fault_code'] - NOT FOUND
```

**问题**：逻辑主键检测器选择了错误的组合（应该是 3 列的业务键）

#### maintenance_work_order 的数据分布

**组合 1: [equipment_id, config_version]**
- 理论最大组合数: 10 × 2 = **20**
- 实际行数: **200**
- **uniqueness ≈ 20/200 = 0.1** (远小于 1.0)
- ❌ **结论**: 这个组合在事实表中**不是唯一的**（一对多关系）

**组合 2: [product_line_code, subsystem_code, fault_code]**
- 理论最大组合数: 3 × 3 × 4 = **36**
- 实际行数: **200**
- **uniqueness ≈ 36/200 = 0.18** (远小于 1.0)
- ❌ **结论**: 这个组合在事实表中**不是唯一的**（一对多关系）

## 根本原因总结

### 🔴 原因 1: 逻辑主键检测器的设计缺陷

**问题**：逻辑主键检测器只生成 **uniqueness = 1.0** 的组合

**影响**：
- 事实表的外键列组合 uniqueness < 1.0（因为是一对多关系）
- 因此**不会被作为逻辑主键候选**
- 结果：目标表的逻辑主键候选中缺失这些外键组合

**代码位置**: `src/metaweave/core/metadata/logical_key_detector.py`

---

### 🔴 原因 2: 复合键匹配只支持"逻辑主键对逻辑主键"

**当前匹配逻辑**（`candidate_generator.py:243-282`）：

```python
# Stage 1: 特权模式（Privilege Mode）
if combo_type in ["physical", "logical"]:
    # 收集目标表的约束组合
    target_combinations = self._collect_source_combinations(
        target_table,
        include_indexes=include_target_indexes
    )

    # 遍历目标表的所有约束组合
    for target_combo in target_combinations:
        target_cols = target_combo["columns"]
        # 长度必须相等
        if len(target_cols) != len(source_columns):
            continue
        # 尝试匹配
        matched = self._match_columns_as_set(...)
```

**问题**：
- `_collect_source_combinations` 只收集 **物理约束（PK/UK/Index）** 和 **逻辑主键候选**
- **不收集纯外键组合**（即使这些列在源表是主键）
- 因此无法匹配"维度表逻辑主键 → 事实表外键"的关系

---

### 🔴 原因 3: fault_catalog 的逻辑主键检测错误

**检测结果**: `['product_line_code', 'fault_name']` (2列)
**正确业务键**: `['product_line_code', 'subsystem_code', 'fault_code']` (3列)

**问题**：
- 逻辑主键检测器选择了更少列的组合（偏好最小候选）
- 但真实的业务键是 3 列组合

---

## 对比：为什么单列关系能发现？

### 成功案例（从日志中）

```
✅ dim_company[company_id] → dim_store[company_id]
✅ dim_product_type[product_type_id] → fact_store_sales_day[product_type_id]
✅ dim_store[store_id] → fact_store_sales_day[store_id]
```

**为什么成功**？

1. **源列有物理约束或逻辑主键**
   ```python
   # candidate_generator.py:795-815
   has_defined_constraint = self._has_defined_constraint(col_profile)
   is_logical_pk = self._is_logical_primary_key(col_name, source_table)

   if not (has_defined_constraint or is_logical_pk):
       continue  # 源列必须是关键列
   ```

2. **目标列只需要存在，不需要是逻辑主键**
   ```python
   # candidate_generator.py:827-862
   # 遍历目标表的所有列（不限于逻辑主键候选）
   for target_col_name, target_col_profile in target_profiles.items():
       # 只要列名匹配 + 类型兼容，就可以匹配
   ```

**关键区别**：
- 单列候选：**源列是关键列 → 目标列只需存在**
- 复合键候选：**源列组合是逻辑主键 → 目标列组合也必须是逻辑主键**

## 为什么复合键无法匹配外键？

### 当前的复合键匹配流程

```
1. 源表: equipment_config
   - 逻辑主键: [equipment_id, config_version], type="logical"
   ↓
2. 目标表: maintenance_work_order
   - Stage 1: 收集目标表的逻辑主键候选（18个）
   - 查找: [equipment_id, config_version]
   - ❌ 未找到（因为这个组合不是逻辑主键，是外键）
   ↓
3. Stage 2: 动态同名匹配
   - ❌ 跳过（因为 combo_type="logical"，不是 "physical"）
   ↓
4. 结果: ❌ 未找到匹配
```

**问题所在**：
- Stage 1 要求目标列组合也是逻辑主键候选（uniqueness=1.0）
- Stage 2 只对源表物理约束执行，逻辑主键不执行

**结果**：
- **维度表逻辑主键 → 事实表外键** 的关系无法被发现
- 只能发现 **维度表逻辑主键 → 维度表逻辑主键** 的关系

---

## 解决方案

### 方案 1: 扩展 Stage 2 到逻辑主键（推荐）

**修改**: `src/metaweave/core/relationships/candidate_generator.py:290`

```python
# 当前逻辑
if combo_type == "physical":
    # 只对物理约束执行 Stage 2

# 改为
if combo_type in ["physical", "logical"]:  # ← 扩展到逻辑主键
    logger.debug(
        "[find_target_columns] Stage 2: 动态同名匹配（物理约束 + 逻辑主键）"
    )

    matched = self._find_dynamic_same_name(
        source_columns,
        source_table,
        target_table,
        is_physical=(combo_type == "physical")
    )
```

**原理**：
- 允许逻辑主键使用动态同名匹配
- `equipment_config[equipment_id, config_version]` 可以直接匹配到 `maintenance_work_order` 中同名的列
- 不再要求目标列组合必须是逻辑主键候选

**优点**：
- 简单直接，只需修改 1 行代码
- 立即生效，不需要重新生成元数据

**缺点**：
- 可能引入更多低质量候选（需要通过评分阈值过滤）

---

### 方案 2: 修改 `_find_dynamic_same_name` 的过滤逻辑

**当前问题**: `_find_dynamic_same_name` 方法在匹配时，可能仍然会过滤目标列

**修改**: `src/metaweave/core/relationships/candidate_generator.py:473-525`

确保当 `is_physical=False` (逻辑主键) 时，目标列也不进行语义角色过滤：

```python
def _find_dynamic_same_name(
    self,
    source_columns: List[str],
    source_table: dict,
    target_table: dict,
    is_physical: bool = False
) -> Optional[List[str]]:
    # ...

    # 目标表：源表为逻辑主键时，目标表也不过滤
    for col_name, col_profile in target_profiles.items():
        semantic_role = col_profile.get("semantic_analysis", {}).get("semantic_role")

        # 新增：如果源表是逻辑主键，目标表也不过滤语义角色
        if is_physical or True:  # ← 暂时完全不过滤
            target_column_map[col_name.lower()] = col_name
```
