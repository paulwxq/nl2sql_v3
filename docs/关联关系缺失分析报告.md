# 三张表关联关系缺失分析报告

## 1. 问题概述

以下三张表之间存在明显的业务关联关系，但系统**未能发现或生成任何关联候选**：

1. **equipment_config** (设备配置维度表)
2. **maintenance_work_order** (维修工单事实表)
3. **fault_catalog** (故障码字典维度表)

### 预期的关联关系：

1. **equipment_config → maintenance_work_order**
   - 关联字段: `[equipment_id, config_version]`
   - 关系类型: 2列复合键外键关系
   - 业务含义: 工单关联到设备的特定配置版本

2. **fault_catalog → maintenance_work_order**
   - 关联字段: `[product_line_code, subsystem_code, fault_code]`
   - 关系类型: 3列复合键外键关系
   - 业务含义: 工单关联到故障码字典的标准故障定义

## 3. 根本原因总结

### 3.1 核心问题 1: 语义角色过滤过于严格

**问题代码位置**: `src/metaweave/core/relationships/candidate_generator.py`

```python
# Line 844-859: 目标列过滤逻辑
if target_has_physical:
    # 物理约束：完全不过滤
    pass
else:
    # 逻辑约束：按配置排除
    if target_role in self.exclude_semantic_roles:  # ❌ 这里排除了 audit
        logger.debug(
            "[single_column_candidate] 跳过目标列 %s，语义角色=%s 被排除",
            target_col_name, target_role
        )
        continue  # 跳过该列
```

**配置**: `configs/metaweave/metadata_config.yaml`

```yaml
single_column:
  exclude_semantic_roles:
    - audit      # ❌ 过滤掉 config_version, fault_ts 等关键外键列
    - metric     # ✅ 合理（度量字段确实不应作为外键）
    - description # ⚠️ 可能过严
```

**影响**:
- `maintenance_work_order.config_version` (audit 角色) 被排除
- 导致 `equipment_config → maintenance_work_order` 的 2列关系无法匹配

### 3.2 核心问题 2: 逻辑主键检测不准确

**问题**: `fault_catalog` 的逻辑主键检测结果与业务键不符

| 维度 | 检测结果 | 真实业务键 |
|------|---------|-----------|
| 列组合 | `[product_line_code, fault_name]` | `[product_line_code, subsystem_code, fault_code]` |
| 列数 | 2列 | 3列 |
| 唯一性 | 1.0 (完美) | 应该是 1.0 |

**原因分析**:

查看 JSON 数据:

```json
// fault_catalog 逻辑主键候选
"candidate_primary_keys": [
  {
    "columns": ["product_line_code", "fault_name"],
    "confidence_score": 0.88,
    "uniqueness": 1.0,
    "null_rate": 0.0
  }
]
```

**问题所在**:
- 系统选择了 `[product_line_code, fault_name]` 因为这个组合的唯一性为 1.0
- 但真实的业务键应该是 `[product_line_code, subsystem_code, fault_code]` (3列)
- **逻辑主键检测器偏好选择更少的列，而不是语义上更合理的业务键**

### 3.3 核心问题 3: 复合键候选生成数为 0（12月4日）

对比历史运行：

| 日期 | equipment_config 逻辑主键 | fault_catalog 逻辑主键 | 复合键候选数 |
|------|--------------------------|----------------------|-------------|
| 12月2日 | 3个 | ? | 1个 |
| 12月4日 | 3个 | 1个 | **0个** |

**可能原因**:
1. **配置变化**: 单列 `exclude_semantic_roles` 配置可能在12月4日之前修改过
2. **代码变化**: 复合键匹配逻辑可能有修改（特别是语义角色过滤部分）
3. **数据变化**: 元数据生成结果可能略有不同

## 4. 解决方案建议

### 方案 1: 调整语义角色过滤策略（推荐）

**修改**: `configs/metaweave/metadata_config.yaml`

```yaml
single_column:
  exclude_semantic_roles:
    - metric        # ✅ 保留：度量字段确实不应作为外键
    # - audit       # ❌ 移除：audit 字段可能是外键（如 config_version, fault_ts）
    # - description # ❌ 移除：description 字段可能是外键（如果有物理约束）
```

**优点**:
- 简单直接，立即生效
- 允许 `config_version` 等 audit 字段参与关联匹配

**缺点**:
- 可能引入噪音（不相关的 audit 字段也会参与匹配）

---

### 方案 2: 修正 fault_catalog 的逻辑主键检测

**问题根源**: 逻辑主键检测器选择了 `[product_line_code, fault_name]`，而不是真实的业务键 `[product_line_code, subsystem_code, fault_code]`

**解决方法 A**: 手动添加逻辑主键候选

修改 `output/metaweave/metadata/json/public.fault_catalog.json`:

```json
"candidate_primary_keys": [
  {
    "columns": ["product_line_code", "fault_name"],
    "confidence_score": 0.88,
    "uniqueness": 1.0,
    "null_rate": 0.0
  },
  // 新增真实业务键
  {
    "columns": ["product_line_code", "subsystem_code", "fault_code"],
    "confidence_score": 0.90,
    "uniqueness": 1.0,
    "null_rate": 0.0,
    "is_business_key": true  // 标记为业务键
  }
]
```

**解决方法 B**: 优化逻辑主键检测算法

修改 `src/metaweave/core/metadata/logical_key_detector.py`，增加启发式规则：
- 对于包含 `code` 后缀的列组合，优先考虑
- 对于维度表，优先选择全部 identifier 列的组合

---

### 方案 3: 降低关联关系评分阈值（临时方案）

**修改**: `configs/metaweave/metadata_config.yaml` (假设有这个配置)

```yaml
relationship_scoring:
  min_score_threshold: 0.60  # 从 0.80 降低到 0.60
```

**优点**:
- `equipment_config → maintenance_work_order` (score=0.675) 可以通过

**缺点**:
- 会引入更多低质量的关联关系
- 治标不治本

---

### 方案 4: 优化复合键匹配逻辑（长期方案）

**问题**: 目标列过滤逻辑过于简单，未考虑外键的特殊性

**建议修改**: `src/metaweave/core/relationships/candidate_generator.py`

```python
# 当前逻辑（Line 844-859）
if target_has_physical:
    pass  # 物理约束不过滤
else:
    if target_role in self.exclude_semantic_roles:
        continue  # ❌ 直接跳过

# 改进逻辑
if target_has_physical:
    pass  # 物理约束不过滤
else:
    # 新增：检查列名是否与源列匹配
    if self._is_likely_foreign_key(target_col_name, source_columns):
        # 即使语义角色是 audit，但列名高度匹配，也应该保留
        pass
    elif target_role in self.exclude_semantic_roles:
        continue  # 非外键候选，按语义角色过滤

def _is_likely_foreign_key(self, target_col: str, source_cols: List[str]) -> bool:
    """检查目标列是否可能是源列的外键"""
    for src_col in source_cols:
        if src_col.lower() == target_col.lower():  # 完全同名
            return True
        # 可以增加更多启发式规则
    return False
```
