- **版本**: v1.0

## 1. 设计目标与范围

### 1.1 Phase 2 目标

实现父图右侧的 **Complex Path** 流程，为复杂问题提供智能拆分与多步骤执行：

```
用户问题 → Router → Planner → Inject Params → SQL生成子图
                                     ↓
                            Check完成度 ← SQL执行
                                ↓
                           [循环] or Summarizer → 返回结果
```

**核心功能**：
1. **问题拆分**：将复杂问题拆分为 2-5 个子查询
2. **依赖管理**：建立子查询之间的依赖关系图
3. **参数注入**：将前序查询结果注入到后续查询
4. **并发执行**：无依赖的子查询并发执行（max_concurrency=3）
5. **循环控制**：多轮执行 + 最大轮次保护
6. **智能汇总**：LLM 综合多个 SQL 结果生成总结

### 1.2 典型问题示例

| 问题类型 | 示例 | 子查询数 | 依赖关系 |
|---------|------|---------|---------|
| **依赖型** | "哪个服务区销售最高？它的地址和公司是？" | 2 | sq1（查最高销售区）→ sq2（查详细信息） |
| **钻取型** | "找出车流量前3的服务区，分别查询它们的营业额" | 2 | sq1（查前3）→ sq2（查各自营业额） |
| **多维度** | "2024年各区域销售趋势，对比同比增长" | 3 | sq1（查2024数据）→ sq2（查2023数据）, sq3（计算对比） |

**判定标准**（由 Router 完成）：
- ✅ **Complex**：必须根据第一次查询结果动态决定后续查询
- ✅ **Complex**：多个独立问题需要分别查询
- ❌ **Simple**：可以用一条 SQL（包括 JOIN、CTE、子查询）解决

### 1.3 与 Phase 1 的关系

#### 共享节点（需增强，保持兼容）

| 节点 | Phase 1 行为 | Phase 2 增强 | 兼容性设计 |
|------|-------------|-------------|----------|
| **Router** | 判定 simple/complex | 不变 | ✅ 无需修改 |
| **SQL 执行** | 串行执行 1 个 SQL | 并发执行多个 SQL | ✅ 自动检测：`len(sqls) == 1` → 串行 |
| **Summarizer** | 单 SQL 总结 | 多 SQL 汇总 | ✅ 自动检测：`len(results)` 决定模板 |

#### 独有节点（Phase 2 新增）

| 节点 | 职责 | 复杂度 |
|------|------|--------|
| **Planner** | 问题拆分 + 依赖建立 | ⭐⭐⭐⭐⭐ |
| **Inject Params** | 依赖解析 + 参数注入 | ⭐⭐⭐ |
| **Check 完成度** | 循环控制 + 异常检测 | ⭐⭐⭐ |

### 1.4 设计原则

1. **最大化复用**：SQL 执行、Summarizer 节点两路共享，避免重复开发
2. **零影响 Fast Path**：所有共享节点修改都要兼容 Phase 1，Fast Path 测试用例无需修改
3. **清晰边界**：Phase 2 新增字段使用 `Optional` 类型，Fast Path 不填充
4. **可控复杂度**：限制子查询数（2-5 个）、最大轮次（5 轮），防止失控
5. **失败友好**：任何节点失败都能优雅降级，返回友好提示

### 1.5 不包含的功能（Phase 3 规划）

本阶段（Phase 2.0 + 2.1）**不包含**以下高级特性，留待 Phase 3 实现：

❌ **缓存策略**：相同 SQL 不重复执行
❌ **自适应 max_rounds**：根据问题复杂度动态调整
❌ **LLM 智能参数提取**：Phase 2 使用简单传递，Phase 3 用 LLM 提取关键字段
❌ **执行计划优化**：拓扑排序优化、关键路径分析等
❌ **事务支持**：多 SQL 原子性保证

### 1.6 Phase 2 分阶段实施（已合并）

**原计划**：
- Phase 2.0: 基础循环（串行执行）
- Phase 2.1: 并发优化

**实际实施**：
- **Phase 2（本文档）**：直接实现基础循环 + 并发执行，一次性完成
- **Phase 3（后续）**：高级特性（缓存、智能提取等）

### 2.1 Complex Path 完整拓扑

```
┌─────────────────────────────────────────────────────────┐
│                      START                              │
└────────────────────┬────────────────────────────────────┘
                     │
                     ▼
            ┌────────────────┐
            │     Router     │ ◄─── 判定 complexity
            └────┬───────┬───┘
                 │       │
        simple   │       │ complex
                 │       │
                 ▼       ▼
    [Fast Path]     ┌──────────────┐
                    │   Planner    │ ◄─── 问题拆分 + 依赖建立
                    │ (Agent LLM)  │
                    └──────┬───────┘
                           │
                           ▼
                    ┌──────────────┐
                    │Inject Params │ ◄─── 参数注入（第1轮：准备无依赖子查询）
                    │(纯函数逻辑)   │
                    └──────┬───────┘
                           │
                           ▼
                    ┌──────────────┐
                    │ SQL生成子图   │ ◄─── 批量调用（当前轮次的子查询）
                    │(已有,复用)    │
                    └──────┬───────┘
                           │
                           ▼
                    ┌──────────────┐
                    │  SQL执行     │ ◄─── 并发执行（max_concurrency=3）
                    │(共享,增强)    │
                    └──────┬───────┘
                           │
                           ▼
                    ┌──────────────┐
                    │Check完成度    │ ◄─── 检查是否所有子查询完成
                    └──┬───────┬───┘
                       │       │
                  未完成│       │完成
                       │       │
                       │       ▼
                       │   ┌──────────────┐
                       │   │ Summarizer   │ ◄─── LLM 多结果汇总
                       │   │(共享,已支持)  │
                       │   └──────┬───────┘
                       │          │
                       │          ▼
                       │      ┌───────┐
                       │      │  END  │
                       │      └───────┘
                       │
                       └─────┐ (循环边)
                             │
                             ▼
                      ┌──────────────┐
                      │Inject Params │ ◄─── 注入依赖结果，准备下一轮
                      └──────────────┘
                             │
                            ...（循环）
```

### 2.2 数据流转说明

#### 第1轮执行
1. **Planner** 输出：
   ```python
   {
       "sub_queries": [
           {"sub_query_id": "sq1", "query": "...", "dependencies": []},
           {"sub_query_id": "sq2", "query": "...", "dependencies": ["sq1"]},
       ],
       "dependency_graph": {...},
       "current_round": 1,
       "max_rounds": 5,
   }
   ```

2. **Inject Params** 输出：
   ```python
   {
       "current_batch_ids": ["sq1"],  # 第1轮：只有 sq1 无依赖
   }
   # 更新 sub_queries[0].status = "in_progress"
   ```

3. **SQL 生成子图**：调用子图生成 sq1 的 SQL
4. **SQL 执行**：执行 sq1，结果写入 `sub_queries[0].execution_result`
5. **Check 完成度**：sq2 还未执行 → 返回 "inject_params"

#### 第2轮执行
1. **Inject Params** 输出：
   ```python
   {
       "current_batch_ids": ["sq2"],
       "current_round": 2,
   }
   # 将 sq1 的结果注入到 sq2.dependencies_results
   # 更新 sq2.status = "in_progress"
   ```

2. **SQL 生成子图**：调用子图生成 sq2 的 SQL（利用 sq1 的结果）
3. **SQL 执行**：执行 sq2
4. **Check 完成度**：所有子查询完成 → 返回 "summarizer"
5. **Summarizer**：LLM 综合 sq1 + sq2 的结果，生成总结

### 2.3 循环控制机制

#### 轮次计数
- **current_round**：从 1 开始，每次循环 +1
- **max_rounds**：默认 5，配置参数

#### 退出条件（3种）
1. **正常完成**：所有子查询 status 为 `completed` 或 `failed`
2. **最大轮次保护**：`current_round >= max_rounds` → 强制退出
3. **依赖环检测**：检测到循环依赖 → 提前退出，返回错误

#### 异常处理
- **中间 SQL 生成失败**：子查询标记为 `failed`，继续执行其他子查询
- **所有子查询都失败**：Check 完成度检测到 → 进入 Summarizer，返回友好提示

### 3.2 SubQueryInfo 扩展（Phase 1 已定义，Phase 2 扩展）

```python
class SubQueryInfo(TypedDict):
    """子查询信息（Phase 1 + Phase 2）

    注意：Phase 1 不设置 dependencies_results 字段（运行时安全，使用 get() 访问）
    """
    sub_query_id: str                               # 子查询ID（如 "sq1", "sq2"）
    query: str                                      # 子查询文本
    status: str                                     # pending/in_progress/completed/failed
    dependencies: List[str]                         # 依赖的 sub_query_id 列表
    dependencies_results: Optional[Dict[str, Any]]  # Phase 2 新增：依赖结果字典（详见下文）
    validated_sql: Optional[str]                    # 生成的SQL
    execution_result: Optional[Dict[str, Any]]      # 执行结果（SQLExecutionResult）
    error: Optional[str]                            # 错误信息
    iteration_count: int                            # 迭代次数
```

**Phase 1 vs Phase 2 字段使用差异**：

| 字段 | Phase 1 | Phase 2 |
|------|---------|---------|
| `dependencies` | `[]`（固定为空列表） | `["sq1", "sq2"]`（Planner 填充） |
| `dependencies_results` | **不设置**（字段不存在） | `{"sq1": {...}}`（Inject Params 注入） |
| `execution_result` | 单次写入（SQL 执行后） | 可能多次更新（每轮执行后） |

**dependencies_results 字段详解**（Phase 2 新增）：

- **类型**: `Optional[Dict[str, SQLExecutionResult]]`
- **Phase 1 行为**:
  - Simple Planner **不设置**这个字段（字段在字典中不存在）
  - 下游代码使用 `sub_query.get("dependencies_results", {})` 安全访问
  - 返回空字典 `{}`，不影响流程

- **Phase 2 行为**:
  - Planner 初始化时**不设置**（字段不存在）
  - Inject Params 在准备子查询时**注入**这个字段
  - 数据结构示例：
    ```python
    {
        "sq1": {  # Key: 依赖的 sub_query_id
            "sub_query_id": "sq1",
            "sql": "SELECT store_id FROM ... LIMIT 1",
            "success": True,
            "columns": ["store_id"],
            "rows": [[101]],
            "row_count": 1,
            "execution_time_ms": 45.2,
            "error": None,
            # ... 完整的 SQLExecutionResult
        },
        "sq2": {...}
    }
    ```

- **用途**: SQL 生成子图读取此字段，将依赖结果传递给 LLM，用于生成依赖于前序查询的 SQL

**兼容性保证**：
```python
# Phase 1 & Phase 2 通用的安全访问方式
dependencies_results = sub_query.get("dependencies_results", {})

# 运行时行为：
# - Phase 1: 字段不存在 → get() 返回默认值 {} ✅
# - Phase 2: 字段存在 → get() 返回实际的依赖结果字典 ✅
```

### 3.3 依赖图结构设计

```python
{
    "nodes": ["sq1", "sq2", "sq3"],                 # 所有子查询ID
    "edges": [                                      # 依赖关系边
        {"from": "sq1", "to": "sq2"},               # sq2 依赖 sq1
        {"from": "sq1", "to": "sq3"}                # sq3 依赖 sq1
    ]
}
```

**字段说明**：
- **nodes**：所有子查询ID列表（用于环检测）
- **edges**：依赖关系边列表（`from` → `to` 表示 `to` 依赖 `from`）

**用途**：

1. **Planner 内部环检测**（唯一的运行时消费者）：
   - 在问题拆分后立即检测依赖环（调用 `_has_cycle(dependency_graph)`）
   - 如果检测到环，Planner 返回错误，不进入循环执行
   - 提前发现问题，避免进入循环后才发现无法推进

2. **API 诊断字段**（暴露给外部监控）：
   - 可视化依赖关系（前端可以绘制依赖图）
   - 分析问题复杂度（节点数、边数、依赖链深度）
   - 监控系统可以基于图结构计算复杂度指标

3. **运行时节点不读取此字段**：
   - **Inject Params**：动态遍历 `sub_queries`，检查每个 `sub_query["dependencies"]` 是否满足
   - **Check Completion**：运行时动态检测依赖环（检查"有 pending 但没有 ready_to_execute"）
   - 优点：能灵活处理子查询失败导致的依赖关系变化，无需依赖静态图

**设计说明**：
- `dependency_graph` 在 Planner 构建后成为**只读诊断数据**，不参与后续循环控制逻辑
- 运行时节点使用 `sub_queries[*].dependencies` 动态检查，更灵活、更准确
- Planner 的静态环检测 + Check Completion 的动态环检测 = 双重保障

2. ✅ **dict.get() 容错**：字段不存在时返回默认值，不会报错
3. ✅ **逻辑等价**：`get("key", {})` 返回 `{}`，与 Phase 1 预期行为一致
4. ✅ **向后兼容**：Phase 1 代码和流程完全不变

## 4. Planner 节点设计（核心）

### 4.1 职责定位

Planner 是 **Complex Path 的核心节点**，负责将复杂问题拆分为可执行的子查询序列。

**核心职责**：
1. **问题意图理解**：分析复杂问题的真实需求
2. **问题拆分**：拆分为 2-5 个子查询
3. **依赖关系建立**：明确哪些子查询依赖其他子查询的结果
4. **生成依赖图**：构建 `dependency_graph` 结构
5. **初始化循环控制**：设置 `current_round = 1`, `max_rounds = 5`

**不做**：
- ❌ 不生成 SQL（由 SQL 生成子图负责）
- ❌ 不执行查询（由 SQL 执行节点负责）
- ❌ 不注入参数（由 Inject Params 负责）

### 4.2 LLM 提示词策略

#### 模型选择
- **模型**：`qwen-max`（强模型，理解能力强）
- **温度**：`temperature=0`（确定性输出）
- **超时**：`timeout=15s`

import json
import time
from typing import Dict, Any
from langchain_community.chat_models import ChatTongyi
from src.services.config_loader import load_config
from src.utils.logger import get_module_logger, with_query_id

logger = get_module_logger("planner")

# 配置缓存
_planner_config_cache = None


def _get_planner_config() -> Dict[str, Any]:
    """获取 Planner 配置（带缓存）"""
    global _planner_config_cache
    if _planner_config_cache is None:
        config_path = "src/modules/nl2sql_father/config/nl2sql_father_graph.yaml"
        full_config = load_config(config_path)
        _planner_config_cache = full_config["planner"]
    return _planner_config_cache


def planner_node(state: NL2SQLFatherState) -> Dict[str, Any]:
    """
    Planner 节点：问题拆分 + 依赖建立

    职责：
    1. 调用 LLM 将复杂问题拆分为 2-5 个子查询
    2. 建立依赖关系图
    3. 初始化循环控制字段

    Args:
        state: 父图 State

    Returns:
        更新的 State 字段：
        - sub_queries: 子查询列表
        - dependency_graph: 依赖图
        - current_round: 1
        - max_rounds: 5
    """
    query_logger = with_query_id(logger, state.get("query_id", "unknown"))
    user_query = state["user_query"]
    query_id = state["query_id"]

    # 加载配置
    config = _get_planner_config()
    model_name = config["model"]
    temperature = config["temperature"]
    timeout = config["timeout"]
    max_sub_queries = config["max_sub_queries"]

    query_logger.info("Planner 开始问题拆分")

    # 构造提示词
    prompt = PLANNER_PROMPT.format(user_query=user_query)

    # 调用 LLM
    start_time = time.time()
    llm = ChatTongyi(
        model=model_name,
        temperature=temperature,
        timeout=timeout,
    )

    try:
        response = llm.invoke(prompt)
        content = response.content.strip()

        # 解析 JSON
        # 去除可能的 markdown 代码块标记
        if content.startswith("```"):
            content = content.split("```")[1]
            if content.startswith("json"):
                content = content[4:]

        result = json.loads(content)
        sub_queries_raw = result.get("sub_queries", [])

        # 验证子查询数量
        if len(sub_queries_raw) > max_sub_queries:
            query_logger.warning(f"子查询数量 {len(sub_queries_raw)} 超过限制 {max_sub_queries}，截断")
            sub_queries_raw = sub_queries_raw[:max_sub_queries]

        if len(sub_queries_raw) < 2:
            query_logger.error("Planner 失败：子查询数量少于2个")
            return {
                "error": "Planner 失败：复杂问题至少需要2个子查询",
                "error_type": "planning_failed",
            }

        # 转换为 SubQueryInfo 格式
        sub_queries = []
        for sq_raw in sub_queries_raw:
            sub_query = {
                "sub_query_id": f"{query_id}_{sq_raw['sub_query_id']}",  # 添加前缀避免冲突
                "query": sq_raw["query"],
                "status": "pending",
                "dependencies": [
                    f"{query_id}_{dep_id}" for dep_id in sq_raw.get("dependencies", [])
                ],
                "validated_sql": None,
                "execution_result": None,
                "error": None,
                "iteration_count": 0,
            }
            sub_queries.append(sub_query)

        # 构建依赖图
        dependency_graph = _build_dependency_graph(sub_queries)

        # 检测依赖环
        if _has_cycle(dependency_graph):
            query_logger.error("Planner 失败：检测到依赖环")
            return {
                "error": "Planner 失败：子查询存在循环依赖",
                "error_type": "planning_failed",
            }

        latency_ms = (time.time() - start_time) * 1000
        query_logger.info(f"Planner 完成：{len(sub_queries)} 个子查询，耗时 {latency_ms:.0f}ms")

        return {
            "sub_queries": sub_queries,
            "dependency_graph": dependency_graph,
            "current_round": 1,
            "max_rounds": config.get("max_rounds", 5),
        }

    except json.JSONDecodeError as e:
        query_logger.error(f"Planner 失败：JSON 解析错误 - {str(e)}")
        return {
            "error": f"Planner 失败：LLM 输出格式错误",
            "error_type": "planning_failed",
        }

    except Exception as e:
        query_logger.error(f"Planner 失败：{str(e)}", exc_info=True)
        return {
            "error": f"Planner 失败：{str(e)}",
            "error_type": "planning_failed",
        }


def _build_dependency_graph(sub_queries: List[Dict]) -> Dict[str, Any]:
    """
    构建依赖图

    Args:
        sub_queries: 子查询列表

    Returns:
        依赖图结构（包含 nodes 和 edges，用于环检测）
    """
    nodes = [sq["sub_query_id"] for sq in sub_queries]
    edges = []

    for sq in sub_queries:
        for dep_id in sq["dependencies"]:
            edges.append({"from": dep_id, "to": sq["sub_query_id"]})

    return {
        "nodes": nodes,
        "edges": edges,
    }


def _has_cycle(dependency_graph: Dict[str, Any]) -> bool:
    """
    检测依赖图中是否存在环（使用 DFS）

    Args:
        dependency_graph: 依赖图

    Returns:
        True 表示有环，False 表示无环
    """
    from collections import defaultdict

    nodes = dependency_graph["nodes"]
    edges = dependency_graph["edges"]

    # 构建邻接表
    adj = defaultdict(list)
    for edge in edges:
        adj[edge["from"]].append(edge["to"])

    # DFS 检测环
    visited = set()
    rec_stack = set()  # 递归栈，用于检测环

    def dfs(node):
        visited.add(node)
        rec_stack.add(node)

        for neighbor in adj[node]:
            if neighbor not in visited:
                if dfs(neighbor):
                    return True
            elif neighbor in rec_stack:
                # 发现环
                return True

        rec_stack.remove(node)
        return False

    # 对所有节点执行 DFS
    for node in nodes:
        if node not in visited:
            if dfs(node):
                return True

    return False

## 5. Inject Params 节点设计

### 5.1 职责定位

Inject Params 是 **循环控制的核心节点**，负责在每轮执行前准备待执行的子查询。

**核心职责**：
1. **依赖检查**：检查哪些子查询的依赖已满足
2. **结果提取**：从已完成的子查询中提取 `execution_result`
3. **参数注入**：将结果注入到待执行子查询的 `dependencies_results`
4. **状态更新**：将待执行子查询的 `status` 更新为 `in_progress`
5. **批量准备**：更新 `current_batch_ids`（当前轮次待执行列表）

**Phase 2 简化策略**：
- ✅ **简单传递**：直接传递完整的 `execution_result`
- ❌ **不做智能提取**：Phase 3 才用 LLM 提取关键字段

### 5.2 依赖解析逻辑

```python
def inject_params_node(state: NL2SQLFatherState) -> Dict[str, Any]:
    """
    Inject Params 节点：参数注入

    职责：
    1. 检查哪些子查询的依赖已满足
    2. 将依赖结果注入到 dependencies_results
    3. 更新子查询状态为 in_progress
    4. 更新 current_batch_ids

    Args:
        state: 父图 State

    Returns:
        更新的 State 字段：
        - current_batch_ids: 当前轮次待执行的子查询ID列表

    注意：
        轮次计数器 current_round 由 Check Completion 节点统一管理和递增
    """
    from src.utils.logger import get_module_logger, with_query_id

    logger = get_module_logger("inject_params")
    query_logger = with_query_id(logger, state.get("query_id", "unknown"))

    sub_queries = state.get("sub_queries", [])
    current_round = state.get("current_round", 1)

    query_logger.info(f"Inject Params 开始准备第 {current_round} 轮执行")

    # 1. 找出所有已完成的子查询
    completed_ids = {
        sq["sub_query_id"]
        for sq in sub_queries
        if sq["status"] == "completed"
    }

    # 2. 找出依赖已满足的子查询（status=pending）
    ready_to_execute = []

    for sq in sub_queries:
        if sq["status"] != "pending":
            continue  # 只处理 pending 状态

        # 检查所有依赖是否都已完成
        dependencies = sq.get("dependencies", [])
        if all(dep_id in completed_ids for dep_id in dependencies):
            ready_to_execute.append(sq)

    if not ready_to_execute:
        query_logger.warning("没有待执行的子查询")
        return {"current_batch_ids": []}

    # 3. 为每个待执行子查询注入依赖结果
    for sq in ready_to_execute:
        dependencies = sq.get("dependencies", [])
        dependencies_results = {}

        for dep_id in dependencies:
            # 找到依赖的子查询
            dep_sq = next((s for s in sub_queries if s["sub_query_id"] == dep_id), None)
            if dep_sq and dep_sq.get("execution_result"):
                # Phase 2: 简单传递完整结果
                dependencies_results[dep_id] = dep_sq["execution_result"]

        # 更新子查询
        sq["dependencies_results"] = dependencies_results
        sq["status"] = "in_progress"

    # 4. 更新 current_batch_ids
    batch_ids = [sq["sub_query_id"] for sq in ready_to_execute]

    query_logger.info(f"第 {current_round} 轮准备完成：{len(batch_ids)} 个子查询")

    return {
        "current_batch_ids": batch_ids,
    }
```

## 6. Check 完成度节点设计

### 6.1 职责定位

Check 完成度是 **循环退出的判定节点**，负责决定是继续循环还是结束流程。

**核心职责**：
1. **完成度检查**：是否所有子查询都已完成（completed 或 failed）
2. **最大轮次保护**：防止无限循环
3. **依赖环检测**：检测是否有子查询卡在 pending 状态（说明存在依赖环）
4. **路由决策**：返回 "inject_params"（继续）或 "summarizer"（结束）

### 6.2 判定逻辑

```python
def check_completion_node(state: NL2SQLFatherState) -> Dict[str, Any]:
    """
    Check 完成度节点：循环控制

    职责：
    1. 检查是否所有子查询都已完成
    2. 判断是否需要继续循环
    3. 最大轮次保护
    4. **统一管理轮次计数器**：本节点是唯一负责递增 current_round 的节点

    Args:
        state: 父图 State

    Returns:
        更新的 State 字段：
        - current_round: 轮次 +1（如果继续循环）
        - 如果结束循环则返回空字典 {}（路由到 Summarizer）

    重要说明：
        本节点是 Complex Path 循环控制的核心，负责：
        - 判定是否所有子查询完成（正常退出）
        - 检测是否达到最大轮次（强制退出）
        - 检测依赖环（异常退出）
        - 递增轮次计数器（继续下一轮）
    """
    from src.utils.logger import get_module_logger, with_query_id

    logger = get_module_logger("check_completion")
    query_logger = with_query_id(logger, state.get("query_id", "unknown"))

    sub_queries = state.get("sub_queries", [])
    current_round = state.get("current_round", 1)
    max_rounds = state.get("max_rounds", 5)

    query_logger.info(f"Check 完成度：当前轮次 {current_round}/{max_rounds}")

    # 统计子查询状态
    status_count = {
        "pending": 0,
        "in_progress": 0,
        "completed": 0,
        "failed": 0,
    }

    for sq in sub_queries:
        status = sq.get("status", "pending")
        status_count[status] += 1

    query_logger.info(
        f"子查询状态：pending={status_count['pending']}, "
        f"in_progress={status_count['in_progress']}, "
        f"completed={status_count['completed']}, "
        f"failed={status_count['failed']}"
    )

    # 判定1：所有子查询都已完成或失败 → 结束
    if status_count["pending"] == 0 and status_count["in_progress"] == 0:
        query_logger.info("所有子查询完成，进入 Summarizer")
        return {}  # 不更新轮次，进入 Summarizer

    # 判定2：达到最大轮次 → 强制结束
    if current_round >= max_rounds:
        query_logger.warning(f"达到最大轮次 {max_rounds}，强制结束")

        # 将所有 pending/in_progress 标记为 failed
        for sq in sub_queries:
            if sq["status"] in ["pending", "in_progress"]:
                sq["status"] = "failed"
                sq["error"] = f"超过最大轮次 {max_rounds}，强制终止"

        return {}  # 进入 Summarizer

    # 判定3：检测依赖环（有 pending 但没有 ready_to_execute）
    if status_count["pending"] > 0:
        # 检查是否有 pending 子查询的依赖已满足
        completed_ids = {
            sq["sub_query_id"]
            for sq in sub_queries
            if sq["status"] == "completed"
        }

        has_ready = False
        for sq in sub_queries:
            if sq["status"] == "pending":
                dependencies = sq.get("dependencies", [])
                if all(dep_id in completed_ids for dep_id in dependencies):
                    has_ready = True
                    break

        if not has_ready:
            # 所有 pending 子查询的依赖都未满足 → 依赖环
            query_logger.error("检测到依赖环或孤立子查询，强制结束")

            for sq in sub_queries:
                if sq["status"] == "pending":
                    sq["status"] = "failed"
                    sq["error"] = "依赖环或孤立子查询"

            return {}  # 进入 Summarizer

    # 判定4：继续下一轮
    query_logger.info(f"进入第 {current_round + 1} 轮")
    return {
        "current_round": current_round + 1,
    }


def route_after_check_completion(state: NL2SQLFatherState) -> str:
    """
    Check 完成度后的路由函数

    Returns:
        "inject_params": 继续下一轮
        "summarizer": 结束流程，进入总结
    """
    sub_queries = state.get("sub_queries", [])

    # 检查是否所有子查询都已完成或失败
    all_done = all(
        sq["status"] in ["completed", "failed"]
        for sq in sub_queries
    )

    if all_done:
        return "summarizer"
    else:
        return "inject_params"
```

### 6.3 退出条件总结

| 退出条件 | 触发场景 | 处理方式 |
|---------|---------|---------|
| **正常完成** | 所有子查询 status 为 completed/failed | 进入 Summarizer |
| **最大轮次** | current_round >= max_rounds | 标记未完成子查询为 failed，进入 Summarizer |
| **依赖环** | pending 子查询无法推进 | 标记为 failed，进入 Summarizer |
| **所有子查询失败** | 所有 status 为 failed | 进入 Summarizer，返回友好提示 |

### 7.3 兼容性设计（重点）

```python
def sql_execution_node(state: NL2SQLFatherState, config: Dict[str, Any] = None) -> Dict[str, Any]:
    """
    SQL 执行节点（共享节点，Phase 1 + Phase 2）

    兼容性设计：
    - Fast Path: 串行执行 1 个 SQL（收集 status=completed）
    - Complex Path: 并发执行多个 SQL（收集 status=in_progress）
    - 自动检测模式，无需修改 Fast Path 代码
    - 避免重复执行：跳过已有 execution_result 的子查询

    状态语义：
    - completed: Phase 1 的"SQL 生成成功，等待执行"
    - in_progress: Phase 2 的"依赖已注入，等待执行"

    Args:
        state: 父图 State
        config: 运行时配置（可选）

    Returns:
        更新的 State 字段：
        - execution_results: SQL 执行结果列表
    """
    from concurrent.futures import ThreadPoolExecutor, as_completed
    from src.services.db.pg_client import PGClient
    from src.utils.logger import get_module_logger, with_query_id

    logger = get_module_logger("sql_execution")
    query_logger = with_query_id(logger, state.get("query_id", "unknown"))

    # 加载配置
    exec_config = _get_sql_execution_config()
    timeout_per_sql = exec_config["timeout_per_sql"]
    max_concurrency = exec_config.get("max_concurrency", 1)
    enable_parallel = exec_config.get("enable_parallel", False)
    log_sql = exec_config.get("log_sql", True)

    # 1. 从 sub_queries 中收集待执行 SQL
    sub_queries = state.get("sub_queries", [])
    sqls_to_execute = []
    sub_query_map = {}

    for sq in sub_queries:
        # 跳过已执行的子查询（避免 Phase 2 循环中重复执行）
        if sq.get("execution_result") is not None:
            continue

        # 收集待执行的 SQL：
        # - Phase 1 (Fast Path): status=completed（SQL 生成成功后的状态）
        # - Phase 2 (Complex Path): status=in_progress（Inject Params 注入依赖后的状态）
        if sq.get("status") in ["completed", "in_progress"] and sq.get("validated_sql"):
            sql_item = {
                "sub_query_id": sq["sub_query_id"],
                "sql": sq["validated_sql"],
            }
            sqls_to_execute.append(sql_item)
            sub_query_map[sq["sub_query_id"]] = sq

    if not sqls_to_execute:
        query_logger.warning("没有待执行的SQL")
        return {"execution_results": []}

    if log_sql:
        query_logger.info(f"准备执行 {len(sqls_to_execute)} 条SQL")

    # 2. 自动检测：单SQL → 串行，多SQL → 并发
    results: List[SQLExecutionResult] = []
    pg_client = PGClient()

    if len(sqls_to_execute) == 1 or not enable_parallel:
        # Fast Path 或 并发未启用：串行执行
        query_logger.debug("使用串行执行模式")
        for idx, sql_item in enumerate(sqls_to_execute, 1):
            result = _execute_single_sql(
                sql_item, pg_client, timeout_per_sql, idx, len(sqls_to_execute), query_logger
            )
            results.append(result)

            # 更新 sub_query 的 execution_result
            sub_query_id = sql_item["sub_query_id"]
            if sub_query_id in sub_query_map:
                sub_query_map[sub_query_id]["execution_result"] = result
                sub_query_map[sub_query_id]["status"] = "completed" if result["success"] else "failed"

    else:
        # Complex Path：并发执行
        query_logger.debug(f"使用并发执行模式（max_concurrency={max_concurrency}）")
        with ThreadPoolExecutor(max_workers=max_concurrency) as executor:
            # 提交所有任务
            future_to_sql = {
                executor.submit(
                    _execute_single_sql,
                    sql_item, pg_client, timeout_per_sql, idx, len(sqls_to_execute), query_logger
                ): sql_item
                for idx, sql_item in enumerate(sqls_to_execute, 1)
            }

            # 收集结果
            for future in as_completed(future_to_sql):
                sql_item = future_to_sql[future]
                try:
                    result = future.result()
                    results.append(result)

                    # 更新 sub_query
                    sub_query_id = sql_item["sub_query_id"]
                    if sub_query_id in sub_query_map:
                        sub_query_map[sub_query_id]["execution_result"] = result
                        sub_query_map[sub_query_id]["status"] = "completed" if result["success"] else "failed"

                except Exception as e:
                    # 并发任务异常
                    query_logger.error(f"并发执行异常: {str(e)}")
                    result = {
                        "sub_query_id": sql_item["sub_query_id"],
                        "sql": sql_item["sql"],
                        "success": False,
                        "error": str(e),
                        "columns": None,
                        "rows": None,
                        "row_count": 0,
                        "execution_time_ms": 0,
                    }
                    results.append(result)

    return {"execution_results": results}


def _execute_single_sql(
    sql_item: Dict,
    pg_client,
    timeout: int,
    idx: int,
    total: int,
    query_logger,
) -> Dict[str, Any]:
    """
    执行单条 SQL（串行/并发复用）

    Args:
        sql_item: {"sub_query_id": "...", "sql": "..."}
        pg_client: PGClient 实例
        timeout: 超时时间
        idx: 当前索引
        total: 总数
        query_logger: 日志器

    Returns:
        SQLExecutionResult
    """
    import time

    sub_query_id = sql_item["sub_query_id"]
    sql = sql_item["sql"]

    query_logger.info(f"执行第 {idx}/{total} 条SQL (sub_query_id: {sub_query_id})")
    start_time = time.time()

    try:
        # 执行SQL
        result = pg_client.execute_query(sql, timeout=timeout)
        execution_time_ms = (time.time() - start_time) * 1000

        exec_result = {
            "sub_query_id": sub_query_id,
            "sql": sql,
            "success": True,
            "columns": result.get("columns"),
            "rows": result.get("rows"),
            "row_count": len(result.get("rows", [])),
            "execution_time_ms": execution_time_ms,
            "error": None,
        }

        query_logger.info(
            f"SQL执行成功，返回 {len(result.get('rows', []))} 行，耗时 {execution_time_ms:.0f}ms"
        )

        return exec_result

    except Exception as e:
        execution_time_ms = (time.time() - start_time) * 1000
        error_msg = str(e)

        exec_result = {
            "sub_query_id": sub_query_id,
            "sql": sql,
            "success": False,
            "columns": None,
            "rows": None,
            "row_count": 0,
            "execution_time_ms": execution_time_ms,
            "error": error_msg,
        }

        query_logger.error(f"SQL执行失败: {error_msg}")

        return exec_result
```

**配置说明**：
- `max_rounds` 由 **Planner 节点**从配置读取并写入 State
- **Check Completion 节点**从 State 中读取此值进行判定
- 操作员应在 `planner` 部分调整此参数（而不是 `check_completion`）

- 简单依赖（sq1→sq2）：2 轮
- 中等复杂（sq1→sq2→sq3）：3 轮
- 复杂并发（sq1→[sq2,sq3]→sq4）：3 轮
- **超过5轮**：可能存在设计问题，强制终止

## 10. 父图拓扑与连线

### 10.1 完整拓扑（Fast + Complex）

```python
from langgraph.graph import StateGraph, END, START

def create_nl2sql_father_graph():
    """
    创建 NL2SQL 父图（Phase 1 + Phase 2）

    拓扑：
    START → router → [simple_planner | planner]
                          ↓              ↓
                      sql_gen    inject_params
                          ↓              ↓
                    [sql_exec |      sql_gen_batch
                   summarizer]          ↓
                          ↓          sql_exec
                         END            ↓
                                  check_completion
                                    ↓         ↓
                              inject_params  summarizer
                              (循环边)          ↓
                                              END
    """
    graph = StateGraph(NL2SQLFatherState)

    # ========== 添加节点 ==========
    # Phase 1 节点
    graph.add_node("router", router_node)
    graph.add_node("simple_planner", simple_planner_node)
    graph.add_node("sql_gen", sql_gen_wrapper)              # Fast Path: 单SQL生成
    graph.add_node("sql_exec", sql_execution_node)          # 共享节点
    graph.add_node("summarizer", summarizer_node)           # 共享节点

    # Phase 2 新增节点
    graph.add_node("planner", planner_node)                 # Complex: 问题拆分
    graph.add_node("inject_params", inject_params_node)     # Complex: 参数注入
    graph.add_node("sql_gen_batch", sql_gen_batch_wrapper)  # Complex: 批量SQL生成
    graph.add_node("check_completion", check_completion_node)  # Complex: 完成度检查

    # ========== 添加边 ==========
    # 入口
    graph.add_edge(START, "router")

    # Router 条件边
    graph.add_conditional_edges(
        "router",
        route_by_complexity,
        {
            "simple_planner": "simple_planner",  # Fast Path
            "planner": "planner",                # Complex Path
        }
    )

    # ===== Fast Path =====
    graph.add_edge("simple_planner", "sql_gen")
    graph.add_conditional_edges(
        "sql_gen",
        route_after_sql_gen,
        {
            "sql_exec": "sql_exec",
            "summarizer": "summarizer",
        }
    )
    graph.add_edge("sql_exec", "summarizer")

    # ===== Complex Path =====
    graph.add_edge("planner", "inject_params")
    graph.add_edge("inject_params", "sql_gen_batch")
    graph.add_edge("sql_gen_batch", "sql_exec")
    graph.add_edge("sql_exec", "check_completion")
    graph.add_conditional_edges(
        "check_completion",
        route_after_check_completion,
        {
            "inject_params": "inject_params",  # 循环边
            "summarizer": "summarizer",        # 结束
        }
    )

    # 统一出口
    graph.add_edge("summarizer", END)

    # ========== 编译 ==========
    app = graph.compile()

    return app
```

### 10.2 SQL 生成批量调用 Wrapper

```python
def sql_gen_batch_wrapper(state: NL2SQLFatherState) -> Dict[str, Any]:
    """
    SQL 生成批量调用 Wrapper（Complex Path 专用）

    职责：
    1. 从 current_batch_ids 中获取待执行的子查询列表
    2. 串行调用 SQL 生成子图（Phase 2 不并发调用子图，避免资源竞争）
    3. 更新每个子查询的 validated_sql 和 status

    Args:
        state: 父图 State

    Returns:
        更新的 State 字段（无，直接修改 sub_queries）
    """
    from src.modules.sql_generation.subgraph.create_subgraph import run_sql_generation_subgraph
    from src.utils.logger import get_module_logger, with_query_id

    logger = get_module_logger("sql_gen_batch")
    query_logger = with_query_id(logger, state.get("query_id", "unknown"))

    current_batch_ids = state.get("current_batch_ids", [])
    sub_queries = state.get("sub_queries", [])

    if not current_batch_ids:
        query_logger.warning("没有待生成SQL的子查询")
        return {}

    query_logger.info(f"批量生成SQL：{len(current_batch_ids)} 个子查询")

    for sub_query_id in current_batch_ids:
        # 找到对应的子查询
        sub_query = next((sq for sq in sub_queries if sq["sub_query_id"] == sub_query_id), None)
        if not sub_query:
            query_logger.error(f"子查询 {sub_query_id} 不存在")
            continue

        query_logger.info(f"生成SQL: {sub_query_id}")

        try:
            # 调用 SQL 生成子图
            subgraph_output = run_sql_generation_subgraph(
                query=sub_query["query"],
                query_id=state["query_id"],
                user_query=state["user_query"],
                dependencies_results=sub_query.get("dependencies_results", {}),
                parse_hints=None,
            )

            # 更新子查询状态
            if subgraph_output.get("validated_sql"):
                sub_query["validated_sql"] = subgraph_output["validated_sql"]
                sub_query["iteration_count"] = subgraph_output.get("iteration_count", 0)
                # 注意：不在这里更新 status，由 SQL 执行节点更新
            else:
                sub_query["status"] = "failed"
                sub_query["error"] = subgraph_output.get("error", "SQL生成失败")

        except Exception as e:
            error_msg = f"SQL生成异常: {str(e)}"
            query_logger.error(error_msg, exc_info=True)
            sub_query["status"] = "failed"
            sub_query["error"] = error_msg

    return {}  # 无需返回，已直接修改 sub_queries
```

### 10.3 路由函数

```python
def route_by_complexity(state: NL2SQLFatherState) -> str:
    """
    Router 后的路由函数

    Returns:
        "simple_planner": Fast Path
        "planner": Complex Path
    """
    complexity = state.get("complexity")

    if complexity == "simple":
        return "simple_planner"
    else:
        return "planner"


def route_after_sql_gen(state: NL2SQLFatherState) -> str:
    """
    SQL 生成后的路由函数（Fast Path）

    Returns:
        "sql_exec": SQL生成成功
        "summarizer": SQL生成失败
    """
    if state.get("validated_sql"):
        return "sql_exec"
    else:
        return "summarizer"


def route_after_check_completion(state: NL2SQLFatherState) -> str:
    """
    Check 完成度后的路由函数（Complex Path）

    Returns:
        "inject_params": 继续下一轮
        "summarizer": 结束流程
    """
    sub_queries = state.get("sub_queries", [])

    # 检查是否所有子查询都已完成或失败
    all_done = all(
        sq["status"] in ["completed", "failed"]
        for sq in sub_queries
    )

    if all_done:
        return "summarizer"
    else:
        return "inject_params"
```
