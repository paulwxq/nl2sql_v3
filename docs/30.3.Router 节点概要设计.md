### Fast Path 第一节点：Router 问题分类（概要设计，LangGraph v1.0.x）

> 目标：以极低成本、极低延迟将问题二分类为 simple | complex，并将 simple 路由至 Fast Path（SQL 生成子图 → SQL 执行 → Summarizer），将 complex 路由至右侧复杂路径（Planner → …）。

---

#### 1. 职责与范围
- 仅做“问题复杂度二分类”，不做重写、不做拆解。
- 输出规范化后的 `complexity`（simple | complex），并提供最小可观测信息（模型、置信度、裁决理由）。
- 不引入额外分支（如 clarify、batch 等），后续版本可扩展为多分类。
- 失败时采取保守策略：默认判为 complex（便于后续 Planner 兜底诊断）。


#### 2. 与父 State 的对接（最小集）
为便于后续节点复用与观测，建议父图 `ParentState` 至少包含下列与 Router 相关字段（Node 0 的父 State 全量定义会单独文档化，本处给出最小必需与建议项）：

```python
# 必需（Router 读入 / 写出）
user_question: str                      # 输入：用户原始自然语言问题
complexity: Optional[Literal["simple", "complex"]]  # 输出：路由结果

# 建议（可观测性 & 调试）
router_model: Optional[str]             # 使用的轻量模型名称
router_confidence: Optional[float]      # 归一化置信度（0~1）；无法估计则为 None
router_reason: Optional[str]            # 简要裁决理由（<= 200 chars）
router_latency_ms: Optional[int]        # 节点耗时（毫秒）
```

与 SQL 生成子图的接口兼容性（参考《30.2.SQL 生成子图 State 定义.md》）：
- 当走 Fast Path 时，父图后续节点会将
  - `query := user_question`
  - `query_id := 由父图生成的短ID（如 uuid4().hex[:8]）`
  - `dependencies_results := {}`
  - `user_query := user_question`
  - `parse_hints := None | 上游注入`
  传入子图 `SQLGenerationState`。


#### 3. 输入 / 输出定义（节点级）
- 输入：`user_question`
- 输出：
  - `complexity ∈ {"simple", "complex"}`（必填）
  - `router_model`（建议）
  - `router_confidence`（建议，无法稳定估计可为空）
  - `router_reason`（建议，<=200 字符）
  - `router_latency_ms`（建议）

说明：输出字段全部写回父 State，供条件边与观测使用。


#### 4. 路由与接线（LangGraph v1.0.x）
- 节点函数：`router_node(state) -> Dict[str, Any]`
- 条件边函数：`route_by_complexity(state) -> str`（返回 `"fast_path"` 或 `"plan_path"`，与《30.0.nl2sql_father_graph_architecture.html》保持一致）
- 不使用 `Command.goto`，而采用条件边函数，便于与右侧复杂路径保持同构接线。

示例（伪代码片段）：

```python
def router_node(state: ParentState) -> Dict[str, Any]:
    question = state["user_question"]

    prompt = """仅输出 simple 或 complex：
Simple: 单个独立问题，可一次SQL完成
Complex: 多个问题/有依赖/需中间结果
问题：{q}
答案：""".format(q=question)

    t0 = now_ms()
    result = cheap_llm.invoke(prompt)  # e.g., gpt-4o-mini, claude-haiku
    latency = now_ms() - t0

    raw = (result.content or "").strip().lower()
    complexity = "simple" if raw.startswith("simple") else (
        "complex" if raw.startswith("complex") else "complex"
    )

    return {
        "complexity": complexity,
        "router_model": result.model_name,
        "router_confidence": None,     # v1 先不承诺稳定置信度
        "router_reason": None,         # 可选：如需审计再打开
        "router_latency_ms": latency,
    }

def route_by_complexity(state: ParentState) -> str:
    return "fast_path" if state.get("complexity") == "simple" else "plan_path"
```


#### 5. 提示词与归一化规范
- 提示词原则：极短、明确、只允许二分类答案；避免多余自然语言。
- 归一化：统一转小写、取首词，非预期输出一律判为 `"complex"`。
- 成本与时延：优先选用低成本、低延迟模型；超时（如 > 800ms）视为失败并判 `"complex"`。


#### 6. 可复用与阶段性约束
- 与右侧复杂路径共享 Summarizer：不在 Router 层做特殊处理，仅作为架构约束记录；后续 Fast Path 与 Complex Path 均调用同一 `Summarizer` 节点。
- SQL 执行节点复用：Router 只负责分流；SQL 执行实现为独立节点（支持单/多 SQL），两条路径共用。
- 保持父 State 的“最小写入原则”：Router 仅写入与路由相关最小字段，避免状态污染。


#### 7. 失败与异常处理（与用户偏好一致）
- 不设置掩盖性“兜底/回退”策略；若调用失败或输出异常，直接判为 `"complex"`，将问题交由 Planner 处理（直面根因，而非隐藏）。
- 度量与报警：结合 `router_latency_ms` 与错误日志进行监控与告警，不在节点内自作恢复。


#### 8. 测试要点（第一阶段）
- 简单问题样例（应 → simple）：
  - “广州服务区的销售额是多少？”
  - “2024年总销售额是多少？”
  - “北京和上海服务区的销售额对比”（单 SQL 可完成的对比）
- 复杂问题样例（应 → complex）：
  - “哪个服务区销售最高？它的地址和公司是？”
  - “找出车流量前3的服务区，分别查询它们的营业额”

验收标准：
- 分类准确率（样例集）≥ 95%
- 平均延迟 ≤ 200ms（本地/同区）
- 失败率（超时/异常）≤ 1%，失败时均正确回退为 complex


#### 9. 与后续节点的边界
- Router 不关心 SQL 生成子图/执行/总结的内部细节，仅产出可靠路由信号。
- 父图在 `"fast_path"` 分支需准备 SQL 子图所需输入（见第 2 节映射约定）。
- Summarizer 节点将被两条路径共享；Router 不参与 Summarizer 的配置。


#### 10. 实施清单（面向开发）
- 定义/补充父 `ParentState` 的 Router 相关字段（第 2 节最小集）。
- 实现 `router_node` 与 `route_by_complexity`，并在父图中串上条件边。
- 增加观测埋点：记录模型名、延迟、原始裁决值（仅日志）。
- 基于样例集做 A/B 验证（不同轻量模型），选择性价比最优者为默认。
