# 对话历史记录功能概要设计

> 说明：本文为早期概要设计，后续讨论已对注入范围、字段命名与数据流做了调整；最终以 `docs/75_对话历史功能优化与bug修复.md` 为准。

## 1. 需求概述

### 1.1 目标
在 NL2SQL 模块中增加从 PostgresStore 读取用户历史对话记录并注入到提示词中的功能，使 LLM 在回答 NL 问题或生成 SQL 时能够参考历史对话上下文，从而给出更准确的回答或 SQL。

### 1.2 核心要求
1. 历史对话记录仅包含 **QUESTION/ANSWER**，不包含其他执行信息
2. 通过配置文件设置返回的 Q/A 数量（如 3 组）
3. 基于现有 `langgraph.store` 表读取历史数据

## 2. 现状分析

### 2.1 Store 数据结构
```sql
-- langgraph.store 表结构
CREATE TABLE langgraph.store (
    prefix      text NOT NULL,
    key         text NOT NULL,
    value       jsonb NOT NULL,
    created_at  timestamp with time zone DEFAULT CURRENT_TIMESTAMP,
    updated_at  timestamp with time zone DEFAULT CURRENT_TIMESTAMP,
    expires_at  timestamp with time zone,
    ttl_minutes integer,
    PRIMARY KEY (prefix, key)
);
```

**Key 格式**：`{thread_id}#{query_id}`
- 示例：`guest:20251222T052927151Z#q_3990d76d`
- `thread_id` = `guest:20251222T052927151Z`（用户名 + 会话创建时间）
- 排序依据：建议按 `store.updated_at/created_at`（或 `value.created_at`）排序；同一 `thread_id` 内的时间戳无法区分轮次

**Value 结构**（JSON）：
```json
{
  "thread_id": "guest:20251222T052927151Z",
  "user_id": "guest",
  "query_id": "q_3990d76d",
  "created_at": "2025-12-22T05:38:49.059526+00:00",
  "user": {"role": "user", "content": "请问广州市的京东便利店的总收入是多少"},
  "assistant": {"role": "assistant", "content": "广州市的京东便利店的总收入为3656.00元。"},
  "metadata": {"complexity": "simple", "path_taken": "fast", ...}
}
```

### 2.2 现有代码分析

| 模块 | 文件 | 现状 |
|------|------|------|
| Store 写入 | `chat_history_writer.py` | ✅ 已实现 `append_turn()` 写入功能 |
| Store 读取 | - | ❌ **未实现**，需要新增 |
| SQL 生成提示词 | `sql_generation.py` | ❌ 未预留历史对话占位符 |

### 2.3 SQL 生成子图提示词结构

当前 `_build_prompt()` 方法组装的提示词包含：
- 用户查询 (`query`)
- 时间信息 (`time_info`)
- 维度过滤 (`dimension_filters`)
- 指标信息 (`metric_info`)
- 依赖结果 (`dependencies_text`)
- 表结构 (`table_cards_text`)
- JOIN 计划 (`join_plans_text`)
- 时间列参考 (`time_columns_text`)
- 维度值匹配 (`dim_values_text`)
- 历史成功 SQL 案例 (`similar_sqls_text`)
- 验证错误 (`errors_text`)

**缺少**：历史对话记录（conversation history）

## 3. 设计决策

### 3.1 问题：历史对话记录与依赖子查询结果是否冲突？

**结论：不冲突**

| 类型 | 来源 | 用途 | 时效性 |
|------|------|------|--------|
| 历史对话记录 | Store 表 | 提供用户意图的上下文参考 | 跨会话/跨问题 |
| 依赖子查询结果 | 当前会话内存 | 提供当前问题的子查询执行结果 | 仅当前问题 |

两者是**互补关系**：
- 历史对话帮助理解用户的业务背景和偏好
- 依赖结果提供当前问题的具体数据值

### 3.2 问题：父图是否需要历史对话记录？

**分析各节点的 LLM 使用情况**：

| 节点 | 使用 LLM | 是否需要历史对话 | 理由 |
|------|----------|------------------|------|
| Router | ✅ 是 | ❌ **不需要** | 仅判断 simple/complex，与历史无关 |
| Simple Planner | ❌ 否 | - | 无 LLM 调用 |
| Planner (Complex) | ✅ 是 | ✅ **需要** | 复杂问题拆分与依赖建立可能受历史指代影响 |
| SQL Generation | ✅ 是 | ✅ **需要** | 核心场景，SQL 生成需要参考历史上下文 |
| Summarizer | ✅ 是 | ✅ **需要** | 省略式追问时用于指代消解；结果以本轮执行结果为准 |

**设计决策（更新）**：
- 本次实现即在 5 处注入/使用 history：SQL 子图 `question_parsing`、`schema_retrieval`（通过 `rewritten_query` 生效）、`sql_generation`；父图 `planner`、`summarizer`
- 父图统一读取一次 history（旧→新）并透传给子图；子图单独运行时允许无 history（不报错，仅缺少 history）

**理由**：
1. SQL 生成是最核心的场景，影响最大
2. 最小化改动范围，降低风险
3. 子图已禁用 checkpoint，读取 Store 不会与 checkpoint 冲突

### 3.3 问题：子图无法直接获取 thread_id

**现状**：
- 子图的 `run_sql_generation_subgraph()` 已预留 `thread_id` 参数
- 父图调用时已传入 `thread_id`

**方案**：
- 在子图的 State 中添加 `thread_id` 字段
- 在 SQL 生成节点读取历史时使用此 `thread_id`

```python
提供 get_recent_turns() 函数，用于从 PostgresStore 读取最近的对话历史。

        exclude_query_id: 要排除的 query_id（通常是当前正在处理的）
        
    Returns:
        对话历史列表，格式：
        [
            {"question": "用户问题1", "answer": "AI回答1"},
            {"question": "用户问题2", "answer": "AI回答2"},
            ...
        ]
        按时间倒序排列（最近的在前）
    """
    # 实现细节见 4.2.1
```

#### 4.2.1 实现要点

1. **查询策略**：使用 PostgresStore 的 `search()` 方法，按 namespace_prefix（落库 prefix）匹配
   ```python
   # 约定：将 thread_id 下钻到 prefix（namespace）
   # 写入：store.put(namespace=("chat_history", thread_id), key=query_id, value=...)
   # 读取：直接在 ("chat_history", thread_id) 范围内 search
   results = store.search((namespace, thread_id), limit=history_max_turns, offset=0)
   ```

2. **排序逻辑**：
   - `PostgresStore.search()`（未启用向量检索时）默认按 `store.updated_at DESC` 返回最近的记录
   - 取最近 N 条后，在代码层翻转为“旧→新”再注入提示词

3. **超时控制**：
   - 使用线程池 + Future 实现超时（复用 writer 的模式）
   - 超时或异常时返回空列表（fail-open）

4. **排除当前问题**：
   - 传入 `exclude_query_id` 排除当前正在处理的问题
   - 避免把当前问题（尚未完成）加入历史

#### 4.5.3 新增 `_format_conversation_history()` 方法

```python
def _format_conversation_history(
    self, 
    history: Optional[List[Dict[str, str]]]
) -> str:
    """格式化历史对话记录
    
    Args:
        history: 对话历史列表
        
    Returns:
        格式化后的字符串，若无历史则返回空字符串
    """
    if not history:
        return ""
    
    lines = ["历史对话（参考）："]
    for idx, turn in enumerate(history, 1):
        q = turn.get("question", "")
        a = turn.get("answer", "")
        lines.append(f"  {idx}. 问：{q[:200]}...")  # 截断避免过长
        lines.append(f"     答：{a[:200]}...")
    
    return "\n".join(lines)
```

## 9. 后续扩展

1. **跨 thread_id 历史**：同一用户的多个会话共享部分上下文（需要配合鉴权/隔离）
2. **历史重要性评分**：基于相似度筛选最相关的历史（降低无关 history 干扰）
3. **history 总预算控制**：对 history 区块的总长度/总轮次设置硬上限（避免极端输入撑爆 prompt）

---

## 10. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| Store 读取超时 | SQL 生成延迟 | fail-open：超时返回空历史，不阻塞主流程 |
| 历史过长导致 token 超限 | LLM 调用失败 | 截断历史内容（max_history_content_length） |
| 无关历史干扰 SQL 生成 | SQL 质量下降 | 限制历史数量（history_max_turns），后续可加相似度过滤 |
| thread_id 为空 | 无法读取历史 | 跳过历史读取，不影响 SQL 生成 |

---

## 11. 总结

本设计采用 **最小化改动** 原则：
- 父图统一读取 history 并透传，避免子图重复读 Store
- 在 5 处注入/使用 history（SQL 子图解析/检索/生成 + 父图 planner/summarizer）
- 复用现有 Store 基础设施
- fail-open 设计，不影响主流程稳定性
- 可配置的历史数量和开关

核心改动点：
1. 新增 `chat_history_reader.py` 读取器
2. 修改 `sql_generation.py` 注入历史到提示词
3. `question_parsing` 输出 `rewritten_query` 并驱动检索/生成
4. 父图 `planner/summarizer` 复用同一份 history
