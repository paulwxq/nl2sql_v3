# 85_当前父图依赖注入与并发执行逻辑缺陷分析

## 1. 文档目的

本文用于重新梳理当前 `nl2sql_father` 在 Complex Path 上的真实问题，重点分析以下几类风险：

- 依赖结果是否真正闭环传递
- 复杂路径的收敛方式是否合理
- SQL 执行阶段的状态更新方式是否安全
- 当前配置开关是否真正生效

本文的目标不是泛泛而谈“架构不够先进”，而是基于现有代码判断：

1. 哪些问题已经是明确缺陷
2. 哪些问题属于设计刚性，但不是 bug
3. 哪些问题应当优先修复

---

## 2. 结论摘要

对当前实现进行复核后，可以将问题分为三类：

### 2.1 明确缺陷

以下问题应视为真实 bug：

- **依赖子查询的信息没有完整进入子图前半段**
  - `dependencies_results` 会进入 SQL 生成阶段，但不会进入 `question_parsing` 的语义重写与结构化解析阶段
- **`complex_path_enabled` 配置开关未真正生效**
  - 虽然配置中写了 `false`，但父图路由仍会进入 Complex Path

### 2.2 设计缺陷，但不是立即错误

以下问题目前更准确地说是“设计偏硬”，而不是实现错误：

- `Check Completion` 的失败收敛策略比较刚性
- Complex Path 缺少真正的动态重规划能力

### 2.3 需要优化，但原结论过度放大

以下问题存在优化空间，但不宜描述成线程竞态类严重 bug：

- `sql_execution_node` 依赖 `in-place` 修改 `sub_queries`
- 但这些状态修改目前发生在主线程收集 future 结果时，不是多个 worker 线程同时改共享状态

---

## 3. 当前 Complex Path 的真实执行链路

当前复杂路径的主流程是：

`Router -> Planner -> Inject Params -> SQL Gen Batch -> SQL Execution -> Check Completion -> Summarizer`

其中：

- `Planner`
  - 一次性生成所有子查询和依赖关系
- `Inject Params`
  - 把上游子查询的执行结果整理为 `dependencies_results`
- `SQL Gen Batch`
  - 串行调用 SQL 子图，为当前批次子查询生成 SQL
- `SQL Execution`
  - 串行或并发执行当前批次的 SQL
- `Check Completion`
  - 判断是否继续下一轮或结束

从代码实现上看，当前系统不是“完全没有依赖注入”，而是：

- **依赖结果会被注入**
- **但注入位置不够理想**

这正是复杂路径目前最关键的真实问题。

---

## 4. 问题一：依赖注入链路没有真正闭环

### 4.1 当前实现做了什么

`Planner` 的提示词允许 LLM 生成类似 `{{sq1.result}}` 的引用表达，例如：

- “查询服务区 {{sq1.result}} 的地址和公司”

随后，`Inject Params` 会把依赖子查询的执行结果收集到 `dependencies_results` 中，并挂到待执行子查询上。

也就是说，系统当前实际是：

- **保留原始 query 文本不变**
- **额外提供一份结构化依赖结果**

这不是“完全没传依赖”，而是“依赖没有以最合理的方式进入整个子图链路”。

### 4.2 真正的问题在哪里

问题不在于“没有做字符串替换”本身，而在于：

- SQL 子图的第一步是 `question_parsing`
- 这一阶段拿到的 `query` 仍然是原始文本
- `question_parsing` 不读取 `dependencies_results`

这意味着：

- 当子查询文本里仍带有 `{{sq1.result}}` 这种占位提示时
- 子图前半段的语义解析、问题重写、结构化抽取，看到的是一个不完整甚至不自然的问题

虽然到了后面的 `sql_generation` 节点，`dependencies_results` 会被格式化进 prompt，LLM 仍然可能“凑合工作”，但这已经造成了明显的链路脱节：

- `question_parsing` 阶段看不到依赖结果
- `sql_generation` 阶段才看到依赖结果

这会带来几个后果：

1. **依赖子查询的结构化解析不稳定**
   - 例如时间、维度、指标抽取可能基于不完整问题进行
2. **问题重写不完整**
   - 子图不能在最早阶段把依赖引用重写成完整自然语言问题
3. **Schema 检索上下文质量下降**
   - 如果前面的解析结果不完整，后续检索与生成质量都受影响

### 4.3 问题定性

这个问题应定性为：

- **高优先级真实缺陷**
- 不是“完全没有依赖注入”
- 而是“依赖注入发生得太晚，导致子图前半段无法正确消费依赖信息”

### 4.4 修复建议

不建议把修复方案简单写成“对 query 做正则替换”。

更稳妥的方案有两类：

#### 方案 A：在父图侧先生成 dependency-aware 的显式子问题

做法：

- 在 `Inject Params` 后增加一个轻量问题重写步骤
- 将原始 query 与 `dependencies_results` 组合成一个更完整的自然语言子问题
- 再送入 SQL 子图

优点：

- 子图接口保持清晰
- `question_parsing` 能从一开始就看到完整问题

#### 方案 B：增强 SQL 子图入口，让 `question_parsing` 感知依赖结果

做法：

- 当 `dependencies_results` 存在时
- 在 `question_parsing` 阶段把依赖结果也作为输入上下文

优点：

- 不需要修改父图 query 文本
- 能让子图内部自己完成更完整的重写与解析

#### 方案选择建议

从侵入性和回归风险角度，**建议优先采用方案 A**。

理由：

- SQL 子图是 Fast Path 和 Complex Path 共享的核心模块，Fast Path 高频使用子图，任何子图改动都有回归风险
- 方案 A 只在父图侧增加一个轻量步骤，子图接口完全不变
- 方案 B 虽然更自然（`dependencies_results` 已在子图 state 中），但需要修改 `question_parsing_node` 的 prompt 和调用签名，会侵入已稳定运行的子图

如果后续验证方案 A 的重写质量不够理想，再考虑切换到方案 B。

## 5. 问题二：Complex Path 收敛策略较硬，但不是“死锁”

### 5.1 原始误判需要纠正

当前系统并不是一旦依赖链出错就会“无限循环”或“死锁”。

原因是 `Check Completion` 已经具备两层保护：

- **最大轮次保护**
- **无可推进 pending 子查询时，将其标记为 failed**

因此，从代码行为上看：

- 它会失败
- 但一般不会无限卡住

### 5.2 当前真实问题

真正的问题是：

- 当前收敛逻辑太刚
- 一旦某条依赖链无法推进，就直接标记为 failed
- 没有”重新规划后续步骤”的能力

举例：

- 第一步 SQL 成功，但结果为空
- 第二步原本依赖第一步结果
- 当前系统不会尝试改变问题拆解方式
- 而是更倾向于进入失败收敛

这不是 bug，而是当前静态 DAG 方案的天然边界。

### 5.3 补充细节：failed 依赖导致的不必要延迟

依赖环检测逻辑（`check_completion.py:119`）在判断 pending 子查询是否可推进时，只检查 `completed` 状态：

```python
completed_ids = {sq[“sub_query_id”] for sq in sub_queries if sq.get(“status”) == “completed”}
```

这意味着：如果 sq1 **failed**，依赖 sq1 的 sq2 不会被立即标记为 failed，而是继续保持 `pending`。系统要等到下一轮循环，由依赖环检测发现 sq2 无法推进后，才将其标记为 failed。

这不算严格意义上的 bug（最终结果正确），但存在**不必要的延迟**——本可以在发现依赖节点 failed 后立即标记下游子查询为 failed，而不是多等一轮。

建议的改进方式：

- 在依赖环检测逻辑中，同时收集 `failed_ids`
- 如果 pending 子查询的某个依赖在 `failed_ids` 中，直接标记为 failed 并附带原因”依赖子查询 {dep_id} 执行失败”
- 这样可以减少一轮无意义的循环，同时让 Summarizer 能输出更精确的失败原因

### 5.4 问题定性

这个问题更适合定义为：

- **架构设计刚性**
- **能力边界不足**
- **不是实现层面的严重 bug**

### 5.5 修复建议

建议把该问题的修复分成两层：

#### 第一层：不改架构，增强失败诊断

在现有 `Check Completion` 基础上增加更清晰的失败分类，例如：

- 依赖节点失败导致的不可推进
- 空结果导致的不可推进
- 轮次耗尽导致的强制终止

这样至少能让 `Summarizer` 输出更准确的失败原因。

#### 第二层：引入动态重规划能力

这一步才是 `deepagents` 或其他动态规划机制真正适合介入的地方。

换句话说：

- `deepagents` 适合解决这个问题
- 但这个问题本身不应被表述为“当前代码死锁”

## 6. 问题三：多个节点的 in-place 状态更新方式需要优化，但不是线程竞态

### 6.1 当前实现做了什么

`sql_execution_node` 在并发模式下使用 `ThreadPoolExecutor` 并发执行多条 SQL。

但需要注意：

- worker 线程只负责执行 SQL
- `sub_queries` 的 `status` 和 `execution_result` 更新发生在主线程遍历 `future.result()` 时

因此，当前不是多个线程同时对 `state["sub_queries"]` 做并发写。

### 6.2 原始问题描述为何不准确

把当前问题描述成“多个线程同时对共享列表中的字典项做 in-place 修改”是不准确的。

现状更接近：

- 多线程只负责 I/O 执行
- 主线程汇总结果后，顺序写回 state

所以这里不应上升为“线程安全 bug”。

### 6.3 当前真实问题

这里真正值得关注的是：

- 多个节点都依赖对 `sub_queries` 列表中的对象做 `in-place` 更新，再显式把整个 `sub_queries` 返回给 LangGraph

受影响的节点不仅是 `sql_execution_node`，还包括：

- **`inject_params_node`**（`inject_params.py:117-118`）：直接修改 `sq[“dependencies_results”]` 和 `sq[“status”]`，再返回同一个列表引用
- **`check_completion_node`**（`check_completion.py:111-112, 135-136`）：直接修改 `sq[“status”]` 和 `sq[“error”]`，再返回同一个列表引用
- **`sql_execution_node`**（`sql_execution.py:221-225, 248-253`）：直接修改 `sq[“execution_result”]` 和 `sq[“status”]`，再返回同一个列表引用

三个节点使用了完全相同的模式：先 in-place 修改列表中字典项的字段，再返回 `state.get(“sub_queries”, [])` 这个同一引用。

这种写法在当前代码里是可工作的，但存在几个工程风险：

1. 可读性较差
   - 读代码时不容易分辨哪些状态是原地修改、哪些是返回覆盖
2. 与”函数式状态更新”风格不一致
   - 会增加未来维护负担
3. 对持久化和调试不够友好
   - 尤其在后续继续扩展复杂路径时，状态来源更难追踪
4. 模式散布范围广
   - 三个节点同时使用这种模式，如果未来某个节点改了而其他没改，行为一致性会被破坏

### 6.4 问题定性

这个问题建议定性为：

- **中优先级工程质量问题**
- **不是当前并发执行的致命线程安全缺陷**

### 6.5 修复建议

建议对三个节点统一改为函数式更新：

- 先根据执行结果构造新的 `sub_queries` 副本（深拷贝或构造新字典）
- 再一次性返回新的列表

三个节点应一起改，避免出现部分节点改了、部分没改的不一致状态。

统一改造的好处：

- 与当前父图”覆盖模式 state”更一致
- 更容易测试
- 更容易迁移到后续更复杂的执行编排
- 三个节点行为一致，降低维护认知负担

## 7. 问题四：`complex_path_enabled` 配置开关未真正生效

### 7.1 问题说明

配置文件中存在：

- `graph_control.complex_path_enabled: false`

从配置意图看，这表示：

- 当前 Complex Path 应当默认关闭

但实际代码中的 `route_by_complexity` 只判断：

- `complexity == "simple"` -> Fast Path
- 否则 -> `planner`

也就是说：

- 只要 Router 判定为 `complex`
- 系统就仍然会进入 Complex Path

配置开关并没有被真正使用。

### 7.2 影响

这个问题比前面很多“潜在风险”更直接，因为它会导致：

- 配置与实际行为不一致
- 复杂路径即使声明为关闭，仍然可能被用户流量触发
- 测试、灰度、回退都会失去可靠性

### 7.3 问题定性

这个问题应定性为：

- **高优先级真实 bug**

### 7.4 修复建议

建议在父图路由层显式读取 `graph_control` 配置：

- 若 `complex_path_enabled` 为 `false`
  - Router 判定为 complex 时，不进入 `planner`
  - 应直接走一个受控的 fallback 分支

fallback 可选方案：

1. 直接进入 `Summarizer`，返回“复杂问题暂未开启”类提示
2. 降级为 Fast Path 尝试
3. 后续接入新的 `deepagents` complex orchestrator

在当前阶段，建议优先采用第一种，行为最清晰。

---

## 8. 当前问题的优先级排序

建议按以下顺序处理：

### P0

- 修复 `complex_path_enabled` 未生效问题

理由：

- 这是配置与实际行为不一致的直接 bug
- 会影响灰度与回退

### P1

- 修复依赖注入链路不闭环问题

理由：

- 这是真正影响 Complex Path 正确性的关键缺陷
- 也是多步依赖查询最核心的问题

### P2

- 增强 `Check Completion` 的失败分类与诊断
- 补充 failed 依赖的即时传播（避免不必要的延迟轮次）

理由：

- 能提升当前静态路径的可解释性
- 减少无意义的循环轮次
- 为后续动态重规划接入打基础

### P3

- 将 `inject_params_node`、`check_completion_node`、`sql_execution_node` 三个节点统一改为函数式状态更新

理由：

- 不是当前最直接的错误来源
- 但三个节点使用相同的 in-place 修改模式，应统一改造以降低维护复杂度

## 9. 对 DeepAgents 关系的重新判断

这些问题并不自动等价于“必须立即引入 DeepAgents”。

更准确的判断是：

### 9.1 哪些问题应先在现有架构中修掉

以下问题不应等待 DeepAgents 才修：

- `complex_path_enabled` 配置未生效
- 依赖注入链路不闭环
- 复杂路径成功链路缺乏完整端到端验证

这些属于当前代码本身应该补齐的基础能力。

### 9.2 哪些问题确实适合 DeepAgents 介入

以下问题更适合在后续由 DeepAgents 提供增强：

- 中间结果驱动的动态重规划
- 空结果后的补充任务生成
- 复杂问题执行过程中的目标重估

也就是说：

- **DeepAgents 适合解决“动态规划不足”**
- **但不应被当成修补当前显式 bug 的借口**

---

## 10. 建议的下一步动作

建议按如下顺序推进：

1. 修复 `complex_path_enabled` 未生效问题
2. 补一条真正的 Complex Path 成功 E2E 测试
3. 让子图前半段能消费依赖结果，补齐依赖注入链路
4. 优化 `Check Completion` 的失败分类与 failed 依赖即时传播
5. 再评估是否引入 DeepAgents 负责动态规划层

如果这些基础问题不先处理，即使接入 DeepAgents，也会遇到两个麻烦：

- 旧复杂路径无法作为稳定 fallback
- 新复杂路径效果无法被客观对比

---

## 11. 最终结论

当前父图在 Complex Path 上确实存在问题，但应当做更准确的分类：

- **真实 bug**
  - `complex_path_enabled` 配置未生效
  - 依赖结果没有完整进入子图前半段，导致依赖注入链路不闭环

- **设计刚性**
  - `Check Completion` 更像静态 DAG 收敛器，不具备动态重规划能力

- **工程优化项**
  - `inject_params_node`、`check_completion_node`、`sql_execution_node` 三个节点的状态更新方式不够函数式，但不是线程竞态 bug
  - `check_completion_node` 的依赖环检测不区分 failed 和 completed，导致 failed 依赖的下游子查询要多等一轮才被清理

因此，最合理的结论是：

**当前 Complex Path 需要先修基础问题，再引入 DeepAgents 做动态规划增强，而不是把两者混为同一个问题。**
