## 1. 背景与目标

### 1.1 背景

当前系统已实现从 `system.sem_object_vec` 表的 `table_category` 字段读取表分类，但存在以下问题：

1. **分类过于简单**：只区分事实表和维度表（二分类），桥接表被归入维度表
2. **缺少原始类型信息**：在提示词中无法展示表的实际类型（如"交易表"、"流水表"等）
3. **桥接表未特殊处理**：桥接表可能被选为 base 表，不符合最佳实践

### 1.2 改造目标

1. ✅ **支持三分类**：事实表、维度表、桥接表
2. ✅ **保留原始类型**：保存并展示 `table_category` 的实际值（如"维度表"、"交易表"、"桥接表"等）
3. ✅ **桥接表优化**：桥接表可以参与 JOIN，但不作为 base 表
4. ✅ **可配置映射**：通过配置文件定义哪些类型归属于哪一类
5. ✅ **提示词增强**：在提示词中增加"表类型"部分，按原始类型分组展示

### 1.3 核心设计理念

- **原始类型 vs 归类**：
  - **原始类型**：`system.sem_object_vec.table_category` 的实际值（如"维度表"、"交易表"、"流水表"）
  - **归类**：为了 JOIN 规划，将原始类型归类为 `fact` / `dimension` / `bridge` 三大类
  
- **使用场景**：
  - **JOIN 规划时**：使用归类（三分类）
  - **提示词展示时**：使用原始类型（保持细粒度）

## 2. 现状分析

### 2.1 当前实现

**代码位置**：`src/tools/schema_retrieval/retriever.py`

**当前逻辑**（第292-318行）：
```python
# 读取 table_category
category = col.get("table_category") or col.get("parent_category") or "维度表"

# 二分类判断
if category in ["fact", "事实表"]:
    # 归入事实表
else:
    # 其他所有类型（包括 bridge）归入维度表
```

**问题**：
- ❌ 桥接表被当作维度表处理
- ❌ 原始类型信息丢失
- ❌ 不支持细化分类（流水表、快照表等）

### 2.2 数据库字段

`system.sem_object_vec.table_category` 字段说明：
- 类型：`varchar(64)`
- 可能的值：`fact`、`dimension`、`bridge`、`事实表`、`维度表`、`桥接表`、`交易表`、`明细表`等
- 该字段由数据灌库程序设置，反映业务对表的实际分类

**重要约束**：
- ⚠️ 查询 `table_category` 字段时，**必须**使用 `WHERE object_type='table'` 条件
- 这是项目规范，确保只获取表级别的分类信息
- 列（`object_type='column'`）的分类信息通过 JOIN 父表获取

## 4. 关键实施规范（必读）

### 4.0 两个核心规范

在开始实施前，请务必理解以下两个核心规范：

#### 规范 1：配置文件驱动，不使用硬编码默认值

**问题**：不能在代码中硬编码默认的类型列表。

**错误示例**：
```python
# ❌ 错误：硬编码默认值
fact_types = category_mapping.get("fact", ["fact", "事实表", "交易表", "明细表"])
```

**正确做法**：
```python
# ✅ 正确：完全依赖配置文件
fact_types = category_mapping.get("fact", [])
if not category_mapping:
    raise ValueError("配置错误：缺少 table_category_mapping 配置")
```

**原因**：
1. 确保配置文件的权威性
2. 避免配置与代码不一致
3. 便于运维人员通过配置文件扩展类型，无需修改代码

#### 规范 2：查询 table_category 必须使用 object_type='table' 条件

**项目规范**：所有查询 `system.sem_object_vec.table_category` 的 SQL，必须包含 `WHERE object_type='table'` 条件。

**示例**：
```sql
-- ✅ 正确
SELECT object_id, table_category
FROM system.sem_object_vec
WHERE object_type = 'table'  -- 必须包含
  AND ...

-- ❌ 错误：缺少 object_type 条件
SELECT object_id, table_category
FROM system.sem_object_vec
WHERE ...
```

**原因**：
1. `table_category` 字段只在表级别（`object_type='table'`）有意义
2. 列级别分类不单独查询：通过父表名查表级别的 `table_category`（`object_type='table'`），不从列对象读取
3. 这是项目的数据规范，确保数据查询的一致性

**当前实现的验证**：
- ✅ `pg_client.py` 的 `search_semantic_tables()` 方法已正确使用该条件（第50行）
- ✅ `_format_table_types()` 方法使用的数据来自 `search_semantic_tables()`，因此已满足规范
- ✅ 无需额外修改，但开发者必须遵守此规范

#### 1.0 性能优化：预处理类型映射表

**重要性**：⭐⭐⭐ 中等优化，减少重复计算

在 `SchemaRetriever.__init__()` 方法中增加预处理逻辑，避免每次调用 `_classify_table_category()` 时都进行小写转换和列表遍历：

```python
def __init__(self, config: Dict[str, Any]):
    """
    初始化 SchemaRetriever

    Args:
        config: 完整的子图配置（包含 schema_retrieval 部分）
    """
    self.config = config
    self.retrieval_config = config.get("schema_retrieval", {})

    # 初始化客户端
    self.pg_client = get_pg_client()
    self.neo4j_client = get_neo4j_client()
    self.embedding_client = get_embedding_client()

    # 🆕 性能优化：预处理类型映射表（小写化）
    category_mapping = self.retrieval_config.get("table_category_mapping", {})
    if not category_mapping:
        raise ValueError(
            "配置错误：缺少 schema_retrieval.table_category_mapping 配置。"
            "请在 sql_generation_subgraph.yaml 中添加表分类映射规则。"
        )

    # 将所有类型值预先转为小写，避免运行时重复转换
    self._category_mapping_lower = {}
    for category_group in ["fact", "dimension", "bridge"]:
        type_list = category_mapping.get(category_group, [])
        self._category_mapping_lower[category_group] = {t.lower() for t in type_list}

    logger.info(
        f"表分类映射已加载: "
        f"fact={len(self._category_mapping_lower['fact'])} 个类型, "
        f"dimension={len(self._category_mapping_lower['dimension'])} 个类型, "
        f"bridge={len(self._category_mapping_lower['bridge'])} 个类型"
    )
```

**优化效果**：
- ✅ 配置只读取一次（初始化时）
- ✅ 小写转换只进行一次（使用 set 提升查找速度 O(1)）
- ✅ 减少 `_classify_table_category()` 的运行时开销（每次分类调用都会受益）

#### 1.1 新增归类方法

在 `SchemaRetriever` 类中新增方法：

```python
def _classify_table_category(self, category: str) -> str:
    """
    根据原始 table_category 值归类到 fact/dimension/bridge

    Args:
        category: 原始 table_category 值（如"事实表"、"交易表"等）

    Returns:
        归类结果: "fact" | "dimension" | "bridge"

    注意：
        - 完全依赖配置文件 sql_generation_subgraph.yaml 中的 table_category_mapping
        - 使用预处理的映射表（_category_mapping_lower），提升性能
    """
    if not category:
        return "dimension"  # 空值默认归为维度表

    # 🆕 使用预处理的小写映射表（O(1) 查找）
    category_lower = category.lower()

    # 检查是否属于事实表
    if category_lower in self._category_mapping_lower["fact"]:
        return "fact"

    # 检查是否属于桥接表
    if category_lower in self._category_mapping_lower["bridge"]:
        return "bridge"

    # 检查是否属于维度表
    if category_lower in self._category_mapping_lower["dimension"]:
        return "dimension"

    # 如果在配置中都找不到，默认归为维度表
    qlog = with_query_id(logger, "")
    qlog.warning(f"表类型 '{category}' 未在配置中定义，默认归为维度表")
    return "dimension"
```

#### 1.2 修改 _collect_and_classify_tables 方法

**位置**：第260-331行

**修改要点**：
1. 增加 `bridge_from_columns` 列表
2. 增加 `semantic_bridge_tables` 列表
3. 增加 `table_categories` 字典（保存原始类型）
4. 🆕 构建 `table_category_map` 映射表（确保只从表级别数据获取分类，符合规范2）
5. 使用 `_classify_table_category` 方法进行归类

```python
def _collect_and_classify_tables(
    self,
    semantic_tables: List[Dict[str, Any]],
    semantic_columns: List[Dict[str, Any]],
    parse_result: Optional[Dict[str, Any]],
) -> Dict[str, Any]:
    """构建 CandidateSet，区分事实表/维度表/桥接表并附加维度值命中"""

    # 1) 维度值检索
    dim_value_hits = self._retrieve_dim_value_hits(parse_result)

    # 2) 通过维度值命中获取维度表
    dim_tables_from_values = []
    seen_dim_from_values = set()
    for hit in dim_value_hits:
        table_name = hit.get("dim_table")
        if not table_name:
            continue
        table_id = str(table_name)
        if "." not in table_id:
            table_id = f"public.{table_id}"
        if table_id not in seen_dim_from_values:
            dim_tables_from_values.append(table_id)
            seen_dim_from_values.add(table_id)

    # 3) 通过列命中收集父表（三分类）
    fact_from_columns: List[str] = []
    dim_from_columns: List[str] = []
    bridge_from_columns: List[str] = []  # 🆕

    # 🆕 仅使用表级查询结果获取分类（object_type='table'）
    table_category_map = {
        t.get("object_id"): (t.get("table_category") or t.get("category") or "")
        for t in semantic_tables
        if t.get("object_id")
    }

    for col in semantic_columns:
        parent_id = col.get("parent_id")
        if not parent_id:
            continue
        # 读取原始 table_category 值（仅从表级别数据获取）
        category = table_category_map.get(parent_id, "")
        
        # 归类逻辑（通过配置映射）
        category_group = self._classify_table_category(category)
        
        if category_group == "fact":
            if parent_id not in fact_from_columns:
                fact_from_columns.append(parent_id)
        elif category_group == "bridge":
            if parent_id not in bridge_from_columns:
                bridge_from_columns.append(parent_id)
        else:
            # 默认归为维度表
            if parent_id not in dim_from_columns:
                dim_from_columns.append(parent_id)

    # 4) 语义检索结果分类（三分类）
    semantic_fact_tables: List[str] = []
    semantic_dim_tables: List[str] = []
    semantic_bridge_tables: List[str] = []  # 🆕
    table_similarities: Dict[str, float] = {}
    table_categories: Dict[str, str] = {}  # 🆕 保存原始类型

    for table in semantic_tables:
        table_id = table.get("object_id")
        if not table_id:
            continue
        similarity = table.get("similarity")
        if similarity is not None:
            table_similarities[table_id] = float(similarity)

        # 读取并保存原始 table_category 值
        # 注意：semantic_tables 来自 search_semantic_tables()，已使用 object_type='table' 条件
        category = table.get("table_category") or table.get("category") or ""
        if category:
            table_categories[table_id] = category
        
        # 归类逻辑
        category_group = self._classify_table_category(category)
        
        if category_group == "fact":
            if table_id not in semantic_fact_tables:
                semantic_fact_tables.append(table_id)
        elif category_group == "bridge":
            if table_id not in semantic_bridge_tables:
                semantic_bridge_tables.append(table_id)
        else:
            if table_id not in semantic_dim_tables:
                semantic_dim_tables.append(table_id)

    # 5) 合并候选集合（保持顺序）
    candidate_fact_tables = list(dict.fromkeys(semantic_fact_tables + fact_from_columns))
    candidate_dim_tables = list(
        dict.fromkeys(dim_tables_from_values + dim_from_columns + semantic_dim_tables)
    )
    candidate_bridge_tables = list(dict.fromkeys(semantic_bridge_tables + bridge_from_columns))  # 🆕

    return {
        "candidate_fact_tables": candidate_fact_tables,
        "candidate_dim_tables": candidate_dim_tables,
        "candidate_bridge_tables": candidate_bridge_tables,  # 🆕
        "table_similarities": table_similarities,
        "table_categories": table_categories,  # 🆕
        "dim_value_hits": dim_value_hits,
    }
```

```python
def _retrieve_join_plans(
    self,
    candidate_fact_tables: List[str],
    candidate_dim_tables: List[str],
    candidate_bridge_tables: List[str],  # 🆕 新增参数
    table_similarities: Dict[str, float],
    parse_result: Optional[Dict[str, Any]],
) -> List[Dict[str, Any]]:
    """
    检索 JOIN 计划
    
    🆕 桥接表可以参与 JOIN，但不作为 base 表
    """
    # 所有表 = 事实表 + 维度表 + 桥接表
    all_tables = list(dict.fromkeys(
        candidate_fact_tables + candidate_dim_tables + candidate_bridge_tables  # 🆕
    ))
    
    if not all_tables:
        return []

    # 1) 维度表优化判断
    should_use_dim_only, dim_only_table = self._should_use_dimension_only(
        parse_result,
        candidate_fact_tables,
        candidate_dim_tables,
        table_similarities
    )

    if should_use_dim_only and dim_only_table:
        print(f"[维度表优化] 跳过 Neo4j 查询，使用单表: {dim_only_table}")
        return [{
            "base": dim_only_table,
            "targets": [],
            "edges": []
        }]

    # 2) 选择 base 表（优先级：事实表 > 维度表；桥接表不可作为 base）🆕
    qlog = with_query_id(logger, "")
    
    if candidate_fact_tables:
        # 优先：事实表作为 base
        base_tables = sorted(
            candidate_fact_tables,
            key=lambda t: table_similarities.get(t, 0.0),
            reverse=True,
        )
        qlog.debug("选择事实表作为 base")
    elif candidate_dim_tables:
        # 次优：维度表作为 base
        base_tables = self._select_best_dim_base(
            candidate_dim_tables,
            table_similarities
        )
        qlog.debug("选择维度表作为 base")
    else:
        # 既无事实表也无维度表（仅桥接表或为空）：不生成 JOIN 计划
        # 🆕 日志监控增强：明确记录只有桥接表的场景
        if candidate_bridge_tables:
            qlog.warning(
                f"仅检测到桥接表 {candidate_bridge_tables}，无法生成JOIN计划。"
                "桥接表不能作为 base 表，请检查查询意图或 Schema 设计。"
            )
        return []

    # 3) 对每个 base 表查询 JOIN 路径
    join_plans = []
    for base in base_tables[:1]:  # 只使用最优的 base
        targets = [t for t in all_tables if t != base]

        if not targets:
            print(f"[JOIN规划] 只有一个表 {base}，无需JOIN")
            join_plans.append({
                "base": base,
                "targets": [],
                "edges": []
            })
            continue

        print(f"[JOIN规划] 准备查询: base={base}, targets={targets}")
        try:
            path_map = self._plan_join_paths_for_base(
                base_table=base,
                target_tables=targets,
            )

            edges = []
            seen = set()
            for target in targets:
                path = path_map.get(target) or []
                for edge in path:
                    key = (edge["src_table"], edge["dst_table"])
                    if key not in seen:
                        edges.append(edge)
                        seen.add(key)

            join_plans.append({
                "base": base,
                "targets": targets,
                "edges": edges
            })
        except Exception as e:
            print(f"[JOIN规划] base={base} 查询失败: {e}")
            continue

    return join_plans
```

```python
def _format_table_types(self, schema_context: Dict[str, Any]) -> str:
    """
    格式化表类型（按原始 table_category 分组）
    
    数据来源：
        - schema_context["table_categories"] 字典
        - 该字典的数据来自 search_semantic_tables() 的查询结果
        - search_semantic_tables() 已使用 WHERE object_type='table' 条件
        - 因此这里的数据已经确保是表级别的 table_category
    
    示例输出：
        - 维度表：public.dim_store, public.dim_region
        - 交易表：public.fact_order_detail
        - 事实表：public.fact_sales_summary
        - 桥接表：public.bridge_store_region
    """
    # 获取 table_categories（来自 system.sem_object_vec WHERE object_type='table'）
    table_categories = schema_context.get("table_categories", {})
    if not table_categories:
        return "（无分类信息）"
    
    # 按原始类型分组
    type_groups: Dict[str, List[str]] = {}
    for table_id, category in table_categories.items():
        if category not in type_groups:
            type_groups[category] = []
        type_groups[category].append(table_id)
    
    # 格式化输出（按类型排序，确保输出稳定）
    lines = []
    for category in sorted(type_groups.keys()):
        tables = ", ".join(sorted(type_groups[category]))
        lines.append(f"- {category}：{tables}")
    
    return "\n".join(lines) if lines else "（无分类信息）"
```

## 9. 总结

### 9.1 改造亮点

1. ✅ **保留原始信息**：不丢失数据库中的细粒度分类
2. ✅ **逻辑清晰**：归类只用于 JOIN 规划，展示使用原始类型
3. ✅ **配置驱动**：映射规则通过配置文件管理，易于扩展
4. ✅ **严格配置**：缺少映射配置将直接报错，避免隐式回退
5. ✅ **状态最小化**：不增加 state 根级别字段，信息存储在 `schema_context` 中
6. ✅ **性能优化**：预处理类型映射表，减少运行时开销
7. ✅ **监控增强**：关键场景日志记录，便于问题排查

### 9.2 关键收益

| 改造点 | 改造前 | 改造后 |
|--------|--------|--------|
| 表分类 | 二分类（fact/其他） | 三分类（fact/dimension/bridge） |
| 原始类型 | 丢失 | 保留在 `table_categories` |
| 桥接表处理 | 当作维度表 | 单独处理，不作为 base |
| 提示词 | 只有表结构 | 增加表类型分组展示 |
| 可扩展性 | 硬编码 | 配置文件驱动 |

### 9.3 优化改进（v1.1 新增）

本版本在 v1.0 基础上增加了三项优化改进：

#### 9.3.1 性能优化：预处理类型映射表

**改进点**：在 `SchemaRetriever.__init__()` 中预处理类型映射表

**优化效果**：
- 配置只读取一次（初始化时），避免每次分类都读取
- 类型值预先转为小写并存储为 `set`，查找复杂度从 O(n) 降至 O(1)
 - 对于包含大量表的查询，性能提升为预估范围（约 10-20%），视数据量与检索规模而定

**适用场景**：
- Schema 中表数量较多（>50 个表）
- 频繁调用分类方法（每次检索都会分类所有表和列）

#### 9.3.2 日志监控增强：桥接表场景告警

**改进点**：在"只有桥接表"场景增加明确的警告日志
