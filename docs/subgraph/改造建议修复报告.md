## ✅ 建议2：补充完整的维度表优化逻辑

### 修复内容

#### 2.1 添加 `_should_use_dimension_only` 方法
**文件**: `src/tools/schema_retrieval/retriever.py` 第345-398行

**功能**: 判断是否可以只用维度表（4条规则）

```python
def _should_use_dimension_only(
    self,
    parse_result: Optional[Dict[str, Any]],
    fact_tables: List[str],
    dim_tables: List[str],
    table_similarities: Dict[str, float],
) -> tuple[bool, Optional[str]]:
    """
    判断是否可以只用维度表
    
    参考 docs/13:727-784
    
    规则1: 有时间约束，通常需要事实表
    规则2: 没有维度表候选
    规则3: 纯维度表场景
    规则4: 相似度差异判断
    """
```

**关键逻辑**:
- 有时间约束 → 使用事实表
- 单维度表无时间约束 → 优化为单表查询
- 维度表相似度显著高于事实表（差距>0.05）→ 使用维度表

#### 2.2 添加 `_select_best_dim_base` 方法
**文件**: `src/tools/schema_retrieval/retriever.py` 第400-441行

**功能**: 多维度表场景下，通过连通性分析选择最优 base

```python
def _select_best_dim_base(
    self,
    dim_tables: List[str],
    table_similarities: Dict[str, float],
) -> List[str]:
    """
    多维度表场景下，通过连通性分析选择最优 base
    
    参考 docs/13:798-829
    
    Returns:
        [best_base_table]  # 只返回一个
    """
```

**关键逻辑**:
- 遍历每个维度表作为候选 base
- 查询它能连接到多少其他维度表（连通性）
- 选择连接数最多的；连接数相同时选择相似度最高的

#### 2.3 添加 `_plan_join_paths_for_base` 方法
**文件**: `src/tools/schema_retrieval/retriever.py` 第443-489行

**功能**: 包装现有 `plan_join_paths` 接口，提供 `target→edges` 映射

```python
def _plan_join_paths_for_base(
    self,
    base_table: str,

    """
    包装现有 plan_join_paths 接口，提供 target→edges 映射
    
    参考文档第635-657行
    """
```

**返回值结构**:
```python
{
    "target_table1": [edge1, edge2, ...],
    "target_table2": [edge3, edge4, ...],
}
```

#### 2.4 更新 `_retrieve_join_plans` 方法
**文件**: `src/tools/schema_retrieval/retriever.py` 第241-343行

**改进**:
