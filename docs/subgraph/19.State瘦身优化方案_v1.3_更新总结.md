# State 瘦身优化方案 v1.3 更新总结

## 更新日期
2025-11-12

## 主要修订内容

### 1. ✅ C1 - join_plans 格式化方案（已修正）
**原方案问题**：转为简单字符串，无法支持多 base 表和完整提示词格式

**修正方案**：
- 保留完整结构（base + edges）
- 删除不必要字段：`cost`、`constraint_name`、`targets`
- 支持多 base 表场景（Base #1, Base #2...）
- 收益：**-6~8%**（降低至更保守估计）

### 2. ✅ C2 - table_cards 精简（已修正）
**用户确认**：代码中只使用 `text_raw` 和 `time_col_hint`

**修正方案**：
- ✅ 保留：`text_raw`（提示词描述）
- ✅ 保留：`time_col_hint`（时间过滤列名）
- ❌ 删除：`grain_hint`（未使用）
- ❌ 删除：`attrs`（未访问）
- 收益：**-35~40%**

### 3. ✅ C3 - dim_value_hits 去重（已修正）
**修正方案**：
- ✅ 按 `(dim_table, dim_col, key_value)` 去重
- ✅ 按 `score` 降序排序
- ❌ **不做 Top-K 限制**（检索层 YAML 配置控制）
- 收益：**-2~3%**（仅去重）

### 4. ✅ D1 - user_query（保留）
**用户决策**：不合并

**理由**：
- 父图会做复杂问题拆分
- `user_query` 复杂度可能远高于子查询
- 需要保留原始查询上下文

### 5. ✅ E1 - validation_history（已修正）
**修正方案**：
- ✅ 只保留最后 3 条
- ✅ 保留完整结构（包括完整 SQL 和 `explain_plan`）
- ❌ **不做 Hash 化或预览**
- 收益：**-5~10%**（仅在重试场景）

### 6. ❌ 删除 E3 - 序列化规约
**用户决策**：不实施

**理由**：保持 JSON 结构完整性，即便是空值也保留

### 7. ❌ 删除 E4 - similar_sqls 精简
**用户决策**：不实施

**理由**：保持原结构，不做精简

### 8. ✅ 已确认删除的检索中间数据
- `schema_context.columns` ✅
- `schema_context.tables` ✅
- `schema_context.candidate_fact_tables` ✅
- `schema_context.candidate_dim_tables` ✅
- `schema_context.candidate_bridge_tables` ✅
- `schema_context.table_similarities` ✅

**收益**：**-15~20%**
