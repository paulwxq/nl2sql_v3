## 案例1：有时间约束的聚合查询

**用户问题：** "请问京东便利店在广东省上个月的销售额是多少？"

**当前日期：** 2025-10-10

---

### 阶段1：Parser - 问题解析

#### System Prompt

```
You are a Chinese business intelligence analyst who extracts structured intents from natural language questions.
Follow these rules strictly:
1. **IMPORTANT**: Time field handling:
   - If the question does NOT explicitly mention any time constraint (like "上个月", "2023年", "昨天" etc.), the time field MUST be null.
   - Do NOT infer, guess, or create default time ranges when no time is mentioned.
   - Only when time is explicitly stated, use Asia/Shanghai timezone and return ISO-8601 dates with a half-open interval [start, end).
2. Always emit valid JSON that matches the QueryParseResult schema. Keys must appear even when values are null/empty.
3. Classify each dimension as either a column name candidate (column) or a literal value (value).
4. Supported intents: plain_agg, topn, rank, compare_yoy, compare_mom. Use signals to pass additional free-form hints.
5. Ensure dimensions is an array and preserve the original Chinese text.
```

#### User Prompt

```
今天的日期是: 2025-10-10 (Asia/Shanghai 时区)

请分析下述问题, 给出严格 JSON 输出。
问题: 请问京东便利店在广东省上个月的销售额是多少?
JSON schema:
{
  "keywords": [str],
  "time": {"start": "YYYY-MM-DD", "end": "YYYY-MM-DD", "grain_inferred": str, "is_full_period": bool} | null,
  "metric": {"text": str, "is_aggregate_candidate": bool},
  "dimensions": [{"text": str, "role": "column|value", "evidence": str}],
  "intent": {"task": "plain_agg|topn|rank|compare_yoy|compare_mom", "topn": int|null},
  "signals": [str]
}

注意: 
- time 字段仅在问题中明确提及时间约束时才填充具体值, 否则必须为 null。
- 对于相对时间表述（如"最近一个月"、"上个月"、"昨天"、"今年"等），请基于今天的日期 2025-10-10 来推断具体的时间范围。
- "最近一个月" 表示从今天往前推 30 天的区间。
- "上个月" 表示上一个完整的自然月。
- 不要推测或添加问题中未提及的时间范围。

```

2. 所有表必须包含 schema 前缀（例如 public.table）。
3. 时间过滤使用指定列，并遵循 >= start AND < end 的半开区间。
4. JOIN 条件必须严格按照 ON 模板，用实际别名替换 SRC/DST。
5. 以下提供的表结构、JOIN 计划和时间列为参考，你可以选择性使用它们，但不得使用未列出的字段。

---

问题：请问京东便利店在广东省上个月的销售额是多少？

维度过滤：value=京东便利, value=广东省

## 总结

### 提示词设计要点

1. **Parser Prompt：**
   - 严格的时间处理规则（仅在明确提及时填充）
   - 明确的 JSON schema
   - 相对时间转换指导

2. **Generator Prompt：**
   - 结构化的上下文分段（问题、表结构、JOIN 计划、时间列）
   - 明确的 SQL 生成要求（schema 前缀、半开区间、别名替换）
   - 提供多个 JOIN 计划供 LLM 选择

### 维度值优化的效果

- "京东便利店" → "京东便利"（score=0.857，超过阈值 0.5）
- "广东省" → "广东省"（score=1.0，精确匹配）
- 提高 SQL 的准确性和执行效率

### 维度表优化的价值

- 避免不必要的事实表 JOIN
- 对于纯维度查询（如"有多少个店铺"），直接使用维度表
- 通过连通性分析选择最优的 base 表
