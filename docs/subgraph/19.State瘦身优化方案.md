### 1.1 问题描述

从生产日志分析（`logs/sql_subgraph.log:1512-1940`）发现，`SQLGenerationState` 中存储了大量**中间数据**和**调试信息**，导致：

1. **State 体积过大**：单个查询的 State 快照约 430 行 JSON（~15KB）
2. **内存占用高**：LangGraph 检查点机制会序列化整个 State
3. **数据冗余**：多个节点重复存储相同或派生数据
4. **可维护性差**：State 混杂业务数据和调试信息

### 1.2 数据分布分析

| 字段分类 | 占比 | 典型字段 |
|---------|------|---------|
| Schema 上下文 | 60% | `schema_context.table_cards.attrs.sample` |
| 解析结果 | 15% | `parse_result` |
| 验证结果 | 12% | `validation_result.explain_plan` |
| 维度匹配 | 8% | `dim_value_hits`（含重复） |
| 核心业务数据 | 5% | `generated_sql`, `validated_sql` |

**结论**：约 **70% 的 State 数据是可删除或精简的**。

## 2. 数据分类与处理策略

### 💡 删除时机说明（v1.3 新增）

根据数据使用特征，采用**两阶段删除策略**：

#### 📌 阶段1：检索节点输出时删除（不写入 State）

**适用场景**：纯检索中间数据，下游节点完全不使用

**字段列表**：
- `schema_context.columns` ✅
- `schema_context.tables` ✅
- `schema_context.candidate_fact_tables` ✅
- `schema_context.candidate_dim_tables` ✅
- `schema_context.candidate_bridge_tables` ✅
- `schema_context.table_similarities` ✅

**实施方式**：在 `schema_retrieval_node` 的 `_slim_schema_context()` 函数中直接不写入这些字段。

**优势**：减少节点间传递的数据量，提升性能。

#### 📌 阶段2：统一瘦身口删除（检查点/输出前）

**适用场景**：仍参与同轮流程的过程字段或需要裁剪的字段

**字段列表**：
- `generated_sql`（验证节点使用后删除）
- `validation_history`（限制数量至最后3条）
- `validation_result.explain_plan`（内部使用后删除）
- `parse_result.keywords/signals/evidence`（调试字段）
- `join_plans.edges[].cost/constraint_name`（精简结构）
- `table_cards[].attrs`（精简结构）

**实施方式**：在 `slim_state_for_checkpoint()` 统一函数中处理，或在 `extract_output()` 输出前过滤。

**优势**：集中管理，统一控制，易于维护。

### 2.1 完全删除类（不影响业务逻辑）

#### ❌ 类型A：调试元数据

| 字段路径 | 理由 | 影响节点 |
|---------|------|---------|
| `schema_context.metadata.retrieval_time` | 纯性能统计，不参与决策 | 无 |
| `schema_context.metadata.table_count` | 冗余信息（可从 tables 数组计算） | 无 |
| `schema_context.table_similarities` | 检索排序后即失效，不参与决策 | 无 |
| `parse_result.keywords` | 仅用于调试解析质量 | 无 |
| `parse_result.signals` | 当前为空数组，未使用 | 无 |

**注**：`columns[].similarity` 已随 `columns` 字段整体删除（见类型C）。

**处理方式**：在各节点输出前**过滤删除**，只保留在日志中。

#### ❌ 类型B：一次性使用数据

| 字段路径 | 使用场景 | 删除时机 |
|---------|---------|---------|
| `schema_context.table_cards[].attrs.sample` | 仅在 SQL 生成提示词中展示 | `sql_generation` 节点输出前 |
| `validation_result.explain_plan` | 仅在 `validation` 节点内部判断性能警告 | 验证完成后 |
| `parse_result.dimensions[].evidence` | LLM 解析的推理过程，不参与 SQL 生成 | `question_parsing` 节点输出前 |

**处理方式**：节点**内部使用，不写入 State**。

#### ❌ 类型C：检索中间数据

| 字段路径 | 理由 | 影响节点 |
|---------|------|---------|
| `schema_context.columns` | 列级检索结果，未在提示词中使用 | 无 |
| `schema_context.tables` | 原始表列表，已通过 `table_cards` 体现 | 无 |
| `schema_context.candidate_fact_tables` | `schema_retrieval` 内部选择 base 表的中间数据 | 无 |
| `schema_context.candidate_dim_tables` | 同上，最终结果已体现在 `join_plans` 中 | 无 |
| `schema_context.candidate_bridge_tables` | 同上，桥接表信息已体现在 `join_plans` 中 | 无 |
| `schema_context.table_similarities` | 检索排序后即失效，不参与 SQL 生成 | 无 |

**分析依据**（基于代码审查）：
- **`sql_generation.py` 实际使用的字段**：仅读取 `table_cards`、`join_plans`、`dim_value_hits`、`table_categories`、`similar_sqls`
- **`candidate_*` 和 `table_similarities`**：仅在 `retriever.py` 内部使用，用于选择 base 表和规划 JOIN 路径，最终结果已合并到 `join_plans` 和 `table_categories`
- **`columns` 和 `tables`**：在当前提示词构建逻辑中未被消费

**处理方式**（v1.3 明确）：
- ✅ **推荐**：在 `schema_retrieval_node` 输出时**直接不写入**这些字段（§2.0 阶段1）
- ⚠️ 备选：在统一 `slim_state_for_checkpoint()` 中移除（§2.0 阶段2）

**删除时机**：属于**纯检索中间数据**，建议在检索节点侧直接不写入 State。

**收益**：预计减少 State 体积 **15-20%**（约 60-80 行 JSON）。

### 2.2 精简压缩类（保留关键信息）

#### 🔧 类型C：结构化数据压缩

##### C1. `schema_context.join_plans`

**当前问题**：
```json
"join_plans": [
  {
    "base": "public.fact_store_sales_day",
    "targets": ["public.dim_store", "public.dim_company", ...],
    "edges": [
      {
        "src_table": "...",
        "dst_table": "...",
        "constraint_name": "fk_sales_day_store",
        "join_type": "INNER JOIN",
        "cardinality": "N:1",
        "on": "SRC.store_id = DST.store_id",
        "cost": 1.0
      },
      // ... 5-10 条边的完整结构
    ]
  }
]
```

- **体积**：~50 行（占 State 15%）
- **使用场景**：`sql_generation` 节点格式化为提示词中的 JOIN 计划

**实际提示词格式**（用户确认）：
```
JOIN 计划：
Base #1：**public.fact_store_sales_day**
- public.fact_store_sales_day --INNER JOIN (N:1)--> public.dim_product_type ON SRC.product_type_id = DST.product_type_id
- public.fact_store_sales_day --INNER JOIN (N:1)--> public.dim_store ON SRC.store_id = DST.store_id
- public.dim_store --INNER JOIN (N:1)--> public.dim_company ON SRC.company_id = DST.company_id
```

**优化方案**（修订版）：

```python
# 压缩为精简结构，删除不必要的字段，保留提示词生成所需的全部信息
"join_plans": [
  {
    "base": "public.fact_store_sales_day",
    "edges": [
      {
        "src_table": "public.fact_store_sales_day",
        "dst_table": "public.dim_store",
        "join_type": "INNER JOIN",
        "cardinality": "N:1",
        "on": "SRC.store_id = DST.store_id"
        # cost 删除（内部使用）
        # constraint_name 删除（不在提示词中使用）
      }
    ]
  }
]
```

**删除字段**：
- `edges[].cost`：内部排序使用，对提示词生成无用
- `edges[].constraint_name`：不在提示词中展示
- `targets`：可从 edges 推导，冗余

**收益**：减少 **20-30 行**，体积降低 **6-8%**。

**注意**：保留完整结构以支持多 base 表场景（Base #1, Base #2...）。

##### C3. `dim_value_hits`（去重）

**当前问题**：
```json
"dim_value_hits": [
  {"dim_table": "dim_store", "key_value": "101", "matched_text": "京东便利天河岗顶店", "score": 0.4},
  {"dim_table": "dim_store", "key_value": "101", "matched_text": "京东便利天河岗顶店", "score": 0.4},  // 重复
  {"dim_table": "dim_store", "key_value": "103", "matched_text": "京东便利南京新街口店", "score": 0.36},
  {"dim_table": "dim_store", "key_value": "103", "matched_text": "京东便利南京新街口店", "score": 0.36},  // 重复
  // 7 条记录，实际只有 3 个唯一匹配
]
```

**优化方案**（用户确认）：

```python
# 1. 按 (dim_table, dim_col, key_value) 去重（保留最高分）
# 2. 按 score 降序排序
# 注意：不做 Top-K 限制，因为检索时已在 YAML 配置中控制最大值
"dim_value_hits": [
  {"dim_table": "dim_company", "dim_col": "company_name", "key_value": "1", 
   "matched_text": "京东便利", "score": 0.8, "source_text": "京东便利店"},
  {"dim_table": "dim_region", "dim_col": "city_name", "key_value": "4401", 
   "matched_text": "广州市", "score": 0.5, "source_text": "广州"}
]
```

**配置说明**：
```yaml
# src/modules/sql_generation/config/sql_generation_subgraph.yaml
schema_retrieval:
  dim_value_matching:
    max_hits_per_dimension: 5  # 检索时控制最大返回数量
```

**收益**：减少 **3-5 行**（仅去重），体积降低 **2-3%**，避免干扰 SQL 生成。

**注意**：数量控制在检索层完成，State 层只负责去重。

### 2.4 进一步优化类（激进瘦身）

#### 🔪 类型E：批量裁剪与序列化优化

##### E1. `validation_history` 裁剪

**当前问题**：
```json
"validation_history": [
  {
    "iteration": 1,
    "sql": "SELECT SUM(fssd.amount) AS total_sales FROM public.fact_store_sales_day fssd INNER JOIN ... WHERE ...",  // 完整SQL，50-200行
    "result": {
      "valid": true,
      "errors": [],
      "warnings": ["检测到顺序扫描（Seq Scan），可能影响性能"],
      "layer": "all_passed",
      "explain_plan": "Aggregate  (cost=39.61..39.62 rows=1 width=32)\n  ->  Nested Loop  ..."  // 长字符串
    }
  },
  // ... 可能有 10+ 条记录
]
```

**问题分析**：
- 每条历史存储**完整 SQL**（50-200 行）
- 未限制数量（理论上可达 10+ 条）

**优化方案**（用户确认）：

```python
# 只保留最后 3 条记录，保持结构不变
"validation_history": [
  {
    "iteration": 1,
    "sql": "SELECT SUM(fssd.amount) AS total_sales FROM ...",  // 保留完整 SQL
    "result": {
      "valid": true,
      "errors": [],
      "warnings": ["检测到顺序扫描（Seq Scan），可能影响性能"],
      "layer": "all_passed",
      "explain_plan": "Aggregate  (cost=39.61..39.62 rows=1 width=32)\n  ->  Nested Loop  ..."  // 保留
    }
  }
  // 最多保留最近 3 条
]
```

**收益**：
- 当迭代次数 > 3 时，减少 **50-200 行**（取决于 SQL 长度）
- 体积降低 **5-10%**（重试场景）

**实现建议**：
```python
def _limit_validation_history(history: List[Dict], max_entries: int = 3) -> List[Dict]:
    """限制验证历史数量"""
    # 只保留最近 N 条，结构保持不变
    return history[-max_entries:] if len(history) > max_entries else history
```

**注意**：保留完整结构和 `explain_plan`，只限制数量。

### 2.5 预格式化策略（混合方案）⭐

#### 🎯 核心思想

**问题**：当前 `schema_context` 存储大量结构化数据，在 `sql_generation_node` 中需要二次格式化才能用于提示词。

**解决方案**：在 `schema_retrieval_node` 输出时，**对简单数据进行预格式化**，同时**保留关键元数据**供逻辑判断使用。

#### 📋 数据分类与处理策略

| 数据类型 | 当前存储 | 优化策略 | 理由 |
|---------|---------|---------|------|
| **JOIN 路径** | 完整边列表（50行） | ✅ 预格式化字符串（5行） | 简单、无歧义、直接用于提示词 |
| **维度过滤** | dim_value_hits 完整对象（15行） | ✅ 预格式化 SQL 片段（3行） | 减少 sql_generation_node 复杂度 |
| **时间过滤** | parse_result.time（5行） | ✅ 预格式化 SQL 片段（1行） | 同上 |
| **表描述** | text_raw（已是字符串） | ⚠️ 保持不变 | 无需二次格式化 |
| **时间列名** | time_col_hint | ❌ 保留结构 | 需要用于生成时间过滤 |
| **候选表** | candidate_fact_tables | ❌ 保留结构 | 需要用于主表选择逻辑 |

#### 实施细节

##### 变更1：新增 `prompt_segments` 字段

**位置**：`schema_context` 下新增字段

**结构**：
```python
"prompt_segments": {
    "join_paths": [  # 预格式化的 JOIN 路径
        "fact_store_sales_day -> dim_store (store_id) -> dim_company (company_id)",
        "fact_store_sales_day -> dim_store (store_id) -> dim_region (region_id)"
    ],
    "dim_filters": "dc.company_name = '京东便利' AND dr.city_name = '广州市'",  # 预格式化的维度过滤
    "time_filter": "fssd.date_day >= '2025-09-01' AND fssd.date_day < '2025-09-05'",  # 预格式化的时间过滤
}
```

**优势**：
1. `sql_generation_node` 可以直接使用，无需格式化
2. 重试时不需要重新格式化
3. State 体积减少（删除原始 dim_value_hits 的冗余字段）

##### 变更2：`join_plans` → `join_paths`

**当前**：
```json

    {
        "base": "fact_store_sales_day",
        "edges": [
            {"src_table": "...", "dst_table": "...", "on": "...", "cost": 1.0},
            // ... 10+ 条边
        ]
    }

```

**优化后**：
```json
"join_paths": [
    "fact_store_sales_day -> dim_store (store_id) -> dim_company (company_id)",
    "fact_store_sales_day -> dim_store (store_id) -> dim_region (region_id)"
]
```

**注意**：这已经是预格式化的字符串，可以直接用于提示词。

##### 变更3：精简 `dim_value_hits`

**当前**：保留完整对象，包含所有字段（用于生成过滤条件）

**优化后**：
- 在 `schema_retrieval_node` 中生成 `prompt_segments.dim_filters`
- `dim_value_hits` 只保留 Top-3（用于日志和调试）
- 或者完全删除 `dim_value_hits`（如果 prompt_segments.dim_filters 已足够）

**代码示例**：

##### D1. `query` vs `user_query`（保留）

**当前问题**：
```json
"query": "请问京东便利店9月1日到9月5日在广州的销售额是多少？",
"user_query": "请问京东便利店9月1日到9月5日在广州的销售额是多少？"
```

**场景分析**：
- **单查询场景**（当前）：两者完全相同
- **复杂查询拆解场景**（架构支持）：
  - `query` = 子查询（拆解后的单个查询）
  - `user_query` = 原始查询（父图传入的完整查询）
  - `user_query` 复杂度可能远高于 `query`

**用户决策**（v1.3 更新）：
- ❌ **不合并**：保留 `user_query` 字段
- **理由**：父图会做复杂问题拆分，需要保留原始查询上下文

**收益**：无（保持现状）。

## 3. 实施方案

### 3.0 统一过滤架构（推荐优先实施）

#### 设计原则

**问题**：在各节点分散实现瘦身逻辑容易导致：
- 逻辑重复
- 遗漏某些节点
- 难以统一开关控制

**解决方案**：在 **LangGraph 检查点持久化前** 统一过滤（单点治理）。

#### 实现方案

**文件**：`src/modules/sql_generation/subgraph/state.py`

```python
from typing import Dict, Any, List
import hashlib

def slim_state_for_checkpoint(state: Dict[str, Any], profile: str = "slim") -> Dict[str, Any]:
    """
    统一 State 瘦身函数（在检查点持久化前调用）
    
    Args:
        state: 原始 State
        profile: 瘦身档位
            - "full": 不瘦身（调试用）
            - "slim": 标准瘦身（生产推荐）
            - "minimal": 极简模式（仅保留核心输出）
    
    Returns:
        瘦身后的 State
    """
    if profile == "full":
        return state
    
    if profile == "minimal":
        return _minimal_state(state)
    
    # 默认：slim 模式
    return _slim_state(state)


def _slim_state(state: Dict[str, Any]) -> Dict[str, Any]:
    """标准瘦身模式"""
    result = {}
    
    # ========== 1. 基础字段 ==========
    result["query"] = state.get("query")
    result["query_id"] = state.get("query_id")
    result["user_query"] = state.get("user_query")  # 保留（支持复杂查询拆解）
    
    # ========== 2. 解析结果（精简） ==========
    if state.get("parse_result"):
        result["parse_result"] = _slim_parse_result(state["parse_result"])
    
    result["parsing_source"] = state.get("parsing_source")
    
    # ========== 3. Schema 上下文（大幅精简） ==========
    if state.get("schema_context"):
        result["schema_context"] = _slim_schema_context(state["schema_context"])
    
    # ========== 4. SQL 生成结果 ==========
    # generated_sql 删除（只保留 validated_sql）
    result["validated_sql"] = state.get("validated_sql")
    result["iteration_count"] = state.get("iteration_count", 0)
    
    # ========== 5. 验证结果（精简） ==========
    if state.get("validation_result"):
        result["validation_result"] = _slim_validation_result(state["validation_result"])
    
    if state.get("validation_history"):
        result["validation_history"] = _limit_validation_history(state["validation_history"])
    
    # ========== 6. 错误信息 ==========
    if state.get("error"):
        result["error"] = state["error"]
    if state.get("error_type"):
        result["error_type"] = state["error_type"]
    
    result["execution_time"] = state.get("execution_time", 0.0)
    
    # ========== 7. 返回结果（保持JSON结构完整性） ==========
    return result


def _slim_parse_result(parse_result: Dict) -> Dict:
    """精简解析结果"""
    return {
        "time": parse_result.get("time"),
        "metric": parse_result.get("metric"),
        "dimensions": [
            {"text": d["text"], "role": d["role"]}  # 删除 evidence
            for d in parse_result.get("dimensions", [])
        ],
        "intent": parse_result.get("intent"),
        # keywords, signals 删除
    }


def _slim_schema_context(schema_context: Dict) -> Dict:
    """精简 Schema 上下文（v1.3 修订版）"""
    result = {}
    
    # ❌ 删除：tables, columns（检索中间数据）
    # ❌ 删除：candidate_fact_tables, candidate_dim_tables, candidate_bridge_tables（检索中间数据）
    # ❌ 删除：table_similarities（检索中间数据）
    # ❌ 删除：metadata（调试信息）
    
    # join_plans：精简结构（删除 cost/constraint_name/targets）
    join_plans = schema_context.get("join_plans", [])
    if join_plans:
        result["join_plans"] = [
            {
                "base": plan.get("base"),
                "edges": [
                    {
                        "src_table": edge["src_table"],
                        "dst_table": edge["dst_table"],
                        "join_type": edge["join_type"],
                        "cardinality": edge["cardinality"],
                        "on": edge["on"]
                    }
                    for edge in plan.get("edges", [])
                ]
            }
            for plan in join_plans
        ]
    
    # table_cards：只保留 text_raw 和 time_col_hint
    table_cards = schema_context.get("table_cards", {})
    result["table_cards"] = {
        table: {
            "text_raw": card.get("text_raw", ""),
            "time_col_hint": card.get("time_col_hint")
        }
        for table, card in table_cards.items()
    }
    
    # dim_value_hits：只去重，不做 Top-K
    dim_value_hits = schema_context.get("dim_value_hits", [])
    if dim_value_hits:
        result["dim_value_hits"] = _deduplicate_dim_hits(dim_value_hits)
    
    # table_categories：保留（用于提示词）
    result["table_categories"] = schema_context.get("table_categories", {})
    
    # similar_sqls：保留原结构
    result["similar_sqls"] = schema_context.get("similar_sqls", [])
    
    return result


def _slim_validation_result(validation_result: Dict) -> Dict:
    """精简验证结果"""
    return {
        "valid": validation_result["valid"],
        "errors": validation_result["errors"],
        "warnings": _normalize_warnings(validation_result.get("warnings", [])),
        "layer": validation_result["layer"],
        # explain_plan 删除
    }


def _limit_validation_history(history: List[Dict], max_entries: int = 3) -> List[Dict]:
    """限制验证历史数量（v1.3 修订版）"""
    # 只保留最近 N 条，结构保持不变
    return history[-max_entries:] if len(history) > max_entries else history


def _normalize_warnings(warnings: List[str]) -> List[str]:
    """规范化警告信息"""
    normalized = []
    for w in warnings:
        if "顺序扫描" in w or "Seq Scan" in w:
            normalized.append("SEQ_SCAN")
        elif "笛卡尔积" in w or "Nested Loop" in w:
            normalized.append("CARTESIAN_PRODUCT")
        else:
            normalized.append(w[:50])
    return normalized


def _deduplicate_dim_hits(hits: List[Dict]) -> List[Dict]:
    """维度匹配去重（v1.3 修订版）"""
    # 按 (dim_table, dim_col, key_value) 去重，保留最高分
    unique_hits = {}
    for hit in hits:
        key = (hit["dim_table"], hit["dim_col"], hit["key_value"])
        if key not in unique_hits or hit["score"] > unique_hits[key]["score"]:
            unique_hits[key] = hit
    
    # 按 score 降序排序
    result = sorted(unique_hits.values(), key=lambda x: x["score"], reverse=True)
    
    return result


def _minimal_state(state: Dict[str, Any]) -> Dict[str, Any]:
    """极简模式（仅保留核心输出）"""
    return {
        "query_id": state.get("query_id"),
        "validated_sql": state.get("validated_sql"),
        "error": state.get("error"),
        "error_type": state.get("error_type"),
        "iteration_count": state.get("iteration_count", 0),
    }
```

```python
def _deduplicate_dim_hits(hits: List[Dict]) -> List[Dict]:
    """
    维度匹配结果去重（v1.3 修订版）

    1. 按 (dim_table, dim_col, key_value) 去重（保留最高分）

    for hit in hits:
        key = (hit["dim_table"], hit["dim_col"], hit["key_value"])
        if key not in unique_hits or hit["score"] > unique_hits[key]["score"]:
            unique_hits[key] = hit
    
    # 按 score 降序排序
    result = sorted(unique_hits.values(), key=lambda x: x["score"], reverse=True)
    
    return result
```

**预期收益**：State 体积减少 **40-50%**。

## 9. 用户建议总结

### 9.1 核心价值

用户基于日志逐字段分析和实际需求，提出了以下**关键改进**：

1. **发现重复冗余**：`generated_sql` vs `validated_sql`（原方案未发现）
2. **优化历史存储**：`validation_history` 使用 Hash 引用而非完整 SQL（体积减少 5%）
3. **架构设计升级**：统一过滤函数 + 配置档位化（单点治理，易维护）
4. **白名单思路**：提供了清晰的目标 State 结构，指导实施
5. **预格式化策略**：⭐ 提出"贴近提示词格式产生数据"的混合策略，兼顾性能和灵活性
