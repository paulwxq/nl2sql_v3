重试机制说明：
  • 最大迭代次数：3次（配置：retry.max_iterations）
  • 重试条件：validation失败 + iteration_count < max_iterations
  • 不可恢复错误：schema_retrieval_failed、generation_failed → 直接结束
  • 错误反馈：重试时会将validation错误信息传递给LLM
条件边逻辑（should_retry函数）：
  • success：state.validated_sql存在 → 流程成功结束
  • fail：不可恢复错误 或 迭代次数达上限 → 流程失败结束
  • retry：验证失败 且 未超过最大迭代次数 → 返回sql_generation重试

3.1 向量检索步骤

数据源：system.sem_object_vec（表和列的向量嵌入）
  • 表检索：topk_tables=10, similarity_threshold=0.45
  • 列检索：topk_columns=10, similarity_threshold=0.45
  • 相似度计算：使用余弦相似度（pgvector: <=> 操作符）

  • 输出：dim_value_hits（包含query_value, matched_text, score, table_id, column_name等）

3.4 历史SQL检索

数据源：system.sql_embedding（历史成功SQL案例的向量嵌入）
  • 检索策略：使用问题向量检索相似的历史SQL
  • 参数：sql_embedding_top_k=3, sql_similarity_threshold=0.6
  • 输出：similar_sqls（包含question, sql, similarity等）

4.1 提示词构建（_build_prompt）

提示词包含以下部分（按顺序）：
  1. 系统要求：PostgreSQL专家，仅输出SQL，包含schema前缀等
  2. 问题信息：query（子查询文本）
  3. 时间信息：time_info（解析出的时间窗口和粒度）
  4. 维度过滤：dimension_filters（优化后的维度过滤条件）
  5. 指标信息：metric_info（指标文本）
  6. 依赖结果：dependencies_text（子查询依赖的其他查询结果）
  7. 表类型分组：table_categories_text（事实表、维度表、桥接表分组）
  8. 表结构：table_cards_text（表卡片，包含列信息）
  9. JOIN计划：join_plans_text（JOIN路径和ON条件模板）
  10. 时间列：time_columns_text（各表的时间列提示）
  11. 历史SQL案例：similar_sqls_text（最多2个相似案例，可选）
  12. 验证错误：errors_text（上一次验证失败的错误信息，重试时使用）

4.2 LLM调用重试机制

重试配置（llm_retry）：
  • 最大尝试次数：max_attempts=3
  • 重试间隔：initial_delay_ms=500（固定间隔，不做指数退避）
  • 触发条件：连接/超时异常 或 LLM返回空结果
  • 与图层重试的区别：这是节点内部重试（处理网络抖动），图层重试是validation失败后的重新生成

5.3 第3层：语义检查（Semantic Check）

检查方式：EXPLAIN（计划分析，不实际执行）
  • 数据库操作：在PostgreSQL上执行EXPLAIN命令
  • 超时配置：explain_timeout=5秒
  • EXPLAIN ANALYZE：explain_analyze=false（默认不实际执行）
  • 配置：enable_semantic_check=true
性能警告检测：
  • 顺序扫描警告：seq_scan_warn=true（检测Seq Scan节点）
  • 嵌套循环警告：nested_loop_warn=true（检测Nested Loop节点）
  • 预估行数阈值：estimated_rows_threshold=100000（超过阈值发出警告）
  • 警告处理：警告不阻止验证通过，仅记录在validation_result.warnings中

5.4 验证历史记录

validation_history结构：
  • 每次验证都会追加一条记录到validation_history（使用add reducer自动追加）
  • 记录包含：iteration（迭代次数）、sql（生成的SQL）、result（验证结果）
  • 用途：调试、分析迭代过程、提供错误反馈给LLM

validation_history: List[Dict]          # 所有验证历史（使用add reducer自动追加）

# ========== 输出字段 ==========
validated_sql: Optional[str]            # 最终验证通过的SQL（成功标志）
error: Optional[str]                    # 错误信息
error_type: Optional[str]               # 错误类型：parsing_failed | schema_retrieval_failed | generation_failed | validation_failed
execution_time: float                   # 执行耗时（秒）

# ========== 继承字段（MessagesState） ==========
messages: List[BaseMessage]             # 消息列表（用于LLM交互）

6.2 错误类型（error_type）

error_type枚举值：
  • parsing_failed：问题解析失败（不可恢复，直接结束）
  • schema_retrieval_failed：Schema检索失败（不可恢复，直接结束）
  • generation_failed：SQL生成失败（LLM不可用，不可恢复，直接结束）
  • validation_failed：SQL验证失败（可恢复，触发重试）

6.3 成功判断逻辑

判断标准：
  • 成功：state.validated_sql 存在（不为None）
  • 失败：state.validated_sql 为None 且 state.error_type 存在
  • 输出清理：成功时自动清理error和error_type字段，避免"成功+历史错误"并存

6.4 State输出结构（extract_output）

最终输出字典：

{

  "error": None,                         // 错误信息（成功时为None）
  "error_type": None,                    // 错误类型（成功时为None）
  "iteration_count": 2,                  // 迭代次数
  "execution_time": 3.45,                // 执行耗时（秒）
  "schema_context": {...},               // Schema上下文（用于调试）
  "validation_history": [...]            // 验证历史（用于调试）
}
