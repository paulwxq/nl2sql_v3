## 时间解析与 SQL 生成改造方案：兼容日期与日期时间

### 背景与目标
- 背景：当前 `time` 字段仅约定为日期字符串（YYYY-MM-DD），日志与单测均只覆盖“日期”场景，无法准确表达小时级/分钟级的时间窗口。
- 目标：在保持 `time` 作为唯一时间字段名的前提下，同时支持“日期”和“日期时间”，使如下请求被精确表达并下推到 SQL：
  - 例：请帮我查询 2025-11-01 12:00 到 2025-11-03 18:00 的销售数据（小时粒度）。
- 非目标：
  - 不引入并行的 `datetime` 字段名；不在 SQL 中通过函数转换（如 `DATE(datetime_col)）`掩盖类型不匹配问题（以免失去索引和语义精度）。

### 现状摘要
- 解析提示词对 `time.start/end` 的约束为日期格式：

```51:56:src/modules/sql_generation/subgraph/nodes/question_parsing.py
"JSON schema:\n{{\n"
"  \"keywords\": [str],\n"
"  \"time\": {{\"start\": \"YYYY-MM-DD\", \"end\": \"YYYY-MM-DD\", \"grain_inferred\": str, \"is_full_period\": bool}} | null,\n"
"  \"metric\": {{\"text\": str, \"is_aggregate_candidate\": bool}},\n"
"  \"dimensions\": [{{\"text\": str, \"role\": \"column|value\", \"evidence\": str}}],\n"
```

- SQL 生成仅转述时间窗口文本，不区分日期与日期时间：

```192:203:src/modules/sql_generation/subgraph/nodes/sql_generation.py
def _format_time_hints(self, parse_result: Optional[Dict[str, Any]]) -> str:
    if not parse_result or "time" not in parse_result:
        return ""
    time_window = parse_result["time"]
    start = time_window.get("start", "")
    end = time_window.get("end", "")
    if start and end:
        return f"时间窗口：{start} ~ {end}"
    return ""
```

### 设计决策
- 保留字段名 `time`，解析仅基于“问题文本”识别时间范围与粒度；不感知表结构/字段类型/时区（后续扩展）。
- 扩展 `grain_inferred` 用于表达粒度：`year | quarter | month | week | day | hour`（`minute|second` 留作后续扩展）；按输入语义自动推断（如“11月1日12点以后”→ `hour`）。
- 单边/区间条件推断：仅依据 `start/end` 的存在性确定条件类型（无需额外枚举字段）。
  - 同时存在 `start` 与 `end` → 区间条件：`>= {start} AND < {end}`（半开区间）；
  - 仅存在 `start` → 下界条件：`>= {start}`；
  - 仅存在 `end` → 上界条件：`< {end}`。
- `is_full_period` 可选：当问题语义明确为完整自然周期时为 `true`，否则 `false` 或省略。
- 后续扩展项（本期不做）：时间标准化（补零/时区）、中文时间词归一、列类型选择与校验、参数化绑定与错误码体系。

### 接口与 Schema 变更（向后兼容）
- 解析结果中的 `time` 字段：
  - `start`/`end`: `"YYYY-MM-DD"` 或 `"YYYY-MM-DD HH:MM:SS"`（字符串原样透传，不做标准化；时间部分统一使用 `HH:MM:SS` 格式）。
  - `grain_inferred`: `year|quarter|month|week|day|hour`（按问题文本推断）。
  - `is_full_period`: 可选布尔值。
  - 条件类型推断：仅根据 `start/end` 的存在性判断为区间/仅下界/仅上界（见设计决策），**无需额外字段标识**。

示例（区间 + 小时粒度）：
```json
{
  "time": {
    "start": "2025-11-01 12:00:00",
    "end": "2025-11-03 18:00:00",
    "grain_inferred": "hour",
    "is_full_period": false
  }
}
```

示例（仅下界 + 日粒度）：
```json
{
  "time": {
    "start": "2025-11-01",
    "grain_inferred": "day"
  }
}
```

示例（仅上界 + 日粒度）：
```json
{
  "time": {
    "end": "2025-11-09",
    "grain_inferred": "day"
  }
}
```

向后兼容策略：如果仅产出 `start/end` 与可选 `grain_inferred/is_full_period`，维持既有行为；区间/单边条件由 `start/end` 的存在性直接确定。

#### is_full_period 判定规则（基于问题文本，不做日历计算）
- 判定依据：仅根据用户表述是否指向“完整自然周期”，不进行日历边界推断或校验（本期不做）。
- 触发用语示例：
  - 年：”全年“/“今年/2025年/上一年”等 → `grain_inferred=year`, `is_full_period=true`
  - 季：”Q1/第一季度/全季度“等 → `grain_inferred=quarter`, `is_full_period=true`
  - 月：”9月份/整个9月/本月/上月“等 → `grain_inferred=month`, `is_full_period=true`
  - 周：”本周/上周/整周“等（按 ISO 周，周一为起始）→ `grain_inferred=week`, `is_full_period=true`
  - 日：”当天/整日“等 → `grain_inferred=day`, `is_full_period=true`
- 明确不触发：给出具体起止但非“整期”表述，如“9月1日到9月30日”“11月1日12点到11月3日18点” → `is_full_period=false`。
- 单边条件：仅 `start` 或仅 `end` 不构成整期 → `is_full_period=false`。
- 备注：
  - `is_full_period=true` 仅表示“语义为整期”，`start/end` 仍按原样透传；是否对齐到自然边界（如月末/下月1日）属于后续扩展。
  - SQL 生成仍使用半开区间；本期不对 `start/end` 做日历边界标准化。

#### 解析提示词（System/User）建议文案（仅改解析节点提示词，其它提示词不动）

       - If a day-of-month appears and no clock time (e.g., "2025-11-08", "11月8日"), choose day.

### SQL 生成层改造
1. 提示词透传（不做类型推断/校正）：
   - 新增两行提示：
     - `日期时间粒度：{grain_inferred}`
     - `日期时间条件：{仅根据 start/end 存在性拼接：仅 start => >=start；仅 end => <end；start+end => >=start AND <end}`
   - 不再输出“日期时间窗口：{start} ~ {end}”。
2. 半开区间说明：保留于提示词规范中，但仅用于 `start+end` 的条件拼接；单边条件按规则直接使用。

### 验证与错误处理
- 轻量验证（本期仅限解析输出一致性）：
  - 仅 `start`：视为下界条件；
  - 仅 `end`：视为上界条件；
  - 同时具备 `start/end`：视为区间条件，要求 `start < end`（字符串层面按常识检查，不做时区/类型校验）。
