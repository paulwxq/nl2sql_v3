📝 子图输入格式说明

必需输入字段

字段名 类型 必需 说明
query str ✅ 必需 子查询文本（自然语言问题）
query_id str ✅ 必需 查询唯一标识符（用于日志追踪）
user_query str ✅ 必需 用户原始查询（可能与query相同）
dependencies_results Dict[str, Any] 可选 依赖查询的结果（如果当前查询依赖其他查询）
格式: {"q1": [结果数据], "q2": {...}}
parse_hints Optional[Dict] 可选 外部传入的解析提示（如果提供，跳过内部LLM解析）

parse_hints / parse_result 结构（可选，但推荐）

如果提供 parse_hints，子图会跳过问题解析步骤，直接使用提供的结构化信息。

parse_hints 完整结构示例：

{
  "keywords": ["销售额", "2024年", "总计"],

  "time": {
    "start": "2024-01-01",
    "end": "2025-01-01",
    "grain_inferred": "年",
    "is_full_period": true
  },

  "metric": {
    "text": "销售额",
    "is_aggregate_candidate": true
  },

  "dimensions": [
    {
      "text": "区域",
      "role": "column",
      "evidence": "问题中提到区域分组"
    },
    {
      "text": "华东",
      "role": "value",
      "evidence": "具体区域名称"
    }
  ],

  "intent": {
    "task": "plain_agg",
    "topn": null
  },

  "signals": ["聚合查询", "时间过滤"]
}

字段说明：

  • keywords: 提取的关键词列表
  • time: 时间约束（如果问题不涉及时间，设为 null）
    • start: 开始日期（YYYY-MM-DD）
    • end: 结束日期（YYYY-MM-DD）
    • grain_inferred: 时间粒度（年/季/月/日）
    • is_full_period: 是否完整周期
  • metric: 指标信息
    • text: 指标名称（销售额、订单数等）
    • is_aggregate_candidate: 是否需要聚合
  • dimensions: 维度列表
    • text: 维度文本
    • role: "column"（列名候选）或 "value"（字面值）
    • evidence: 判断依据
  • intent: 查询意图
    • task: plain_agg（普通聚合）| topn（排名）| rank（排序）| compare_yoy（同比）| compare_mom（环比）
    • topn: Top-N 数量（如果是 topn 任务）
  • signals: 语义信号列表

    run_sql_generation_subgraph

    query="查询2024年总销售额",
    query_id="q1",

    print(f"类型: {output['error_type']}")

📋 节点详细说明

1️⃣ 问题解析节点 (question_parsing_node)

属性 说明
代码位置 src/modules/sql_generation/subgraph/nodes/question_parsing.py
类型 Agent（调用LLM）
实现类 QuestionParsingAgent
使用模型 qwen-plus（通义千问）
主要功能 将自然语言问题解析为结构化的 parse_result，包括：
• keywords: 关键词列表
• time: 时间窗口（start, end, grain_inferred, is_full_period）
• metric: 指标信息（text, is_aggregate_candidate）
• dimensions: 维度列表（text, role, evidence）
• intent: 意图（task, topn）
• signals: 信号列表
处理逻辑 1. 优先使用外部传入的 parse_hints（parsing_source="external"）
2. 如果配置禁用内部解析，返回空结果（parsing_source="disabled"）
3. 调用LLM进行问题解析（parsing_source="llm"）
4. 解析失败时，如果允许fallback，返回空结果继续流程（parsing_source="fallback"）
输出字段 • parse_result: Dict[str, Any]
• parsing_source: "external" | "llm" | "disabled" | "fallback"
• error (可选): 错误信息
• error_type (可选): "parsing_failed"
配置参数 • parser_model: qwen-plus
• temperature: 0
• max_tokens: 1500
• timeout: 20s
• enable_internal_parser: true
• fallback_to_empty: true

2️⃣ Schema检索节点 (schema_retrieval_node) - 详细流程

属性 说明
代码位置 • 节点: src/modules/sql_generation/subgraph/nodes/schema_retrieval.py
• 实现: src/tools/schema_retrieval/retriever.py
类型 Tool（确定性函数，不调用LLM）
实现类 SchemaRetriever

🔍 步骤1: 生成查询向量

  • 实现：embedding_client.embed_query(query)
  • 逻辑：如果输入已包含 query_embedding，则复用；否则调用嵌入服务生成
  • 输出：向量（List[float]，通常768或1024维）

🔍 步骤2: 向量检索（PostgreSQL + pgvector）

子步骤 方法 数据源 参数
2a. 检索相关表 search_semantic_tables system.sem_object_vec
（表的语义向量）		 • topk: 10
• threshold: 0.45
2b. 检索相关列 search_semantic_columns system.sem_object_vec
（列的语义向量）		 • topk: 10
• threshold: 0.45

返回数据包含：object_id（表/列名）、similarity（相似度）、table_category（事实表/维度表）、grain_hint（粒度）、time_col_hint（时间列）等元信息

🔍 步骤3: 分类与维度值匹配

子步骤 说明
3a. 维度值检索 条件：parse_result.dimensions 中包含 role="value" 的维度
方法：search_dim_values(query_value, topk=5)
数据源：system.dim_value_index（维度值索引表）
输出：维度值命中列表，包含：
  • query_value: 查询中的原始值（如"华东"）
  • matched_text: 匹配到的值（如"华东地区"）
  • dim_table: 所属维度表（如"public.dim_region"）
  • score: 匹配分数
  • source_index: 在原问题中的位置
3b. 分类事实表/维度表 合并多个来源：
1. 向量检索的表（基于 table_category）
  • category="事实表" 或 "fact" → candidate_fact_tables
  • 其他 → candidate_dim_tables
2. 列的父表（基于 parent_category）
3. 维度值命中的表（dim_table）→ candidate_dim_tables

输出：
• candidate_fact_tables: 候选事实表列表
• candidate_dim_tables: 候选维度表列表
• table_similarities: 表相似度字典
• dim_value_hits: 维度值匹配结果

🔍 步骤4: JOIN路径规划（Neo4j图检索）

代码位置：retriever.py:333-437 (_retrieve_join_plans)

子步骤 说明
4.0 维度表优化判断 方法：_should_use_dimension_only()
规则：
1. 有时间约束 → 不优化（需要事实表）
2. 没有维度表候选 → 不优化
3. 只有1个维度表 + 无事实表 → 使用单表，跳过JOIN
4. 维度表相似度 > 事实表相似度 + 阈值（0.05）→ 使用单维度表

如果优化：返回单表计划，不调用Neo4j
4.1 选择Base表 有事实表：选择相似度最高的事实表作为Base
纯维度表：通过连通性分析选择Base
  • 遍历每个维度表作为候选Base
  • 查询它能连接到多少个其他表（连通性）
  • 选择连接数最多的；如果相同，选择相似度最高的
4.2 Neo4j查询JOIN路径 方法：neo4j_client.plan_join_paths()
参数：
• base_tables: [选中的Base表]
• target_tables: [其他所有候选表]
• max_hops: 5（最大跳数）
• strategy: "apoc_dijkstra"（最短路径算法）

返回：JOIN计划，包含：
• base: Base表
• targets: 目标表列表
• edges: JOIN边列表，每条边包含：
  - src_table: 源表
  - dst_table: 目标表
  - on_clause: JOIN条件（如 "SRC.id = DST.fk_id"）

🔍 步骤5: 获取表卡片

  • 方法：pg_client.fetch_table_cards(all_table_ids)
  • 输入：所有候选表的ID列表（事实表 + 维度表）
  • 输出：表卡片字典，包含每个表的详细信息：
    • text_raw: 表的文本描述（包含所有字段）
    • grain_hint: 粒度提示
    • time_col_hint: 时间列提示
    • table_category: 表类别

📦 最终输出 (schema_context)

{
  "tables": [...],                   # 候选表名列表
  "columns": [...],                  # 候选列列表
  "join_plans": [...],               # JOIN计划
  "table_cards": {...},              # 表卡片字典
  "dim_value_hits": [...],           # 维度值匹配结果
  "candidate_fact_tables": [...],    # 候选事实表
  "candidate_dim_tables": [...],     # 候选维度表
  "table_similarities": {...},       # 表相似度
  "metadata": {
    "retrieval_time": 2.35,
    "table_count": 5,
    "column_count": 20,
    "join_plan_count": 1,
    "dim_match_count": 3
  }
}

3️⃣ SQL生成节点 (sql_generation_node)

属性 说明
代码位置 src/modules/sql_generation/subgraph/nodes/sql_generation.py
类型 Agent（调用LLM）
实现类 SQLGenerationAgent
使用模型 qwen-max（通义千问）
主要功能 基于多种上下文生成PostgreSQL SQL：
1. Schema上下文：
  • 表卡片（table_cards）
  • JOIN计划（join_plans）
  • 时间列提示（time_col_hint）
  • 维度值匹配（dim_value_hits）
2. 解析结果（parse_result）：
  • 时间窗口（time.start, time.end）
  • 维度过滤（dimensions）
  • 指标提示（metric）
3. 历史SQL案例（similar_sqls）：
  • 复用 query_embedding 检索相似SQL
  • 最多2个相似案例
  • 包含：相似度、问题、SQL
4. 依赖结果（dependencies_results）
5. 验证错误（重试时）
提示词要求 1. 所有表必须包含schema前缀（例如 public.table）
2. 时间过滤使用半开区间（>= start AND < end）
3. JOIN条件必须严格按照ON模板
4. 仅输出SQL，不附加说明
输出字段 • generated_sql: 生成的SQL字符串
• iteration_count: 当前迭代次数（自动+1）
• error (可选): "generation_failed"
配置参数 • llm_model: qwen-max
• temperature: 0
• max_tokens: 2000
• timeout: 30s
• sql_similarity_threshold: 0.6
• max_similar_sqls: 2

4️⃣ 三层验证节点 (validation_node)

属性 说明
代码位置 src/modules/sql_generation/subgraph/nodes/validation.py
类型 Tool（确定性函数，不调用LLM）
实现类 SQLValidationTool（src/tools/validation/sql_validation.py）
三层验证 第1层：语法检查
  • 使用 sqlparse 解析SQL
  • 允许的语句类型：SELECT, WITH, UNKNOWN
  • 检查SQL是否可解析

第2层：安全检查
  • 禁止关键字：DROP, DELETE, TRUNCATE, ALTER, CREATE, INSERT, UPDATE, GRANT, REVOKE, EXEC, EXECUTE
  • 禁止SQL注释（防注入）
  • 禁止多条SQL语句（防注入）

第3层：语义检查（EXPLAIN）
  • 执行 EXPLAIN 分析SQL执行计划
  • 检查表是否存在
  • 检查字段是否存在
  • 性能警告：顺序扫描、嵌套循环、预估行数阈值
输出字段 • validation_result:
  - valid: bool（是否通过验证）
  - errors: List[str]（错误列表）
  - warnings: List[str]（警告列表）
  - layer: "syntax" | "security" | "semantic" | "all_passed"

🔄 重试机制详解

重试判断逻辑 (should_retry)

代码位置：src/modules/sql_generation/subgraph/create_subgraph.py:24-77

条件 返回值 说明
validated_sql 存在 "success" 验证通过，流程成功结束 → END
error_type 为 "schema_retrieval_failed" "fail" Schema检索失败，不可恢复 → END
error_type 为 "generation_failed" "fail" SQL生成失败，不可恢复 → END
iteration_count < 3 "retry" 验证失败但未超过最大迭代次数 → 回到 sql_generation
iteration_count >= 3 "fail" 重试次数已达上限 → END

重试流程：

  • 第1次生成：iteration_count=0 → 生成SQL → 验证失败 → iteration_count=1
  • 第2次生成：iteration_count=1 → 重新生成（包含上次错误）→ 验证失败 → iteration_count=2
  • 第3次生成：iteration_count=2 → 重新生成（包含所有错误）→ 验证失败 → iteration_count=3
  • 第4次检查：iteration_count=3 → 达到上限 → 返回 "fail"

错误反馈：

  • 重试时，sql_generation_node 会获取上一次的 validation_result
  • 将 validation_result.errors 添加到提示词中
  • LLM基于错误信息生成改进的SQL

⚙️ 配置文件说明

配置文件位置

src/modules/sql_generation/config/sql_generation_subgraph.yaml

主要配置项：

  • question_parsing: 问题解析配置（模型、超时、开关）
  • schema_retrieval: Schema检索配置（topk、相似度阈值、超时）
  • sql_generation: SQL生成配置（模型、温度、提示词配置）
  • validation: 验证配置（三层验证开关、阈值）
  • retry: 重试配置（最大迭代次数、错误反馈策略）
  • cache: 缓存配置（开关、TTL）
  • logging: 日志配置（级别、详细度）
  • monitoring: 监控配置（指标收集、告警）

⚠️ 关键注意事项

  • 向后兼容：如果外部传入 parse_hints，会直接使用，不调用内部解析
  • 可配置性：所有参数都可以通过 yaml 配置文件或环境变量覆盖
  • 错误处理：
    • parsing_failed: 问题解析失败（如果不允许fallback）
    • schema_retrieval_failed: Schema检索失败（不可恢复）
    • generation_failed: SQL生成失败（不可恢复）
    • validation_failed: 验证失败且达到最大重试次数
  • 重试策略：最多3次迭代（包含首次生成），每次重试都会将上一次的错误反馈给LLM
  • 日志追踪：使用 query_id 进行日志关联，便于追踪整个流程
  • 性能监控：记录每个阶段的耗时，支持指标收集和告警

✨ 子图设计优势

  • 模块化：每个节点职责单一，易于测试和维护
  • 可复用：子图可以被主流程多次调用（批次处理）
  • 容错性：多层验证 + 自动重试 + 错误反馈
  • 可观测性：详细的日志记录和验证历史
  • 高度配置化：所有参数都可外部配置
  • 向后兼容：支持外部传入解析结果，兼容旧版本调用方式
  • 智能Schema检索：结合向量、图、维度值多种检索方式，提供全面的Schema上下文
