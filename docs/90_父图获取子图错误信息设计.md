# 90 - 父图获取子图错误信息设计

> 日期：2026-03-06
> 相关代码：`src/modules/nl2sql_father/graph.py`、`src/modules/sql_generation/subgraph/state.py`、`src/modules/sql_generation/subgraph/nodes/validation.py`

---

## 1. 目标

当前父图可以知道"子图成功或失败"，但在部分失败场景下，拿不到明确的失败阶段和错误原因。

本设计的目标是让父图在子图失败时，至少拿到以下信息：

1. 子图失败发生在哪一步
2. 子图失败的错误类型
3. 子图失败的错误摘要

输出不追求保留所有底层细节，但要满足以下场景：

- 能区分是 `question_parsing`、`schema_retrieval`、`sql_generation` 还是 `validation` 失败
- 能看到类似"表不存在""列不存在""数据库连接失败"这类关键错误摘要
- 父图最终返回结果中可以直接消费这些错误信息

---

## 2. 当前问题

### 2.1 已有能力

当前子图在以下场景会把错误写到顶层状态，因此父图可以直接拿到：

- `question_parsing` 异常：写入 `error` / `error_type`
- `schema_retrieval` 异常：写入 `error` / `error_type`
- `sql_generation` 异常：写入 `error` / `error_type`
- 子图整体执行异常：父图 wrapper 会兜底生成 `error`

因此这些场景下，父图已经具备基础感知能力。

### 2.2 关键缺口

当前最大缺口在 `validation` 节点。

当 SQL 通过语法生成，但在验证阶段失败时：

- `validation_result.errors` 中通常已经有明确错误
- 例如：
  - 表不存在
  - 列不存在
  - 数据库连接失败
  - EXPLAIN 执行失败

但 `validation_node()` 在"验证失败但未抛异常"的分支中，只返回 `validation_result`，没有把错误同步写入顶层 `error` / `error_type`。

结果是：

1. 子图内部知道验证失败原因
2. `extract_output()` 只读取顶层 `error`
3. 父图 wrapper 拿不到具体验证错误
4. 最终父图只能知道"SQL 没生成成功"或"验证失败后结束"，但不知道原因

### 2.3 影响

这会导致以下问题：

- Fast Path 下，父图 `result["error"]` 缺少关键信息
- Complex Path 下，`sub_queries[i]["error"]` 可能为空或过于笼统
- Summarizer 难以向用户解释真实失败原因
- CLI / API 无法稳定显示"表不存在"等重要信息

---

## 3. 设计原则

### 3.1 顶层状态只保留摘要

子图顶层状态中的 `error` 不保存所有原始错误，只保存可读摘要。

建议规则：

- 若有多条错误，取第一条作为主摘要
- 若需要完整诊断，继续保留在 `validation_result.errors` 或 `validation_history`

### 3.2 失败阶段单独建模

不要依赖 `error_type` 反推失败阶段。

新增单独字段：

- `failed_step`

推荐值：

- `question_parsing`
- `schema_retrieval`
- `sql_generation`
- `validation`

这样父图、CLI、Summarizer 都能直接使用。

**与 `should_retry()` 的兼容性**：子图的 `should_retry()`（`create_subgraph.py:61`）采用**黑名单**模式，只把 `schema_retrieval_failed` 和 `generation_failed` 视为不可恢复错误，其余 `error_type` 均允许重试。本次新增的 `validation_syntax_failed`、`validation_semantic_failed`、`validation_security_failed` 不在黑名单中，验证失败仍会正常触发重试，不受影响。

### 3.3 机器路由与人类可读分离

字段职责拆分如下：

- `failed_step`：失败发生在哪个节点
- `error_type`：机器可路由的错误类别
- `error`：面向人类展示的错误摘要

示例：

```json
{
  "failed_step": "validation",
  "error_type": "validation_semantic_failed",
  "error": "表不存在: sales_order"
}
```

### 3.4 summary 与 error 的职责边界

- `summary`：面向终端用户的自然语言提示，保持简洁友好，不拼接原始技术细节
- `error` / `sub_queries[*].error`：携带具体错误文本（如 `relation "sales_order" does not exist`），供展示层按需呈现

Summarizer 只负责生成友好提示，不承担传递完整错误的职责。具体错误始终通过 `error` 字段暴露，由 CLI / API / 前端展示层决定如何向用户展示。

**例外：Complex Path 多子查询失败**。Complex Path 下可能有多个子查询分别失败在不同阶段，固定模板无法区分各子查询的失败原因。此时 `summary` 可包含错误摘要以区分各子查询，例如"子查询在 validation 阶段失败：表不存在"。这是为了保证多子查询场景下 `summary` 仍有诊断价值。

#### 4.2.4 `validation`

**文件**：`src/modules/sql_generation/subgraph/nodes/validation.py`

这是本次最关键的修改点。需要修改三个分支。

**分支 1：空 SQL 防御分支（第 37-55 行）**

当 `generated_sql` 为空且不是上游 `generation_failed` 时，当前已返回 `error` 和 `error_type`，需补充 `failed_step`：

```python
return {
    "validation_result": { ... },
    "error": "系统错误：空SQL进入验证",
    "error_type": "validation_failed",
    "failed_step": "validation",  # 新增
}
```

**分支 2：验证失败但未抛异常（当前核心缺口）**

新增以下行为：

1. 从 `validation_result["errors"]` 提取摘要
2. 写入顶层 `error`
3. 根据 `layer` 动态构造 `error_type`
4. 写入 `failed_step = "validation"`

**分支 3：验证成功**

成功时需同步清空 `failed_step`，避免重试场景下残留值：

建议实现（分支 2 + 分支 3）：

```python
if validation_result["valid"]:
    updates["validated_sql"] = generated_sql
    updates["error"] = None
    updates["error_type"] = None
    updates["failed_step"] = None  # 清空残留
else:
    error_list = validation_result.get("errors") or ["未知错误"]
    layer = validation_result.get("layer") or "unknown"
    updates["error"] = error_list[0]
    updates["error_type"] = f"validation_{layer}_failed"
    updates["failed_step"] = "validation"
```

效果：

- 表不存在：`error_type = "validation_semantic_failed"`
- SQL 语法解析失败：`error_type = "validation_syntax_failed"`
- 安全检查失败：`error_type = "validation_security_failed"`

**注意**：已有的 `except` 异常分支（第 96-108 行）保持原样（`error_type = "validation_failed"`），不走动态构造，只需补充 `"failed_step": "validation"`。

### 4.5 Summarizer 使用失败阶段信息

**文件**：`src/modules/nl2sql_father/nodes/summarizer.py`

#### 4.5.1 设计策略

Summarizer 的 `summary` 只生成用户友好的提示，**不拼接原始错误文本**。

具体错误（如 `relation "sales_order" does not exist`）通过以下字段暴露给展示层：

- 顶层 `error`（Fast Path）
- `sub_queries[*].error`（Fast Path + Complex Path）

展示层（CLI / Streamlit / API 消费方）自行决定是否向用户展示具体错误。例如 Streamlit 可在折叠面板中显示，CLI 可在详情区域打印。

#### 4.5.2 扩展错误模板映射

当前 `_build_error_summary()` 的映射表只含 `validation_failed`，需补充新增的子类型：

```python
error_templates = {
    "parsing_failed": "抱歉，系统无法理解您的问题。建议您换一种方式描述，或提供更多上下文信息。",
    "schema_retrieval_failed": "抱歉，系统暂时无法找到相关的数据表。请确认您的问题是否涉及系统已有的数据。",
    "generation_failed": "抱歉，系统在生成查询时遇到了问题。可能是问题过于复杂，建议您简化问题后重试。",
    "validation_failed": "抱歉，系统生成的查询存在问题，无法执行。建议您换一种方式提问。",
    "validation_syntax_failed": "抱歉，系统生成的SQL存在语法问题，建议您换一种方式描述。",
    "validation_semantic_failed": "抱歉，查询中引用了不存在的表或列，请确认数据范围后重试。",
    "validation_security_failed": "抱歉，系统检测到不允许的SQL操作类型。",
}
```

#### 4.5.3 Fast Path

逻辑不变：若顶层 `error` 存在，走 `_build_error_summary()`。扩展映射表后自动覆盖新的 `error_type`。

#### 4.5.4 Complex Path

当前 `sql_gen_batch_wrapper` 不设置顶层 `error`/`error_type`，因此 Summarizer 的"场景1"（`not execution_results and error`）在 Complex Path 下不会触发。

需要增强的场景有两个：

**场景 A：SQL 生成全失败，根本没执行**（`execution_results` 为空，`error` 也为空）

当前走入「场景2」，只返回笼统的"系统未能生成有效的SQL查询"。建议从 `sub_queries` 提取失败详情：

```python
# 替换原有的「场景2」逻辑
if not execution_results:
    # 尝试从 sub_queries 提取失败信息（Complex Path 增强）
    failed_sqs = [sq for sq in state.get("sub_queries", []) if sq.get("status") == "failed"]
    if failed_sqs:
        lines = []
        for sq in failed_sqs:
            step = sq.get("failed_step", "未知阶段")
            err = sq.get("error", "未知错误")
            lines.append(f"子查询在{step}阶段失败：{err}")
        summary = "抱歉，查询未能完成。\n" + "\n".join(lines)
    else:
        summary = "抱歉，系统未能生成有效的SQL查询。"
    return {"summary": summary}
```

**场景 B：SQL 执行了但全部失败**（`execution_results` 非空，`success_results` 为空）

当前走入 `not success_results` 分支，只拼接了 `execution_results` 中的执行层错误（如数据库超时）。这个分支**不需要修改**——执行层错误已经足够说明问题，不需要再向上追溯到 `sub_queries` 的生成阶段错误。

## 5. 推荐错误映射

建议采用以下映射规则：

| 场景 | failed_step | error_type | error 示例 |
|------|-------------|------------|------------|
| 问题解析异常 | `question_parsing` | `parsing_failed` | `问题解析失败: ...` |
| Schema 检索异常 | `schema_retrieval` | `schema_retrieval_failed` | `Schema检索失败: ...` |
| LLM 生成异常 | `sql_generation` | `generation_failed` | `SQL生成失败: ...` |
| 语法验证失败 | `validation` | `validation_syntax_failed` | `SQL解析失败：...` |
| 安全验证失败 | `validation` | `validation_security_failed` | `不允许的SQL类型：DELETE` |
| 语义验证失败 | `validation` | `validation_semantic_failed` | `relation "sales_order" does not exist` |
| 空SQL进入验证 | `validation` | `validation_failed` | `系统错误：空SQL进入验证` |
| 验证节点异常 | `validation` | `validation_failed` | `验证过程异常：...` |
| 子图整体异常（wrapper 兜底） | `sql_generation` | `generation_failed` | `SQL生成子图执行异常: ...` |

> **说明**：`error` 字段的值由各节点在失败时直接构造或从下游结果提取（如 `validation_result["errors"][0]`），不做额外转换。

## 6. 实施顺序

建议按以下顺序修改：

1. 修改 `SQLGenerationState`，新增 `failed_step` 字段
2. 修改 `NL2SQLFatherState`，新增 `failed_step` 字段；修改 `create_initial_state()`
3. 修改 `SubQueryInfo`，新增 `error_type` 和 `failed_step` 字段
4. 修改 `extract_output()` 和 `get_error_summary()`，透传 `failed_step`
5. 修改 `extract_final_result()`，顶层暴露 `error_type` 和 `failed_step`
6. 修改 `validation_node()` 三个分支：空 SQL 防御、验证失败、验证成功
7. 修改其他子图节点（`question_parsing`、`schema_retrieval`、`sql_generation`），统一补充 `failed_step`
8. 修改父图 `sql_gen_wrapper()`：成功/失败/兜底三个分支都补齐 `error_type`/`failed_step`
9. 修改父图 `sql_gen_batch_wrapper()`：成功/失败/兜底三个分支补齐
10. 修改 Summarizer：扩展错误模板映射 + 增强 Complex Path 场景2
11. 补充单元测试

## 7. 测试建议

### 7.1 子图层

建议新增或补充以下测试：

- `validation` 语义失败时，子图输出包含：
  - `error`（取 `errors[0]`）
  - `error_type = validation_semantic_failed`
  - `failed_step = validation`
- `validation` 语法失败时，子图输出包含：
  - `error_type = validation_syntax_failed`
  - `failed_step = validation`
- `validation` 安全失败时，子图输出包含：
  - `error_type = validation_security_failed`
  - `failed_step = validation`
- `validation` 空 SQL 防御分支：
  - `error_type = validation_failed`
  - `failed_step = validation`
- 验证失败后重试成功时，子图输出中 `error`、`error_type`、`failed_step` 均为 `None`
- 各节点（`question_parsing`、`schema_retrieval`、`sql_generation`）失败时 `failed_step` 正确

### 7.2 父图层

建议验证：

- Fast Path 子图失败时，父图顶层 `error`、`error_type`、`failed_step` 正确
- Fast Path 子图失败时，`sub_queries[0]` 也包含 `error_type` 和 `failed_step`
- Fast Path 子图失败时，`extract_final_result()` 顶层包含 `error_type` 和 `failed_step`
- Fast Path wrapper 兜底异常时，`sub_queries[0]` 包含 `error_type` 和 `failed_step`
- Complex Path 某个子查询失败时，对应 `sub_queries[i]` 保留完整失败信息
- Complex Path 全部 SQL 生成失败（无 `execution_results`）时，Summarizer 输出包含各子查询的失败阶段和错误摘要

## 8. 预期收益

改造完成后，父图将具备以下能力：

1. 知道子图失败发生在哪个节点
2. 知道失败属于哪一类错误
3. 能直接向上层返回关键错误摘要
4. 能在复杂路径下定位具体失败子查询
5. 能向 CLI / API / Summarizer 提供更可解释的失败信息

这会显著提升排障效率，也能降低"只知道失败、不知道为什么失败"的问题。
