⚡ 快速路径 (Fast Path)

适用于简单的单个查询问题

1️⃣ Simple Planner（参数准备，< 1ms）
2️⃣ SQL生成子图
3️⃣ SQL执行
4️⃣ 简单总结（可选）
✅ 优势：
  • 只需4个节点（Simple Planner 纯函数）
  • 可选LLM调用（0-1次）
  • 响应速度快（Planner < 1ms）
  • 成本低
~0.5s
响应时间
$0.005
单次成本

🧠 复杂路径 (Complex Path)

适用于多步骤或有依赖的问题

1️⃣ Planner规划
2️⃣ Inject Params
3️⃣ SQL生成子图
4️⃣ SQL执行
5️⃣ Check完成度
6️⃣ 循环或进入Summarizer
✅ 优势：
  • 智能规划依赖
  • 支持多轮执行
  • 并行优化
  • 处理复杂场景
~3-5s
响应时间
$0.02
单次成本

💡 问题分类示例

⚡ Simple - 走快速路径

单个独立查询

"广州服务区的销售额是多少？"

单表聚合

"2024年总销售额是多少？"

简单筛选

"销售额超过100万的服务区有哪些？"

简单对比（单SQL）

"北京和上海服务区的销售额对比"

🧠 Complex - 走复杂路径

多个依赖问题

"哪个服务区销售最高？它的地址和公司是？"

需要中间结果

"找出车流量前3的服务区，分别查询它们的营业额"

多维度分析

"分析2024年各区域销售趋势，对比同比增长"

逐步钻取

"销售下降的服务区有哪些？原因是什么？"

🔧 Router节点实现

使用轻量级LLM快速分类，成本可忽略不计

def router_node(state: NL2SQLState): """快速分类器，判断问题复杂度""" question = state["user_question"] # 使用GPT-3.5-turbo或Claude-Haiku等快速模型 prompt = """判断问题类型，只返回simple或complex： Simple: 单个独立问题，一次SQL查询 Complex: 多个问题或有依赖关系 问题：{question} 答案：""" result = llm.invoke(prompt.format(question=question)) complexity = result.content.strip().lower() return {"complexity": complexity} def route_by_complexity(state: NL2SQLState) -> str: """路由函数""" return "simple_planner" if state["complexity"] == "simple" else "planner"

⚡ Simple Planner 节点

轻量级参数准备节点，为 Fast Path 提供对称设计

核心特点
  • 纯函数，不调用 LLM
  • 执行时间 < 1ms
  • 创建单个子查询
  • 设置 current_sub_query_id
设计原因
  • 简化 Router 职责（仅判定复杂度）
  • 对称设计（Simple/Complex 都有 Planner）
  • 为 Phase 2 预留优化空间
  • 保持架构一致性
