# 93 日志体系治理与优化方案

## 1. 背景与目标

当前项目已经具备基本日志能力，但从“整个项目统一治理”的角度看，日志体系存在明显割裂：

- 核心链路日志文件名为 `sql_subgraph.log`，但实际包含父图与子图两部分内容，命名与内容不一致。
- FastAPI、Streamlit、CLI 的日志初始化入口不同，配置分散在 `src/configs/logging.yaml`、`frontend/logger.py` 和启动命令重定向中。
- `src/configs/config.yaml` 中存在一套 `logging:` 配置，但当前运行链路并未实际接入，容易误导维护者。
- 部分模块使用 `get_module_logger("...")`，部分模块使用 `logging.getLogger(__name__)`，导致日志是否进文件、进哪份文件，不够直观。
- `nohup ... > logs/*.log 2>&1` 这类进程输出重定向文件，与应用级日志文件混在同一个 `logs/` 目录下，职责边界不清。
- Streamlit 日志未启用轮转，长期运行可能导致单文件持续膨胀。
- 缺少统一的日志命名、目录规划、级别策略、字段规范和排障文档。

本方案的目标不是“简单再多加几个日志文件”，而是建立一套可持续维护的日志治理规则，使团队能够明确回答以下问题：

1. 一条日志属于哪个层次。
2. 它应该写到哪个文件。
3. 它的级别由哪里控制。
4. 它如何按 `query_id` / `thread_id` 追踪。
5. 新模块接入时应该遵守什么规范。

---

## 2. 现状问题诊断

### 2.1 文件命名与实际内容不一致

当前核心文件名是 `logs/sql_subgraph.log`，但其实际承载的是 `nl2sql.*` 整个命名空间日志，包含：

- 父图主流程
- SQL 生成子图
- schema retrieval
- planner / router / summarizer / execution 等模块

这会带来两个问题：

- 新维护者会误以为“只有子图有文件日志，父图没有”。
- 父图和子图未来若要拆分级别或拆分文件，当前命名会成为阻碍。

### 2.2 配置源不唯一

当前与日志有关的配置至少分布在以下位置：

- `src/configs/logging.yaml`
- `src/configs/config.yaml`
- `frontend/logger.py`
- 启动脚本或运维命令中的 stdout/stderr 重定向

这会导致“看起来能配，实际上不生效”的情况，尤其是 `config.yaml` 中的 `logging:` 段落。

### 2.3 Logger 命名空间不统一

当前项目同时存在两类写法：

- `get_module_logger("sql_subgraph")` 这类 `nl2sql.*` 体系
- `logging.getLogger(__name__)` 这类模块名体系

后者在未显式挂 handler 时，通常只走 root/console，不会进入核心文件日志。结果是：

- 维护者难以预判某模块的日志去向
- 同一条调用链中，不同模块的日志落点可能不一致

### 2.4 进程日志与应用日志混用

例如：

- `logs/fastapi.log` 是应用日志
- `logs/uvicorn.log` 可能是进程重定向日志
- `logs/streamlit.log` 是应用日志
- `logs/streamlit_server.log` 可能是进程重定向日志

这些文件混在同一层目录，语义不清晰，排障时容易打开错误文件。

### 2.5 结构化上下文不足

目前已经有 `with_query_id()` 能给部分日志加上 `[query_id]` 前缀，但还不够系统：

- 并非所有关键节点都稳定携带 `query_id`
- `thread_id` / `user_id` / `sub_query_id` 未形成统一字段规范
- API 层与核心层之间虽然能共享追踪 ID，但没有形成明确定义的日志字段标准

### 2.6 生命周期与保留策略不统一

- 核心日志与 API 日志使用 `RotatingFileHandler`
- Streamlit 日志使用普通 `FileHandler`
- stdout/stderr 重定向日志的保留完全依赖运维习惯

这意味着不同文件的增长风险、清理方式、历史保留策略并不一致。

## 3. 优化原则

本项目后续的日志治理建议遵循以下原则：

### 3.1 单一事实来源

同一类配置只能有一个权威来源。对“日志级别、文件路径、轮转策略、logger 命名空间”的解释权，只保留一份。

### 3.2 先按职责分层，再决定是否拆文件

不要按“想到一个模块就加一个文件”的方式扩散日志文件。应先定义日志层次，再决定文件数量。

### 3.3 文件名必须表达真实语义

如果一份文件承载的是“整个核心链路”，就不能继续命名为 `sql_subgraph.log`。

### 3.4 进程输出与应用日志分离

`nohup` 重定向文件不应与应用级 handler 输出文件混在一起。

### 3.5 以链路追踪为中心

项目是 NL2SQL 多阶段链路系统，日志设计必须围绕：

- `query_id`
- `thread_id`
- `sub_query_id`
- `user_id`

来组织，而不是仅靠文本搜索错误关键词。

### 3.6 默认可运维

方案需要同时适用于：

- 本地 CLI 调试
- FastAPI 服务运行
- Streamlit 前端运行
- 后续长期后台运行

### 3.7 CLI 交互输出不纳入日志治理范围

这里需要明确一个边界：

- CLI 中的 `print()`
- Rich 组件渲染输出
- 交互式状态提示、表格、彩色面板、spinner

这些都属于**用户交互输出**，不是日志治理对象。

本方案治理的范围仅包括：

- Python `logging` 体系输出
- 文件 handler 落盘日志
- 进程 stdout/stderr 重定向日志

因此，CLI 界面的显示体验优化，与日志文件命名、logger 命名空间、级别配置，不应混为一谈。当前文档不将 `print()` / Rich 纳入日志改造范围，这个边界划分是正确的。

## 4. 目标日志架构

### 4.1 推荐的日志层次

结合当前项目实际，建议先将项目日志分为三层：

1. **核心业务日志**
   记录 NL2SQL 核心推理链路，包括父图、子图、检索、SQL 生成、SQL 执行、总结等。

2. **API 应用日志**
   记录 FastAPI 路由级别事件、请求失败、参数校验失败、接口异常、生命周期事件。

3. **进程输出日志**
   记录 uvicorn / streamlit 进程标准输出与标准错误，不与应用日志混用。

### 4.2 推荐的目录结构

建议将 `logs/` 目录显式分层：

```text
logs/
  app/
    nl2sql.log
    api.log
  process/
    uvicorn.out.log
    streamlit.out.log
```

说明：

- `logs/app/`：应用级 logger handler 输出
- `logs/process/`：启动命令 stdout/stderr 重定向输出
- 当前阶段不默认规划 `frontend.log` 与 `access.log`，避免在没有明确收益时增加排障复杂度

### 4.3 推荐的文件命名

建议最终采用以下命名：

| 文件 | 说明 | 是否推荐 |
|------|------|---------|
| `logs/app/nl2sql.log` | 核心 NL2SQL 全链路日志，包含父图与子图 | 推荐 |
| `logs/app/api.log` | FastAPI 应用层日志 | 推荐 |
| `logs/process/uvicorn.out.log` | uvicorn 进程 stdout/stderr，独立于项目应用日志配置 | 推荐 |
| `logs/process/streamlit.out.log` | streamlit 进程 stdout/stderr，作为前端侧主排障入口 | 推荐 |

当前阶段不建议默认引入以下文件：

| 文件 | 原因 |
|------|------|
| `logs/app/frontend.log` | 与 `streamlit.out.log` 职责重叠，容易造成前端排障时双文件并看 |
| `logs/app/access.log` | 当前代码中不存在这类日志，若引入需要额外配置，甚至需要改代码 |

不建议恢复 `all.log`，原因如下：

- “all” 的边界不清，不知道是否包含前端、API、核心、第三方库、stdout/stderr。
- 当系统继续扩展时，`all.log` 容易再次演变成“什么都往里塞”的大杂烩。
- 更合理的聚合边界是“NL2SQL 业务链路”，因此推荐 `nl2sql.log`，而不是 `all.log`。

## 5. Logger 命名空间设计

### 5.1 统一命名空间

建议将项目内部可治理的日志统一收敛到以下命名空间：

```text
nl2sql
nl2sql.father
nl2sql.sql_subgraph
nl2sql.retrieval
nl2sql.execution
nl2sql.api
```

当前已有的 `nl2sql.<module>` 思路是对的。结合当前项目复杂度，没有必要再额外引入一层 `core` 抽象；直接使用 `nl2sql.*` 与 `nl2sql.api` 两级边界更清晰。

### 5.2 推荐映射关系

| 模块类别 | 推荐 logger 前缀 | 目标文件 |
|------|------|------|
| 父图主流程 | `nl2sql.father` | `nl2sql.log` |
| SQL 生成子图 | `nl2sql.sql_subgraph` | `nl2sql.log` |
| Schema 检索/向量检索 | `nl2sql.retrieval` | `nl2sql.log` |
| SQL 执行/数据库执行 | `nl2sql.execution` | `nl2sql.log` |
| FastAPI 路由/生命周期 | `nl2sql.api` | `api.log` |

### 5.3 对 `logging.getLogger(__name__)` 的治理建议

后续应明确规定：

- 项目内部业务代码，统一使用项目封装的 logger 获取函数。
- 只有第三方库兼容层、极底层通用工具层，才允许使用 `logging.getLogger(__name__)`。

否则会持续出现“有的日志能进核心文件，有的只能打到 console”的现象。

## 6. 配置治理方案

### 6.1 单一权威配置文件

建议选择 `src/configs/logging.yaml` 作为唯一运行时日志配置入口。

原因：

- 当前 CLI 与 FastAPI 已经实际接入它。
- `dictConfig` 更适合集中描述 handler、formatter、logger 层级关系。
- 文件级别、logger 级别、轮转策略可以完整表达。

### 6.2 `config.yaml` 的处理建议

当前 `src/configs/config.yaml` 中的 `logging:` 段落，建议二选一：

1. **推荐方案**：移除该段日志配置，避免形成第二套“看得见但不生效”的伪配置。
2. **备选方案**：只保留环境级别开关，并明确注明“由启动阶段转译到 `logging.yaml` 后才生效”。

从项目治理角度，推荐第一种，避免双配置源。

### 6.3 Streamlit 与进程日志的边界

这里需要明确区分两件事：

1. **项目应用日志**
   由你们自己的 Python 代码通过 `logging` 主动写入。

2. **服务进程日志**
   由 `uvicorn` 或 `streamlit run` 启动后的 stdout/stderr 产生。

当前阶段更务实的做法是：

- 不把 Streamlit 单独提升为一套新的应用日志目标文件
- 前端排障优先查看 `logs/process/streamlit.out.log`
- 如后续确实需要“前端业务事件审计”，再单独评估是否恢复 `frontend.log`

这样可以避免前端侧同时维护两份含义接近的日志文件。

### 6.4 环境分层建议

建议按环境控制默认级别：

| 环境 | console | nl2sql file | api file |
|------|---------|-----------|----------|
| `dev` | `INFO` | `DEBUG` | `INFO` |
| `test` | `WARNING` | `INFO` | `INFO` |
| `prod` | `INFO` | `INFO` | `INFO` |

说明：

- 开发环境保留核心 `DEBUG`，便于追链路。
- 生产环境默认不建议核心长期 `DEBUG`，避免日志量失控与敏感信息泄漏。
- `uvicorn` / `streamlit` 进程日志级别不由本表控制，它们属于各自进程或启动参数的配置范畴。

## 7. 日志内容规范

### 7.1 统一字段建议

项目内关键日志应尽量稳定包含以下上下文字段：

- `query_id`
- `thread_id`
- `sub_query_id`
- `user_id`
- `path_taken`
- `module`
- `elapsed_ms`

推荐方向是将这些字段做成结构化上下文，而不是依赖“手工拼接中文句子”。

### 7.2 核心链路关键节点

对于 NL2SQL 主链路，至少应覆盖以下关键日志节点：

1. 收到用户问题
2. 生成或接收 `query_id`
3. Router 判路结果
4. Planner 输出
5. 子查询拆分结果
6. Schema Retrieval 结果摘要
7. SQL 生成结果摘要
8. SQL 校验结果
9. SQL 执行结果摘要
10. 最终总结输出
11. 异常中断点

这些日志应优先进入 `nl2sql.log`。

### 7.3 日志内容分级

建议统一以下写法：

- `DEBUG`：中间状态、召回明细、Prompt 裁剪信息、候选表细节、性能细节
- `INFO`：链路里程碑、关键结果摘要、开始/结束事件
- `WARNING`：可恢复异常、降级路径、缺省值兜底、配置缺失但已兼容
- `ERROR`：请求失败、调用失败、节点异常、外部依赖错误

### 7.4 敏感信息控制

后续日志治理必须同时纳入以下约束：

- 不记录明文密码、API Key、数据库完整连接串
- 不在生产环境完整打印超长 Prompt、完整结果集、完整 SQL 大量返回数据
- 对用户输入、模型输出、SQL 执行结果设置可控的最大打印长度与最大行数

## 8. 轮转与保留策略

### 8.1 应用日志

建议统一采用按大小轮转：

| 文件 | 单文件大小 | 保留份数 |
|------|-----------|---------|
| `nl2sql.log` | 10 MB | 10 |
| `api.log` | 10 MB | 10 |

### 8.2 进程输出日志

进程日志建议采用更保守策略：

- 默认只在部署/后台运行时启用
- 推荐交由进程管理器或 logrotate 管理
- 不建议把进程 stdout/stderr 直接视为主排障日志
- `uvicorn.out.log` 和 `streamlit.out.log` 不属于项目 `logging.yaml` 的治理范围

### 8.3 历史清理建议

建议在文档中明确：

- 应用日志是主日志
- 进程输出日志是辅助日志
- 排障优先看 `logs/app/`，不是先看 `logs/process/`

## 9. 推荐迁移方案

### 9.1 第一阶段：命名与目录收敛

目标：

- 将 `sql_subgraph.log` 的语义升级为 `nl2sql.log`
- 将应用日志和进程日志分目录管理
- 补齐日志目录约定文档

产出：

- 明确新的目标文件拓扑
- 标注文档废弃项：`sql_subgraph.log`、`all.log`、混用的 `uvicorn.log`

### 9.2 第二阶段：配置源收敛

目标：

- 只保留一套真正生效的应用日志配置源
- 让 CLI / FastAPI 共享同一套规则

产出：

- 日志级别、路径、轮转策略从单一入口控制
- 移除或废弃 `config.yaml` 中未接线的日志配置
- 明确声明 `uvicorn` / `streamlit` 进程日志独立于项目应用日志配置

### 9.3 第三阶段：命名空间治理

目标：

- 将内部业务 logger 纳入统一命名空间
- 统一日志落点

产出：

- 父图、子图、检索、执行等模块的 logger 命名可预测
- 减少 `__name__` 型 logger 带来的不可控落点

### 9.4 第四阶段：链路上下文化

目标：

- 让所有关键链路都能稳定检索 `query_id`
- 强化 `thread_id` / `sub_query_id` 的追踪能力

产出：

- 单次用户问题可从 API 层追到核心层，再追到子查询层
- 日志检索从“读全文猜上下文”升级为“按 ID 精准定位”

### 9.5 第五阶段：运维与文档收口

目标：

- 提供统一的排障手册
- 明确“看哪个日志文件解决什么问题”

产出：

- 服务启动文档
- 日志查看文档
- 生产问题排查流程

- 将 API 日志文件收敛为 `logs/app/api.log`
- 将 API logger 命名统一到 `nl2sql.api`
- 将部分核心模块 logger 命名收敛到更清晰的 `nl2sql.father`、`nl2sql.retrieval`、`nl2sql.execution`

## 10. 推荐的最终职责边界

### 10.1 `nl2sql.log`

用于回答：

- 一次 NL2SQL 请求走了哪条路径
- 父图和子图分别做了什么
- SQL 是怎么生成、校验、执行的
- 问题为何失败

### 10.2 `api.log`

用于回答：

- 某次 HTTP 请求是否进入系统
- 接口层是否校验失败
- 路由层是否抛出异常
- 应用启动/关闭是否正常

### 10.3 `process/*.out.log`

用于回答：

- 进程是否启动失败
- 运行时是否有未被 logging 捕获的标准错误输出
- Streamlit 前端侧是否存在框架级或运行期输出异常

### 10.4 关于 `frontend.log` 与 `access.log`

当前阶段，这两类日志都不作为默认目标：

- `frontend.log`
  当前没有足够必要性单独保留，且容易与 `streamlit.out.log` 重叠。

- `access.log`
  当前代码中没有这类日志。如果要引入，要么依赖 `uvicorn` 自身 access log，要么新增 FastAPI 中间件/配置，属于后续增强项，不应放入本轮基础治理范围。

## 11. 验收标准

当日志治理完成后，团队应能稳定满足以下验收条件：

1. 维护者看到文件名就能判断其职责边界。
2. 维护者只需查看一个权威配置位置，即可知道日志级别与落盘路径。
3. 任何核心链路日志都能在 `nl2sql.log` 中按 `query_id` 检索到。
4. API 层问题无需翻核心日志即可先在 `api.log` 中定位。
5. 前端问题优先在 `streamlit.out.log` 中排查，而不是在多个前端日志文件之间来回切换。
6. 进程输出日志与应用日志不再混放。
7. 不再存在“文档写着能配，但代码实际不生效”的日志配置项。
8. 不再出现“父图日志到底在哪份文件里”的认知歧义。

## 12. 本方案的明确建议

如果只给出一句结论，本项目日志体系应朝以下方向收敛：

- **核心链路统一写入 `logs/app/nl2sql.log`，不要继续沿用 `sql_subgraph.log` 这个误导性名称。**
- **CLI 和 FastAPI 共享一套生效的应用日志配置源；`uvicorn` / `streamlit` 进程日志保持独立，不强行并入项目 yaml。**
- **应用日志与进程输出日志分目录管理，彻底消除“哪个才是主日志”的混乱。**
- **围绕 `query_id` / `thread_id` 设计日志上下文，而不是围绕模块文件名设计。**

从整个项目的长期维护成本来看，这比恢复一个含义模糊的 `all.log` 更稳健，也更适合后续继续演进。
