# 3.1 数据画像模块改造说明

本文件独立定义 Step 2（JSON 数据画像生成）的目标、输入输出、核心算法与实现要求，可直接作为开发与测试依据。

---

## 3.1.1 模块定位与目标

- **定位**：在 Step 1（DDL 生成）完成后，利用“人工审核过的 DDL”提供的结构信息，加上实时数据库采样数据，为每张表生成完整的数据画像 JSON，作为 Step 3（Neo4j CQL）与 Step 4（Markdown）的唯一数据来源。
- **核心目标**：
  1. 仅依赖 DDL 获取表结构、主外键、唯一约束、索引、注释及样例记录，彻底断开直接读取数据库结构的通道。
  2. 通过数据库采样计算列统计指标，识别语义角色、结构标志、表类型等画像信息。
  3. 输出与 Step 1 一一对应的 JSON 文件，覆盖写入 `output/metaweave/metadata/json`。
  4. 为后续步骤提供一致可靠的结构定义，可容纳 DBA 在 DDL 上的人工修订。

---

## 3.1.2 输入与输出

| 类型 | 说明 | 备注 |
| --- | --- | --- |
| DDL 输入 | `output/metaweave/metadata/ddl/{schema}.{table}.sql` | Step 1 产出；可包含人工编辑的主键/外键/注释 |
| 样例记录 | DDL 内 `/* SAMPLE_RECORDS ... */` JSON 注释块 | 固定 3 条记录（不足插入 placeholder），供 Step 2/Step 4 共用 |
| 数据库连接 | 仅用于采样/统计/逻辑主键推断 | 禁止再读取表结构；所有结构以 DDL 为准 |
| JSON 输出 | `output/metaweave/metadata/json/{schema}.{table}.json` | 每张表一个文件，覆盖写入 |

---

## 3.1.3 执行流程

```
1. 扫描 DDL 目录（output/metaweave/metadata/ddl）
2. 逐表解析 SQL → 生成 TableMetadata + SampleRecords
3. 连接数据库，根据 TableMetadata.schema/table 拉取采样数据
4. 计算列统计、逻辑主键候选
5. 生成列画像（column_profile）、表画像（table_profile）
6. 合并结构信息 + 画像信息 → 写入 JSON 输出
7. 记录日志：DDL 缺失或解析失败即中止；统计失败单表跳过
```

> **关键约束**：从头到尾，结构信息仅来源于第 2 步的解析结果，后续逻辑不得再次访问数据库元数据。

---

## 3.1.4 DDL 解析职责

1. **结构恢复**：解析 `CREATE TABLE`、`CONSTRAINT`、`INDEX`、`COMMENT`，生成 `TableMetadata` 对象：
   - 列：名称、类型、长度/精度、是否可空、默认值、注释。
   - 主键、唯一约束、外键（含 on delete / on update）、索引。
   - 表注释。
2. **人工优先**：若 DDL 中包含数据库中不存在的约束或注释，也必须视为真实来源，解析器不可回连数据库做校验。
3. **样例记录**：提取 `/* SAMPLE_RECORDS ... */` 注释块并解析 JSON，得到 3 条样例数据的结构化表示。
4. **错误处理**：DDL 文件缺失或语法异常时，抛出明确错误并中止当前执行。

建议封装 `DDLLoader` 组件，对外提供：

```python
class ParsedDDL:
    metadata: TableMetadata
    sample_records: list[dict]

class DDLLoader:
    def load(self, ddl_path: Path) -> ParsedDDL: ...
```

---

## 3.1.5 数据采样与统计职责

1. **采样策略**：可复用 Step 1 的 sampling 配置（`limit` / `tablesample` / 范围采样），默认 1000 行。
2. **列统计指标**：
   - `count`, `unique_count`, `null_count`, `null_rate`, `uniqueness`
   - 数值型附带 `min`, `max`, `mean`
   - 低基数字段输出 `value_distribution`
3. **逻辑主键检测**：在缺失主键约束时，使用采样数据按照既定规则推断（唯一度=100% 且无空值），结果写入 JSON 的 `logical_keys`。
   - **最小候选键过滤**：系统会自动过滤掉 superkeys，只保留字段数最少的候选键（详见《2.数据库元数据生成模块设计》第 2.3.2 节）。
   - 例如：若 `[region_id]` 已唯一，则 `[region_id, city_id]` 等包含它的复合键会被自动过滤。
4. **画像写入**：根据统计结果填充 `column_profile` 和 `table_profile`（详见后续章节）。

## 3.1.6 列画像设计

### 语义角色（互斥 5 类）
| 角色 | 判定依据 | 示例 |
| --- | --- | --- |
| `identifier` | 数据类型白名单 + (字段名匹配 `*_id/_key/_code/_no/_number` 或完全是 `id/key` 或有物理约束) | `store_id`, `company_code` |
| `metric` | 数值型 + 字段名匹配 `amount/total/count/price/...` | `sales_amount`, `order_count` |
| `datetime` | 类型为 `date/datetime/timestamp/time` 或命名为 `*_date/_day/_month/...` | `created_at`, `date_day` |
| `enum` | `unique_count` ≤ 阈值（默认10）并存在 `value_distribution` | `status`, `level` |
| `attribute` | 兜底分类（名称、描述、地址等） | `store_name`, `address` |

**identifier 识别的前置条件（规则0）**：

必须通过数据类型白名单检查，才会进入后续的物理约束、统计特征、命名模式判断。

**允许的数据类型（白名单）**：
- 整数类型：`integer`, `bigint`, `smallint`, `serial`, `bigserial`, `smallserial`
- 字符串类型：`varchar`, `char`, `bpchar`
- UUID类型：`uuid`
- 数值类型（无小数）：`numeric(scale=0)`, `decimal(scale=0)`

**排除的数据类型（所有不在白名单中的）**：
- JSON类型：`json`, `jsonb`
- 数组类型：`array`, `integer[]`, `text[]` 等
- 二进制类型：`bytea`, `blob`, `binary`, `varbinary`
- 几何类型：`geometry`, `geography`, `point`, `linestring`, `polygon`
- 键值类型：`hstore`
- XML类型：`xml`
- 范围类型：`int4range`, `int8range`, `numrange`, `tsrange`, `daterange`
- 网络类型：`inet`, `cidr`, `macaddr`
- 位串类型：`bit`, `varbit`
- 其他：`boolean`, `text`, `float`, `timestamp` 等

**配置方式**：在 `configs/metaweave/metadata_config.yaml` 中的 `sampling.identifier_detection.allowed_data_types` 配置项中定义。使用白名单而非黑名单，确保安全性和可预测性。

### 结构标志（独立多选）
- `is_primary_key`：来自 DDL 的主键约束；若缺失则参考逻辑主键推断结果。
- `is_foreign_key`：仅来自 DDL 中的外键定义。
- `is_unique`：采样统计唯一度为 1.0。
- `is_unique_constraint`：DDL 唯一约束命中。
- `is_indexed`：DDL 中存在索引包含该字段。
- `is_nullable`：DDL 列定义。

### 列画像 JSON 结构

```json
"column_profile": {
  "semantic_role": "identifier",
  "semantic_confidence": 0.9,
  "structure_flags": {
    "is_primary_key": false,

    "is_unique_constraint": false,
    "is_indexed": true,
    "is_nullable": false
  },
  "identifier_info": {
    "naming_pattern": "table_id",

  },
  "metric_info": null,

    "source": "constraint",

    "composite_columns": null
  },
  "foreign_key_info": {
    "target_schema": "public",
    "target_table": "dim_store",
```

## 3.1.7 表画像设计

### 表类型
| 类型 | 判定条件 | 典型特征 |
| --- | --- | --- |
| `fact` | 度量列 ≥1，标识符列 ≥2，日期列 ≥1（满足其中 ≥2 项），可加命名/注释加分 | 销售事实表、流水表 |
| `dim` | 存在主键（或逻辑主键），度量列=0，标识符/属性列为主 | 公司维表、商品维表 |
| `bridge` | 外键 ≥2，列数 ≤5，度量列=0 或仅权重类字段 | 关系桥接表 |
| `unknown` | 不符合以上任一模式 | 临时表、配置表 |

### 表画像 JSON 结构

```json
"table_profile": {
  "table_category": "fact",
  "confidence": 0.95,
  "column_statistics": {
    "total_columns": 4,
    "identifier_count": 2,
    "metric_count": 1,
    "datetime_count": 1,
    "enum_count": 0,
    "attribute_count": 0,
    "primary_key_count": 0,
    "foreign_key_count": 0
  },
  "key_columns": {
    "primary_keys": [],
    "logical_keys": ["store_id", "date_day", "product_type_id"],
    "foreign_keys": []
  },
  "fact_table_info": {
    "grain": ["store_id", "date_day", "product_type_id"],
    "metrics": ["amount"],
    "dimensions": ["store_id", "date_day", "product_type_id"],
    "time_dimension": "date_day"

  "dim_table_info": null,
  "bridge_table_info": null,

    "has_metric_columns",
    "has_multiple_identifiers",
    "has_datetime_column",
    "naming_pattern_fact"
  ]
}
```

## 3.1.8 数据模型

定义与 Step 2 代码中的 dataclass 保持一致，核心模型包括：

- `TableMetadata`：描述表结构、约束、索引、逻辑主键（仅最小候选键）、样例记录。
- `ColumnInfo`：单字段信息。
- `ColumnProfile` / `StructureFlags` / `IdentifierInfo` / `MetricInfo` / `DateTimeInfo` / `EnumInfo`.
- `PrimaryKeyInfo`, `ForeignKeyInfo`, `IndexInfo`.
- `TableProfile`, `ColumnStatistics`, `KeyColumns`, `FactTableInfo`, `DimTableInfo`, `BridgeTableInfo`.
- `GenerationResult`：记录处理统计与错误。

开发时需保证 JSON 输出与这些模型字段一一对应。

## 3.1.9 配置项

`configs/metaweave/metadata_config.yaml` 中相关配置：

```yaml
sampling:
  enabled: true
  sample_size: 1000
  sample_method: limit
  column_statistics:
    enabled: true
    value_distribution_threshold: 10

profiling:
  enabled: true
  column_profiling:
    enabled: true
    enum_threshold: 10
    metric_patterns: [amount, total, sum, count, qty, quantity, price, cost, revenue, sales, profit, num, number, volume, rate, ratio, percent]
    identifier_patterns: [_id$, _key$, _code$, _no$, _number$, ^id$, ^key$]
    datetime_types: [date, datetime, timestamp, timestamptz, time]
  table_profiling:
    enabled: true
    fact_table:
      min_metric_columns: 1
      min_dimension_columns: 2
      require_datetime: true
      naming_patterns: ["^fact_", "_fact$", "^agg_", "^sum_"]
      comment_keywords: [事实表, 流水, 明细, 汇总]
    dim_table:
      require_primary_key: true
      max_metric_columns: 0
      naming_patterns: ["^dim_", "_dim$", "^d_"]
      comment_keywords: [维表, 维度, 字典]
    bridge_table:
      min_foreign_keys: 2
      max_columns: 5
      max_metric_columns: 1
      naming_patterns: ["^bridge_", "_bridge$"]
```

## 3.1.10 与 Step 3 / Step 4 的衔接

- **Step 3（Neo4j CQL）**：只消费 Step 2 的 JSON（结构 + 画像 + 采样统计），可额外连库获取需要的聚合指标，但不得直接解析数据库结构。
- **Step 4（Markdown）**：读取 Step 1 的 DDL 与 `SAMPLE_RECORDS`，保证字段示例与 Step 2 一致；必要时可复用 Step 2 计算的统计信息。
- **人工修改流程**：DBA 对 DDL 做的任何主外键/注释调整都会自动在 Step 2/3/4 的输出中体现，形成可审计链路。

## 3.1.11 实现与测试要求

1. **CLI 调用**：`python -m src.metaweave.cli.main metadata --config ... --step json`；`--step all` 需确保顺序为 Step 1 → Step 2 → Step 3 → Step 4。
2. **模块划分**：
   - `DDLLoader`：只负责文件解析。
   - `MetadataProfiler`：封装采样、统计、画像逻辑。
   - `JsonWriter`：结构化输出。
3. **错误处理**：
   - 缺少 DDL 文件或解析失败 → 立即终止并打印具体路径。
   - 样例记录缺失 → 允许继续但告警。
   - 采样/统计失败 → 标记该表失败并记录错误，其他表继续。
4. **测试建议**：
   - 单元测试：DDL 解析、列画像分类、表画像分类、统计函数。
   - 集成测试：mock 数据库 + 示例 DDL，验证端到端输出。
   - 回归测试：覆盖人工修改 DDL（新增外键/唯一约束）的场景，确保 JSON 结构同步更新。
