# 数据库元数据生成模块设计

## 1. 模块概述

### 1.1 模块定位
数据库元数据生成模块是MetaWeave RAG平台的基础模块，负责从PostgreSQL数据库中自动抽取和增强元数据信息，为后续的数据画像、关系发现、SQL生成等模块提供基础数据支撑。

### 1.2 核心目标
- 自动化抽取PostgreSQL数据库的完整元数据信息
- 智能补全缺失的表注释和字段注释
- 自动识别逻辑主键和唯一约束
- 生成标准化的DDL和MD文档

## 2. 功能需求

### 2.1 元数据抽取
#### 2.1.1 基础结构信息
- **表级信息**
  - 表名（schema.table_name）
  - 表注释（comment）
  - 表类型（table/view/materialized view）
  - 创建时间、修改时间（如果可获取）
  - 表的数据行数（估算值）

- **字段级信息**
  - 字段名称
  - 字段类型（含精度和长度）
  - 是否可空（nullable）
  - 默认值
  - 字段注释
  - 字段序号

#### 2.1.2 约束与索引信息
- **主键约束（Primary Key）**
  - 约束名称
  - 包含的字段列表
  - 约束定义

- **外键约束（Foreign Key）**
  - 约束名称
  - 源表和目标表
  - 源字段和目标字段的映射关系
  - 引用动作（ON DELETE/UPDATE）

- **唯一约束（Unique Constraint）**
  - 约束名称
  - 包含的字段列表
  - 是否部分索引

- **索引信息（Index）**
  - 索引名称
  - 索引类型（B-tree, Hash, GiST, GIN等）
  - 包含的字段列表及排序方式
  - 是否唯一索引
  - 索引条件（部分索引）

### 2.2 智能注释生成
当数据库元数据中缺少注释信息时，需要调用LLM生成语义化的注释。

#### 2.2.1 表注释生成
- **输入信息**
  - 表名
  - 所有字段名称和类型
  - 样本数据（前N行，可配置）
  - 字段注释（如果已存在）
  
- **生成策略**
  - 分析表名的语义含义
  - 结合字段结构推断表的业务用途
  - 参考样本数据确定数据特征
  - 生成简洁清晰的中文注释（一句话描述）

#### 2.2.2 字段注释生成
- **输入信息**
  - 字段名称
  - 字段类型
  - 该字段的样本数据
  - 表名和表注释（上下文）
  
- **生成策略**
  - 解析字段名的语义（驼峰、下划线命名等）
  - 分析字段类型和样本值的特征
  - 识别常见业务字段（ID、时间、状态等）
  - 生成准确的字段说明（10-30字）

#### 2.2.3 批量处理优化
- 支持批量调用LLM接口，减少网络开销
- 实现调用失败重试机制
- 支持缓存已生成的注释，避免重复调用
- 提供手动修正注释的接口

### 2.3 逻辑主键识别
对于缺少主键定义的表，通过样本数据分析识别潜在的逻辑主键。

#### 2.3.1 识别规则
- **单字段候选**
  - 字段唯一度 = 100%
  - 无空值
  - 字段名包含关键词（id, code, key, no等）
  
- **复合字段候选**
  - 字段组合的唯一度 = 100%
  - 所有字段均无空值
  - 优先考虑2-3个字段的组合
  
- **置信度评分**
  - 命名匹配度：30%
  - 唯一度：40%
  - 非空率：20%
  - 数据类型适配度：10%

#### 2.3.2 最小候选键过滤规则
根据关系数据库理论，候选键（Candidate Key）必须满足最小性原则：

**过滤算法**：
1. **Step 1**：从所有唯一组合（uniqueness=1.0, null_rate=0.0）中，找出字段数最少的所有组合作为候选键集合 S
2. **Step 2**：所有字段数大于 S 的组合视为 superkey（冗余组合），全部淘汰
3. **Step 3**：允许多个同级别候选键并存（互不包含的情况下）

**示例**：
- 如果 `[region_id]` 已满足唯一性，则 `[region_id, city_id]`、`[region_id, province_id]` 等都是 superkey，应被过滤
- 如果同时存在 `[order_id, product_id]` 和 `[customer_id, sku_id]` 两个2字段组合，且互不包含，则两者都保留

#### 2.3.3 输出标记
- 在元数据中标记识别出的逻辑主键（仅包含最小候选键）
- 记录置信度分数
- 支持多个候选主键的排序输出

### 2.4 数据采样
从每张表中抽取样本数据，用于后续的数据画像和注释生成。

#### 2.4.1 采样策略
- **采样数量**：可配置（默认1000行）
- **采样方法**
  - 方法1：`SELECT * FROM table LIMIT N`（快速）
  - 方法2：`SELECT * FROM table TABLESAMPLE SYSTEM(n)`（随机）
  - 方法3：按主键范围均匀采样（适用于大表）

#### 2.4.2 采样数据用途
- LLM注释生成的输入
- 逻辑主键识别的依据
- 后续数据画像模块的输入

### 3.4 SQL查询模板

#### 3.4.1 获取表信息
```sql
SELECT 
    t.schemaname,
    t.tablename,
    obj_description((t.schemaname || '.' || t.tablename)::regclass, 'pg_class') as table_comment,
    pg_stat_get_live_tuples((t.schemaname || '.' || t.tablename)::regclass) as row_count
FROM pg_tables t
WHERE t.schemaname = %s
ORDER BY t.tablename;
```

#### 3.4.2 获取字段信息
```sql
SELECT 
    c.column_name,
    c.ordinal_position,
    c.data_type,
    c.character_maximum_length,
    c.numeric_precision,
    c.numeric_scale,
    c.is_nullable,
    c.column_default,
    pgd.description as column_comment
FROM information_schema.columns c
LEFT JOIN pg_catalog.pg_statio_all_tables st 
    ON c.table_schema = st.schemaname 
    AND c.table_name = st.relname
LEFT JOIN pg_catalog.pg_description pgd 
    ON pgd.objoid = st.relid 
    AND pgd.objsubid = c.ordinal_position
WHERE c.table_schema = %s AND c.table_name = %s
ORDER BY c.ordinal_position;
```

#### 3.4.3 获取主键
```sql
SELECT 
    tc.constraint_name,
    array_agg(kcu.column_name ORDER BY kcu.ordinal_position) as columns
FROM information_schema.table_constraints tc
JOIN information_schema.key_column_usage kcu 
    ON tc.constraint_name = kcu.constraint_name
    AND tc.table_schema = kcu.table_schema
WHERE tc.constraint_type = 'PRIMARY KEY'
    AND tc.table_schema = %s
    AND tc.table_name = %s
GROUP BY tc.constraint_name;
```

#### 3.4.4 获取外键
```sql
SELECT
    tc.constraint_name,
    array_agg(kcu.column_name ORDER BY kcu.ordinal_position) as source_columns,
    ccu.table_schema AS target_schema,
    ccu.table_name AS target_table,
    array_agg(ccu.column_name ORDER BY kcu.ordinal_position) as target_columns,
    rc.delete_rule,
    rc.update_rule
FROM information_schema.table_constraints tc
JOIN information_schema.key_column_usage kcu
    ON tc.constraint_name = kcu.constraint_name
    AND tc.table_schema = kcu.table_schema
JOIN information_schema.constraint_column_usage ccu
    ON ccu.constraint_name = tc.constraint_name
    AND ccu.table_schema = tc.table_schema
JOIN information_schema.referential_constraints rc
    ON rc.constraint_name = tc.constraint_name
    AND rc.constraint_schema = tc.table_schema
WHERE tc.constraint_type = 'FOREIGN KEY'
    AND tc.table_schema = %s
    AND tc.table_name = %s
GROUP BY tc.constraint_name, ccu.table_schema, ccu.table_name, 
         rc.delete_rule, rc.update_rule;
```

#### 3.4.5 获取唯一约束
```sql
SELECT 
    tc.constraint_name,
    array_agg(kcu.column_name ORDER BY kcu.ordinal_position) as columns
FROM information_schema.table_constraints tc
JOIN information_schema.key_column_usage kcu 
    ON tc.constraint_name = kcu.constraint_name
    AND tc.table_schema = kcu.table_schema
WHERE tc.constraint_type = 'UNIQUE'
    AND tc.table_schema = %s
    AND tc.table_name = %s
GROUP BY tc.constraint_name;
```

#### 3.4.6 获取索引信息
```sql
SELECT
    i.indexname as index_name,
    am.amname as index_type,
    array_agg(a.attname ORDER BY array_position(ix.indkey, a.attnum)) as columns,
    ix.indisunique as is_unique,
    ix.indisprimary as is_primary,
    pg_get_expr(ix.indpred, ix.indrelid) as condition
FROM pg_indexes i
JOIN pg_class c ON c.relname = i.tablename
JOIN pg_index ix ON ix.indexrelid = (i.schemaname || '.' || i.indexname)::regclass
JOIN pg_class ic ON ic.oid = ix.indexrelid
JOIN pg_am am ON am.oid = ic.relam
JOIN pg_attribute a ON a.attrelid = c.oid AND a.attnum = ANY(ix.indkey)
WHERE i.schemaname = %s AND i.tablename = %s
GROUP BY i.indexname, am.amname, ix.indisunique, ix.indisprimary, 
         ix.indpred, ix.indrelid
ORDER BY i.indexname;
```

### 4.1 主流程
```
开始
  ↓
读取配置文件
  ↓
连接PostgreSQL数据库
  ↓
获取Schema列表 ←─────┐
  ↓                  │
获取表列表           │
  ↓                  │
┌─────────────┐      │
│  遍历每张表   │      │
└─────┬───────┘      │
      ↓              │
提取表基础信息        │
      ↓              │
提取字段信息          │
      ↓              │
提取约束和索引        │
      ↓              │
数据采样             │
      ↓              │
检查注释完整性        │
      ↓              │
  [缺少注释?]         │
   Yes ↓  No         │
调用LLM生成 ─→        │
      ↓              │

      ↓              │

      ↓              │

      ↓              │

      ↓              │

  ↓

```

### 4.2 注释生成流程
```
检查表注释
  ↓
[缺少?] ──Yes──→ 准备上下文（表名、字段、样本）
  │                     ↓
  No                构建LLM Prompt
  ↓                     ↓
检查字段注释          调用LLM API
  ↓                     ↓
[有缺少?] ──Yes──→   解析返回结果
  │                     ↓
  No                验证和清洗
  ↓                     ↓
  └──────────────→  保存生成的注释
                        ↓
                    记录生成日志
```

### 4.3 逻辑主键识别流程
```
检查是否已有主键
  ↓
[有主键?] ──Yes──→ 跳过识别
  │
  No
  ↓
分析单字段候选
  ↓
计算唯一度和非空率
  ↓
分析2字段组合
  ↓
分析3字段组合
  ↓
[找到候选?] ──No──→ 记录无主键
  │
  Yes
  ↓
计算置信度分数
  ↓
过滤 superkeys（保留最小候选键）
  ↓
排序候选项
  ↓
选取Top候选
  ↓
标记逻辑主键
```

- 字段补充说明：**仅在有内容时显示**，包含主键、外键关系、唯一约束、索引、数据类型精度等结构化信息
- 无空行，格式紧凑

## 7. 异常处理

### 7.1 数据库连接异常
- 连接超时重试（最多3次）
- 连接失败记录详细错误日志
- 提供友好的错误提示

### 7.2 LLM调用异常
- API调用失败自动重试
- 超时设置（默认30秒）
- 限流处理（Rate Limit）
- 降级策略：LLM不可用时跳过注释生成

### 7.3 数据异常
- 空表处理：记录警告，继续处理
- 超大表处理：采样行数自适应调整
- 特殊字符处理：转义和编码处理

### 7.4 权限异常
- 检查数据库用户权限
- 记录无权限访问的表
- 提供权限检查工具

### 10.2 命令行接口
```bash
# 完整生成
python -m src.metaweave.cli.main metadata --config configs/metaweave/metadata_config.yaml

# 指定schema
python -m src.metaweave.cli.main metadata --config configs/metaweave/metadata_config.yaml --schemas public,schema2

# 指定表
python -m src.metaweave.cli.main metadata --config configs/metaweave/metadata_config.yaml --tables users,orders

# 增量更新模式
python -m src.metaweave.cli.main metadata --config configs/metaweave/metadata_config.yaml --incremental

# 指定执行步骤
python -m src.metaweave.cli.main metadata --config configs/metaweave/metadata_config.yaml --step ddl

# 调试模式
python -m src.metaweave.cli.main metadata --config configs/metaweave/metadata_config.yaml --debug
```

`--step` 参数支持 `ddl/json/cql/md/all`，默认 `all`，会顺序执行 DDL→JSON→CQL→Markdown。指定 `ddl` 时，仅会在 `output/metaweave/metadata/ddl` 下覆盖生成 `.sql` 文件，不再产出其它格式或汇总文件。

为支撑 Step 2/3 的依赖解析，每个 DDL 文件末尾都会附带 `SAMPLE_RECORDS` 注释块，封装三条样例记录（不足三条时会使用占位记录），格式如下：

```
/* SAMPLE_RECORDS
{
  "version": 1,
  "table": "public.dim_company",
  "generated_at": "2025-11-19T12:34:56Z",
  "records": [
    {
      "label": "Record 1",
      "data": {
        "company_id": "1001",
        "company_name": "星环科技"
      }
    },
    ...
  ]
}
*/
```

解析时直接读取该 JSON 即可获取三条样例数据，无需执行 `INSERT`。
