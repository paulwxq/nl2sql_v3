## 1. 需求概述

### 1.1 需求背景

在关系发现模块生成的 `relationships_*.json` 文件中，每个关系对象需要增加一个 `cardinality` 属性，用于标识源表到目标表的关联基数关系。

### 1.2 基数类型

| 基数类型 | 含义 | 示例 |
|---------|------|------|
| `1:1` | 一对一 | `dim_store.store_id → dim_store_ext.store_id` |
| `1:N` | 一对多（源唯一→目标重复） | `dim_store.store_id → fact_sales.store_id` |
| `N:1` | 多对一（源重复→目标唯一） | `fact_sales.store_id → dim_store.store_id` |
| `M:N` | 多对多 | `fact_sales.product_id → fact_inventory.product_id`（通过中间表关联）|

**方向性说明**：基数关系与源表（source）和目标表（target）的指向相关，格式为 `source → target`。

### 1.3 设计目标

1. **准确性**：基于采样数据的真实计算，不依赖简化假设
2. **性能**：搭车评分计算流程，复用现有采样数据，最小化额外开销
3. **鲁棒性**：静态预判 + 动态验证，处理边界情况，M:N 兜底
4. **一致性**：单列关系和复合键关系使用统一逻辑

## 2. 计算逻辑设计

### 2.1 总体策略（方案C：混合判断）

采用"静态预判 + 动态验证 + M:N 兜底"三层策略：

```
┌─────────────────────────────────────┐
│  第一层：静态预判（基于唯一性）      │
│  - 使用采样数据的组合唯一性          │
│  - 快速识别明确的 1:1、1:N、N:1     │
└─────────────┬───────────────────────┘
              │
              ↓ 不确定？
┌─────────────────────────────────────┐
│  第二层：动态验证（基于 JOIN 倍率）  │
│  - 执行真实 JOIN COUNT 查询          │
│  - 精确判断边界情况                  │
└─────────────┬───────────────────────┘
              │
              ↓ 仍不确定？
┌─────────────────────────────────────┐
│  第三层：M:N 兜底                    │
│  - 无法判断时统一返回 M:N            │
└─────────────────────────────────────┘
```

### 2.2 核心参数定义

| 参数名 | 值 | 含义 |
|-------|-----|------|
| `HIGH_UNIQUENESS` | 0.95 | 高唯一度阈值，≥此值认为列"几乎唯一" |
| `LOW_UNIQUENESS` | 0.80 | 低唯一度阈值，<此值认为列"显著重复" |
| `JOIN_MULTIPLICITY_ONE` | 1.1 | JOIN倍率接近1的阈值，≤此值认为是一对一或多对一 |
| `JOIN_MULTIPLICITY_MANY` | 1.5 | JOIN倍率显著>1的阈值，>此值认为是一对多或多对多 |

### 2.3 算法流程

#### 算法输入

`_calculate_cardinality()` 方法的参数（来自 `_sample_and_calculate_inclusion()` 的部分返回值）：

| 参数 | 类型 | 说明 |
|-----|------|------|
| `source_uniqueness` | float | 源列组合唯一性（基于采样数据，使用有效行数作为分母，已排除 NULL 行） |
| `target_uniqueness` | float | 目标列组合唯一性（基于采样数据，使用有效行数作为分母，已排除 NULL 行） |
| `join_multiplicity` | float | JOIN 倍率（基于采样数据，使用 `source_valid_count` 作为分母） |

**说明**：
- `_sample_and_calculate_inclusion()` 实际返回 **5 个值**：`(inclusion_rate, jaccard_index, source_uniqueness, target_uniqueness, join_multiplicity)`
- 其中 `inclusion_rate` 和 `jaccard_index` 用于**评分计算**
- 其中 `source_uniqueness`、`target_uniqueness`、`join_multiplicity` 用于**基数计算**
- `source_valid_count`、`target_valid_count` 等是方法内部的中间变量，不作为返回值暴露

#### 判断逻辑（决策树）

```
                     开始
                      ↓
        ┌─────────────────────────┐
        │ source_uniq >= 0.95？    │
        └──────┬──────────┬────────┘
              YES        NO
               ↓          ↓
    ┌──────────────┐  ┌──────────────┐
    │target_uniq   │  │source_uniq   │
    │>= 0.95？     │  │< 0.8？       │
    └──┬─────┬─────┘  └──┬─────┬─────┘
      YES   NO          YES   NO
       ↓     ↓           ↓     ↓
     1:1   ┌────────┐  ┌────────┐  动态
           │target  │  │target  │  判断
           │< 0.8？ │  │>= 0.95?│   ↓
           └──┬──┬──┘  └──┬──┬──┘
             YES NO      YES NO
              ↓   ↓       ↓   ↓
            1:N  动态    N:1  动态
                判断          判断
```

**详细规则**：

1. **1:1 判断**
   ```
   IF source_uniqueness >= 0.95 AND target_uniqueness >= 0.95
   THEN return "1:1"
   ```

2. **1:N 判断**
   ```
   IF source_uniqueness >= 0.95 AND target_uniqueness < 0.8
   THEN return "1:N"
   ```

3. **N:1 判断**
   ```
   IF source_uniqueness < 0.8 AND target_uniqueness >= 0.95
   THEN return "N:1"
   ```

4. **动态判断（边界情况）**
   ```
   IF 任一唯一性在 [0.8, 0.95) 区间
   THEN 使用 join_multiplicity 辅助判断：
   
       IF join_multiplicity <= 1.1
       THEN 
           IF source_uniqueness >= target_uniqueness
           THEN return "1:1"
           ELSE return "N:1"
       
       ELSE IF join_multiplicity > 1.5
       THEN
           IF source_uniqueness >= 0.85
           THEN return "1:N"
           ELSE return "M:N"
       
       ELSE return "M:N"  # 倍率在 [1.1, 1.5] 不确定
   ```

5. **M:N 兜底**
   ```
   IF 所有条件都不满足
   THEN return "M:N"
   ```

### 2.4 关键计算公式

#### 2.4.1 组合唯一性（Composite Uniqueness）

**重要说明**：计算唯一性时，分母必须是"有效行数"（排除 NULL 后实际参与比较的行数），而不是原始采样行数。否则当列中存在大量 NULL 值时，唯一性会被严重低估。

**单列关系**：
```python
# 错误示例（分母不一致）：
# uniqueness = unique_count / sample_count  # ❌ NULL 行被排除但分母包含

# 正确公式：
uniqueness = unique_count / valid_count  # ✅ 分子分母一致
```

**复合键关系**：
```python
# 不使用平均值！使用组合后的唯一值数量
composite_values = {(col1_val, col2_val, ...), ...}  # 去重后的元组集合（已排除 NULL）
valid_count = 实际参与比较的行数（排除含 NULL 的行）
uniqueness = len(composite_values) / valid_count
```

**NULL 值处理**：
- 在构建值集合时，包含任意 NULL 的行会被跳过（现有 `_extract_value_set()` 逻辑）
- 因此 `unique_count` 只统计非 NULL 行的唯一值
- 分母必须同步使用 `valid_count`（非 NULL 行数），而非原始 `sample_count`

**示例（无 NULL）**：
- 采样 1000 行，无 NULL 值
- valid_count = 1000，去重后得到 950 个唯一组合
- 组合唯一性 = 950 / 1000 = 0.95 ✅

**示例（有 NULL）**：
- 采样 1000 行，其中 200 行含 NULL
- valid_count = 800（有效行数）
- 去重后得到 800 个唯一组合
- **错误计算**: 800 / 1000 = 0.80 ❌（误判为有重复）
- **正确计算**: 800 / 800 = 1.0 ✅（实际上完全唯一）

#### 2.4.2 JOIN 倍率（Join Multiplicity）

**计算方法**：在采样数据上执行真实 JOIN COUNT

```sql
WITH source_sample AS (
    SELECT col1, col2, ...
    FROM "schema"."source_table"
    LIMIT 1000  -- 采样大小（实际可能拿不到足额）
)
SELECT COUNT(*) AS join_count
FROM source_sample s
INNER JOIN "schema"."target_table" t
ON s.col1 = t.col1 AND s.col2 = t.col2  -- 多列 JOIN
```

**倍率公式**：
```python
# 错误公式（分母不准确）：
# join_multiplicity = join_count / self.sample_size  # ❌ 配置值，可能大于实际采样行数

# 正确公式：使用 source_valid_count 作为分母
join_multiplicity = join_count / source_valid_count  # ✅ 有效行数
```

**为什么使用 `source_valid_count` 而非 `sample_size` 或 `len(source_rows)`**：

1. **小表问题**：配置 `sample_size=1000`，但表只有 50 行时，`LIMIT 1000` 实际只取到 50 行。若用 1000 作为分母会严重低估倍率。
2. **NULL 问题**：INNER JOIN 会自动排除含 NULL 的行（`NULL = anything` 返回 FALSE），所以 `join_count` 的统计已排除 NULL。分母也应使用排除 NULL 后的有效行数，保持一致。
3. **语义一致**：与唯一性计算使用相同的分母 `source_valid_count`，确保基数判断的各个指标在同一口径下计算。

**含义**：
- `= 1.0`：每个有效源记录对应 1 条目标记录（1:1 或 N:1）
- `> 1.0`：每个有效源记录对应多条目标记录（1:N 或 M:N）
- `< 1.0`：部分有效源记录无法 JOIN 上（数据质量问题或目标表缺失）

**示例（正常情况）**：
- 源表采样 1000 行，无 NULL，`source_valid_count = 1000`
- JOIN 后得到 1500 行
- 倍率 = 1500 / 1000 = 1.5（说明源表记录在目标表有扩散，可能是 1:N）

**示例（小表 + NULL）**：
- 配置 `sample_size = 1000`，实际表只有 100 行
- 其中 20 行含 NULL，`source_valid_count = 80`
- JOIN 后得到 160 行
- **错误计算**：160 / 1000 = 0.16 ❌（严重低估）
- **正确计算**：160 / 80 = 2.0 ✅（正确识别为 1:N）

```python
def _extract_value_set(self, rows: List[Dict], columns: List[str]) -> Tuple[Set[Tuple], int]:
    """从查询结果中提取值集合
    
    Args:
        rows: 查询结果行列表
        columns: 列名列表
    
    Returns:
        (value_set, valid_count): 
        - value_set: 值元组集合（多列组合为元组，已排除含 NULL 的行）
        - valid_count: 有效行数（排除含 NULL 的行后实际参与比较的行数）
    """
    value_set = set()
    valid_count = 0
    
    for row in rows:
        values = tuple(row.get(col) for col in columns)
        # 跳过包含 NULL 的值
        if None not in values:
            value_set.add(values)
            valid_count += 1
    
    return value_set, valid_count
```

##### 修改点 3：新增 `_execute_join_count()` 方法

**位置**：在 `_sample_and_calculate_inclusion()` 之后新增

**代码**：
```python
def _execute_join_count(
        self,
        source_schema: str,
        source_table: str,
        source_columns: List[str],
        target_schema: str,
        target_table: str,
        target_columns: List[str],
        source_valid_count: int  # 新增参数：源表有效行数（排除 NULL 后）
) -> float:
    """执行 JOIN COUNT 获取 JOIN 倍率
    
    在采样数据上执行真实 JOIN 查询，计算源表记录的扩散倍数。
    
    Args:
        source_schema: 源schema
        source_table: 源表
        source_columns: 源列列表
        target_schema: 目标schema
        target_table: 目标表
        target_columns: 目标列列表
        source_valid_count: 源表有效行数（排除 NULL 后实际参与比较的行数）
    
    Returns:
        join_multiplicity: JOIN 倍率（join_count / source_valid_count）
        - = 1.0: 每个有效源记录对应 1 条目标记录（1:1 或 N:1）
        - > 1.0: 每个有效源记录对应多条目标记录（1:N 或 M:N）
        - < 1.0: 部分有效源记录无法 JOIN 上（数据质量问题）
    
    Note:
        使用 source_valid_count 而非 self.sample_size 作为分母，原因：
        1. 小表问题：LIMIT 可能拿不到足额样本
        2. NULL 问题：INNER JOIN 自动排除 NULL 行，分母应保持一致
    """
    try:
        # 构造列表达式
        source_col_expr = ", ".join([f'"{col}"' for col in source_columns])
        
        # 构造 JOIN ON 条件
        join_conditions = " AND ".join([
            f's."{src_col}" = t."{tgt_col}"'
            for src_col, tgt_col in zip(source_columns, target_columns)
        ])
        
        # JOIN COUNT SQL（在采样数据上执行）
        join_sql = f'''
            WITH source_sample AS (
                SELECT {source_col_expr}
                FROM "{source_schema}"."{source_table}"
                LIMIT %s
            )
            SELECT COUNT(*) AS join_count
            FROM source_sample s
            INNER JOIN "{target_schema}"."{target_table}" t
            ON {join_conditions}
        '''
        
        result = self.connector.execute_query(join_sql, (self.sample_size,))
        join_count = result[0].get('join_count', 0) if result else 0
        
        # 计算倍率：使用传入的 source_valid_count 作为分母（修复小表和 NULL 问题）
        multiplicity = join_count / source_valid_count if source_valid_count > 0 else 0.0
        
        logger.debug(
            f"JOIN COUNT: {source_schema}.{source_table} → {target_schema}.{target_table}, "
            f"source_valid_count={source_valid_count}, join_count={join_count}, multiplicity={multiplicity:.3f}"
        )
        
        return multiplicity
        
    except Exception as e:
        logger.error(f"执行 JOIN COUNT 失败: {e}", exc_info=True)
        return 1.0  # 失败时返回 1.0（保守估计，避免误判为 M:N）
```

##### 修改点 4：新增 `_calculate_cardinality()` 方法

**位置**：在 `_execute_join_count()` 之后新增

**代码**：
```python
def _calculate_cardinality(
        self,
        source_uniqueness: float,
        target_uniqueness: float,
        join_multiplicity: float
) -> str:
    """计算关系基数（静态+动态混合判断）
    
    三层判断策略：
    1. 静态预判（基于唯一性）：快速识别明确的 1:1、1:N、N:1
    2. 动态验证（基于 JOIN 倍率）：处理边界情况
    3. M:N 兜底：无法判断时统一返回 M:N
    
    Args:
        source_uniqueness: 源列组合唯一性（采样数据）
        target_uniqueness: 目标列组合唯一性（采样数据）
        join_multiplicity: JOIN 倍率（采样数据）
    
    Returns:
        基数类型: "1:1" | "1:N" | "N:1" | "M:N"
    """
    HIGH = 0.95  # 高唯一度阈值
    LOW = 0.80   # 低唯一度阈值
    
    logger.debug(
        f"基数判断输入: source_uniq={source_uniqueness:.3f}, "
        f"target_uniq={target_uniqueness:.3f}, join_mult={join_multiplicity:.3f}"
    )
    
    # === 第一层：静态预判（基于唯一性） ===
    
    # 1:1 - 双方都高度唯一
    if source_uniqueness >= HIGH and target_uniqueness >= HIGH:
        logger.debug("基数判断: 1:1（双方高唯一）")
        return "1:1"
    
    # 1:N - 源唯一，目标重复
    if source_uniqueness >= HIGH and target_uniqueness < LOW:
        logger.debug("基数判断: 1:N（源唯一，目标重复）")
        return "1:N"
    
    # N:1 - 源重复，目标唯一
    if source_uniqueness < LOW and target_uniqueness >= HIGH:
        logger.debug("基数判断: N:1（源重复，目标唯一）")
        return "N:1"
    
    # === 第二层：动态验证（边界情况） ===
    
    logger.debug(f"进入动态判断（唯一性在边界区间 [{LOW}, {HIGH})）")
    
    # 倍率接近 1 → 一对一或多对一
    if join_multiplicity <= 1.1:
        if source_uniqueness >= target_uniqueness:
            logger.debug("基数判断: 1:1（倍率≈1，源更唯一）")
            return "1:1"
        else:
            logger.debug("基数判断: N:1（倍率≈1，目标更唯一）")
            return "N:1"
    
    # 倍率显著 > 1 → 一对多或多对多
    if join_multiplicity > 1.5:
        if source_uniqueness >= 0.85:
            logger.debug("基数判断: 1:N（倍率>1.5，源较唯一）")
            return "1:N"
        else:
            logger.debug("基数判断: M:N（倍率>1.5，双方都不唯一）")
            return "M:N"
    
    # === 第三层：M:N 兜底 ===
    
    logger.debug("基数判断: M:N（兜底，无法明确判断）")
    return "M:N"
```

#### 3.2.5 `repository.py` - 外键关系的基数计算

**背景说明**：

外键关系从 `MetadataRepository.extract_fk_relations()` 方法直接创建 `Relation` 对象，**不经过 scorer**。如果不修改，这些关系会保持默认的 `cardinality="N:1"`，与"基于数据的真实计算"目标不符。

**解决方案**：升级 `_infer_cardinality()` 方法，使用 JSON 元数据中的物理约束和统计值进行静态判断。

##### 唯一性的两种来源

| 来源 | 位置 | 优先级 | 说明 |
|-----|------|-------|------|
| 物理约束 | `structure_flags.is_primary_key`<br>`structure_flags.is_unique_constraint` | 高 | DDL 定义的约束，最可靠 |
| 统计值 | `statistics.uniqueness` | 低 | 采样计算的唯一性，作为 fallback |

##### 修改点：升级 `_infer_cardinality()` 方法

**位置**：第 260-287 行

**当前代码（简化实现）**：

**修改后（静态判断）**：
```python
def _infer_cardinality(
        self,
        fk: Dict[str, Any],
        tables: Dict[str, dict],
        source_full_name: str,
        target_schema: str,
        target_table: str
) -> str:
    """推断外键关系的基数
    
    优先级：物理约束 > 统计值
    
    判断逻辑：
    - 源列唯一 + 目标列唯一 → 1:1
    - 源列唯一 + 目标列不唯一 → 1:N
    - 源列不唯一 + 目标列唯一 → N:1
    - 双方都不唯一 → M:N
    
    Args:
        fk: 外键信息
        tables: 所有表元数据
        source_full_name: 源表全名
        target_schema: 目标schema
        target_table: 目标表名
    
    Returns:
        基数（1:1 | 1:N | N:1 | M:N）
    """
    source_columns = fk["source_columns"]
    target_columns = fk.get("target_columns", fk.get("referenced_columns", []))
    target_full_name = f"{target_schema}.{target_table}"
    
    # 判断源列和目标列的唯一性
    source_is_unique = self._is_columns_unique(tables, source_full_name, source_columns)
    target_is_unique = self._is_columns_unique(tables, target_full_name, target_columns)
    
    # 判断基数
    if source_is_unique and target_is_unique:
        return "1:1"
    elif source_is_unique and not target_is_unique:
        return "1:N"
    elif not source_is_unique and target_is_unique:
        return "N:1"
    else:
        return "M:N"
```

##### 新增辅助方法：`_is_columns_unique()`

```python
def _is_columns_unique(
        self,
        tables: Dict[str, dict],
        full_name: str,
        columns: List[str]
) -> bool:
    """判断列（单列或复合列）是否唯一
    
    优先级：物理约束 > 统计值
    
    Args:
        tables: 所有表元数据
        full_name: 表全名（schema.table）
        columns: 列名列表
    
    Returns:
        True: 列是唯一的
        False: 列不唯一或无法判断
    """
    table = tables.get(full_name)
    if not table:
        logger.warning(f"表元数据不存在: {full_name}")
        return False
    
    profiles = table.get("column_profiles", {})
    table_profile = table.get("table_profile", {})
    
    # === 1. 检查物理约束（优先） ===
    
    # 单列情况
    if len(columns) == 1:
        col_name = columns[0]
        col_profile = profiles.get(col_name, {})
        flags = col_profile.get("structure_flags", {})
        
        # 主键或唯一约束 → 唯一
        if flags.get("is_primary_key") or flags.get("is_unique_constraint"):
            logger.debug(f"{full_name}.{col_name}: 物理约束判定为唯一")
            return True
    
    # 复合列情况：检查复合主键/唯一约束
    else:
        physical = table_profile.get("physical_constraints", {})
        
        # 检查复合主键
        pk = physical.get("primary_key")
        if pk and set(pk.get("columns", [])) == set(columns):
            logger.debug(f"{full_name}.{columns}: 复合主键，判定为唯一")
            return True
        
        # 检查复合唯一约束
        for uk in physical.get("unique_constraints", []):
            if set(uk.get("columns", [])) == set(columns):
                logger.debug(f"{full_name}.{columns}: 复合唯一约束，判定为唯一")
                return True
    
    # === 2. Fallback 到统计值 ===
    
    HIGH_UNIQUENESS = 0.95  # 与 scorer 使用相同阈值
    
    if len(columns) == 1:
        col_name = columns[0]
        col_profile = profiles.get(col_name, {})
        stats = col_profile.get("statistics", {})
        uniqueness = stats.get("uniqueness", 0.0)
        
        if uniqueness >= HIGH_UNIQUENESS:
            logger.debug(f"{full_name}.{col_name}: 统计值 uniqueness={uniqueness:.3f} >= {HIGH_UNIQUENESS}，判定为唯一")
            return True
    else:
        # 复合列：取最小唯一性（保守估计）
        # 注：更准确的做法需要组合计算，这里简化处理
        min_uniqueness = 1.0
        for col_name in columns:
            col_profile = profiles.get(col_name, {})
            stats = col_profile.get("statistics", {})
            uniqueness = stats.get("uniqueness", 0.0)
            min_uniqueness = min(min_uniqueness, uniqueness)
        
        if min_uniqueness >= HIGH_UNIQUENESS:
            logger.debug(f"{full_name}.{columns}: 组合统计值 min_uniqueness={min_uniqueness:.3f} >= {HIGH_UNIQUENESS}，判定为唯一")
            return True
    
    logger.debug(f"{full_name}.{columns}: 未满足唯一条件")
    return False
```

##### 关系类型对比

| 关系类型 | 数据来源 | 基数判断方式 | 是否经过 scorer |
|---------|---------|------------|---------------|
| **外键关系** | `physical_constraints.foreign_keys` | 静态判断（物理约束 + 统计值） | ❌ 否 |
| **推断关系** | 候选生成 + 评分 | 动态计算（采样数据 + JOIN COUNT） | ✅ 是 |

```python
    )

```

## 6. 风险与缓解

### 6.1 风险点

| 风险 | 影响 | 概率 | 缓解措施 |
|-----|------|------|---------|
| JOIN COUNT 查询性能慢 | 影响整体运行时间 | 中 | 1. 仅在采样数据上执行<br>2. 增加超时和异常处理 |
| 采样数据不代表整体 | 基数判断不准确 | 中 | 1. 增大采样量（1000→5000）<br>2. 结合静态和动态判断 |
| 复杂JOIN导致错误 | 程序崩溃 | 低 | 1. 完善异常处理<br>2. 失败时返回保守估计 |
| 阈值不适配特定数据 | 误判基数类型 | 中 | 1. 提供配置项<br>2. 根据实际数据调优 |
| **列 NULL 率过高** | 有效行数太少，基数判断不可靠 | 低 | 1. 使用有效行数作为分母（已实现）<br>2. NULL 率 > 50% 时输出警告日志<br>3. 极端情况下可考虑返回 M:N 兜底 |

### 6.2 回退方案

如果 JOIN COUNT 性能不可接受，可以回退到方案 B（仅基于唯一性判断）：

```python
def _calculate_cardinality_simple(
    self,
    source_uniqueness: float,
    target_uniqueness: float
) -> str:
    """简化版：仅基于唯一性判断，不执行 JOIN COUNT"""
    HIGH = 0.95
    LOW = 0.80
    
    if source_uniqueness >= HIGH and target_uniqueness >= HIGH:
        return "1:1"
    elif source_uniqueness >= HIGH and target_uniqueness < LOW:
        return "1:N"
    elif source_uniqueness < LOW and target_uniqueness >= HIGH:
        return "N:1"
    else:
        return "M:N"
```
