> **文档目的**：规划并记录复合键候选生成中语义角色过滤逻辑的优化方案
> 
> **背景**：当前实现在处理物理约束和逻辑约束时缺乏差异化策略，且配置方式不一致

## 二、优化目标

### 2.1 核心原则

1. **尊重物理约束**：物理约束是 DBA 的明确意图，系统应**完全遵从，不进行任何语义角色过滤**
2. **逻辑约束可配置**：逻辑主键由系统推断，应该可以通过配置控制排除规则
3. **配置一致性**：单列和复合键的配置方式应保持一致
4. **特权模式真正特权**：物理约束匹配时**完全不受** semantic_role 过滤限制（包括 metric）
5. **配置统一管理**：`LogicalKeyDetector` 和 `CandidateGenerator` 必须使用相同的配置源

### 2.2 物理约束术语说明

> ⚠️ **重要**：源表和目标表的"物理约束"定义不同，必须严格区分

#### 源表物理约束（Source Physical Constraints）

**定义**：
- ✅ **仅包含**：物理主键（Primary Key）、物理唯一约束（Unique Constraint）
- ❌ **不包含**：
  - 索引（Index）：索引不作为源表的关系发起约束
  - 物理外键（Foreign Key）：外键走专用的"直通通道"，不参与常规候选生成

**适用范围**：
- 复合键匹配的源表约束（`combo_type == "physical"`）
- 单列匹配的源列筛选（`has_defined_constraint`）

**代码检查方法**：
```python
# 在 candidate_generator.py 中
def _has_defined_constraint(self, col_profile: dict) -> bool:
    flags = col_profile.get("structure_flags", {})
    return (
        flags.get("is_primary_key") or      # ✅ 物理主键
        flags.get("is_unique") or           # ✅ 物理唯一约束
        flags.get("is_unique_constraint")   # ✅ 物理唯一约束（另一种标记）
    )
    # ❌ 注意：不检查 is_indexed
```

**设计理由**：
- 源表的约束是"发起关系查找"的依据，应该更严格
- 只有 PK/UK 才明确表达了 DBA 定义的唯一性约束
- 索引仅是性能优化标记，不代表业务唯一性

---

#### 目标表物理约束（Target Physical Constraints）

**定义（广义）**：
- ✅ **包含**：物理主键（Primary Key）+ 物理唯一约束（Unique Constraint）+ **索引（Index）**
- 这些字段在关系候选阶段一律视为**"物理特权字段"**，**不进行语义角色过滤**

**适用范围**：
- 复合键匹配的目标表约束（`target_combo["type"] == "physical"`）
- 单列匹配的目标列筛选（`target_has_physical`）

**代码检查方法**：
```python
# 在 candidate_generator.py 中（单列匹配）
target_structure_flags = target_col_profile.get("structure_flags", {})
target_has_physical = (
    target_structure_flags.get("is_primary_key") or          # ✅ 物理主键
    target_structure_flags.get("is_unique") or               # ✅ 物理唯一约束
    target_structure_flags.get("is_unique_constraint") or    # ✅ 物理唯一约束（另一种标记）
    target_structure_flags.get("is_indexed")                 # ✅ 索引（目标表特有）
)
```

**设计理由**：
- 目标表是"接收关系匹配"的一方，应该更宽松
- 索引表明该字段经常被用于查询条件（包括 JOIN），很可能是关系字段
- 即使索引字段的语义角色是 `metric`，也可能被用于关联（如按金额范围关联）

---

#### 对比总结

| 维度 | 源表物理约束 | 目标表物理约束 |
|------|------------|--------------|
| **包含范围** | PK + UK | PK + UK + **Index** |
| **是否包含索引** | ❌ 否 | ✅ **是** |
| **是否包含外键** | ❌ 否（走直通通道） | ❌ 否 |
| **设计理念** | 严格：只有明确的唯一性约束 | 宽松：包含所有可能的关联字段标记 |
| **语义过滤** | 完全不过滤 | 完全不过滤 |
| **适用阶段** | 候选关系的"发起侧" | 候选关系的"接收侧" |

---

### 2.3 配置架构统一说明

**配置源头（单一真相源）**：

```yaml
# metadata_config.yaml
composite:
  exclude_semantic_roles:  # 👈 唯一配置源
    - metric
    # 将来可以添加 description
```

**配置消费者（两处使用同一配置）**：

| 消费者 | 用途 | 代码位置 | 使用场景 |
|--------|------|---------|---------|
| `LogicalKeyDetector` | 生成**复合逻辑主键**时过滤字段 | `logical_key_detector.py` | Step 2: JSON 生成阶段 |
| `CandidateGenerator` | 匹配关系时过滤**逻辑约束**的目标列 | `candidate_generator.py` | Step 3: 关系发现阶段 |

**配置传递链路**：

```
metadata_config.yaml (composite.exclude_semantic_roles)
    ↓
MetadataGenerator._init_components
    ├→ 传递给 LogicalKeyDetector(config["composite_exclude_roles"])
    └→ 传递给 CandidateGenerator(config["composite"])
         ↓
    两者读取到的配置必须一致！
```

**关键要求**：
1. ✅ **单一配置源**：只在 YAML 中定义一次
2. ✅ **统一传递**：通过 config 传递给所有消费者
3. ✅ **默认值一致**：所有地方的默认值都是 `["metric"]`
4. ❌ **禁止各自硬编码**：不允许在代码中硬编码不同的值

### 2.4 预期效果

| 场景 | 修改前（问题行为） | 修改后（正确行为） |
|------|------------------|-------------------|
| 源表 PK 包含 metric（物理约束） | ❌ 被错误拒绝 | ✅ 尊重物理约束，完全不过滤 |
| 源表逻辑主键包含 metric（单列） | ❌ 重复过滤 | ✅ 元数据生成时已过滤，候选生成时不再重复过滤 |
| 源表逻辑主键包含 metric（复合键） | ❌ 重复过滤风险 | ✅ 元数据生成时已过滤，候选生成时不会出现 |
| 目标表 PK 包含 metric（物理） | ⚠️ 被错误过滤 | ✅ **完全不过滤，完整匹配** |
| 目标表逻辑约束包含 metric | ⚠️ 被过滤但缺乏配置 | ✅ 按配置过滤（可配置） |
| 复合键排除角色配置 | ❌ 硬编码 | ✅ 从 YAML 读取（配置化） |
| 动态同名对物理约束的处理 | ⚠️ 被错误过滤 | ✅ **完全不过滤** |

## 三、关键问题解决方案

### 3.0 本次修改解决的8个关键问题

> 本节详细说明本次修改针对的具体问题，每个问题都来自实际代码审查

#### 问题 1：逻辑主键检测器复合键排除角色硬编码

**问题描述**：
- `LogicalKeyDetector.__init__` 中 `self.composite_exclude_roles = {"metric"}` 硬编码
- 没有从 YAML 配置读取，与设计文档不一致

**解决方案**：
- 在 `metadata_config.yaml` 中新增 `composite.exclude_semantic_roles`
- 在 `MetadataGenerator._init_components` 中传递配置
- 在 `LogicalKeyDetector.__init__` 中从配置读取，移除硬编码

**影响范围**：逻辑主键生成（Step 2: JSON 生成阶段）

---

#### 问题 2：候选生成器复合键排除角色硬编码 & 未走配置

**问题描述**：
- `CandidateGenerator.__init__` 中 `self.composite_exclude_semantic_roles = {"metric", "description"}` 硬编码
- 没有使用 YAML 中的 `composite.exclude_semantic_roles`
- 默认值与 `LogicalKeyDetector` 不一致（一个是 `{"metric"}`，一个是 `{"metric", "description"}`）

**解决方案**：
- 从 `composite.exclude_semantic_roles` 读取配置
- 确保与 `LogicalKeyDetector` 使用相同的配置源
- 统一默认值为 `["metric"]`

**影响范围**：关系候选生成（Step 3: 关系发现阶段）

---

#### 问题 3：复合键匹配中源列被语义角色过滤，破坏物理特权

**问题描述**：
- `_match_columns_as_set` 中对源列检查 `semantic_role`，如果在排除列表中直接 `return None`
- 不区分物理约束（PK/UK）和逻辑约束，统一拒绝
- 导致源表复合 PK/UK 中包含 metric 时被拒绝

**解决方案**：
- 源表列**完全不过滤**，直接使用 `source_columns`
- 移除所有源表的语义角色过滤逻辑
- 理由：所有约束（物理 PK/UK + 逻辑主键）都已经在其来源阶段进行了过滤

**影响范围**：复合键匹配算法（Stage 1）

**注意**：源表物理约束仅包括 PK/UK，不包含索引

---

#### 问题 4：复合键匹配中目标列统一按语义角色过滤，未区分物理/逻辑

**问题描述**：
- `_match_columns_as_set` 中对目标列统一使用 `if tgt_semantic_role in self.composite_exclude_semantic_roles: continue`
- 不区分目标组合是物理约束还是逻辑约束
- 导致目标表 PK/UK/索引中包含 metric 时被过滤

**解决方案**：
- 增加 `target_is_physical` 参数
- 物理约束（PK/UK/索引）：**完全不过滤**任何语义角色（包括 metric）
- 逻辑约束：按 `self.composite_exclude_semantic_roles` 配置过滤

**影响范围**：复合键匹配算法（Stage 1）

**注意**：目标表物理约束包括 PK/UK/**索引**（广义物理约束）

---

#### 问题 5：`_find_dynamic_same_name` 对物理/逻辑不区分且使用语义过滤

**问题描述**：
- `_find_dynamic_same_name` 不区分物理/逻辑，统一对源表和目标表进行语义过滤
- 与"动态同名是为源表物理约束设计的兜底策略"的设计意图冲突

**解决方案**：
- 调整调用时机：**只对源表 `combo_type == "physical"` 执行**
- 增加 `is_physical` 参数
- 源表物理约束（PK/UK）：**完全不过滤**
- 目标表：**在源表为物理约束（PK/UK）时，目标表任何列都可作为候选，且完全不过滤语义角色**
- 移除动态同名匹配中的所有语义角色过滤逻辑

**影响范围**：复合键匹配算法（Stage 2）

**关键前提**：
- ⚠️ **触发条件**：只在源表为物理约束（PK/UK）且 Stage 1 失败时才执行 Stage 2
- 源表物理约束：仅 PK/UK（不含索引）
- 目标表：任何列都可匹配（包括有索引的列），完全不过滤语义角色

---

#### 问题 6：单列候选中源列先按语义角色过滤，物理列被提前丢弃

**问题描述**：
- `_generate_single_column_candidates` 中先检查 `semantic_role`，再检查约束类型
- 导致物理约束列（PK/UK）和逻辑主键如果是 metric 会被提前过滤

**解决方案**：
- 调整顺序：先检查约束类型，确认是源列（物理约束或逻辑主键）
- **源列完全不过滤**：无论复合键还是单列，物理约束（PK/UK）和逻辑主键在候选生成阶段都不再按语义角色过滤
- 理由：逻辑主键在元数据生成阶段已按 `single_column_exclude_roles` 过滤，候选阶段只负责使用，不再二次过滤
- 设计原则：与 3.1 节"源列完全不过滤"原则保持一致

**影响范围**：单列候选生成（源列）

**注意**：源列物理约束仅包括 PK/UK，不包含索引

---

#### 问题 7：单列候选中目标列语义过滤未区分物理/逻辑

**问题描述**：
- 目标列统一使用 `if target_role in self.exclude_semantic_roles: continue`
- 不区分目标列是否有物理约束（PK/UK/索引）

**解决方案**：
- 检查目标列的 `structure_flags`（`is_primary_key`/`is_unique`/`is_indexed`）
- 物理约束（PK/UK/索引）：**完全不过滤**
- 非物理约束：按 `self.exclude_semantic_roles` 配置过滤

**影响范围**：单列候选生成（目标列）

**注意**：目标列物理约束包括 PK/UK/**索引**（广义物理约束）

---

#### 问题 8：`composite.exclude_semantic_roles` 与实际代码未打通

**问题描述**：
- YAML 中没有 `composite.exclude_semantic_roles` 字段
- `LogicalKeyDetector` 和 `CandidateGenerator` 各自硬编码排除集合
- 配置与实现脱节

**解决方案**：
- 在 `metadata_config.yaml` 中正式添加 `composite.exclude_semantic_roles`
- 在 `MetadataGenerator._init_components` 中统一读取并传递给两个消费者
- 确保配置在整个系统中保持一致

**影响范围**：配置管理（全局）

### 3.1 修改原则总结

| 原则 | 说明 | 适用范围 |
|------|------|---------|
| **物理约束完全不过滤** | **源表**：只考虑 PK/UK（不含索引、外键走直通通道），不做语义过滤；<br>**目标表**：PK/UK/**索引**都视为物理特权字段，不做语义过滤 | 源列、目标列、复合键、单列 |
| **逻辑约束可配置过滤** | 1）**逻辑主键**在**元数据生成阶段**按 `single_column_exclude_roles` / `composite_exclude_roles` 过滤；<br>2）**候选生成阶段**仅对**目标表逻辑约束列**按 `exclude_semantic_roles` / `composite_exclude_semantic_roles` 过滤，**不再对源列逻辑主键二次过滤** | 元数据生成（逻辑主键检测）、候选生成（目标列逻辑约束） |
| **源列完全不过滤** | 无论复合键还是单列，**源列**（物理约束+逻辑主键）在候选生成阶段**完全不过滤**；逻辑主键在元数据阶段已按配置筛选，候选阶段只负责使用，不再二次过滤 | 复合键匹配（Stage 1 & 2）、单列候选 |
| **配置统一管理** | `composite.exclude_semantic_roles` 在 YAML 中定义一次，被两个消费者共享 | 全局配置 |
| **动态同名只对物理约束** | Stage 2 是为**源表物理约束**（PK/UK）设计的兜底策略，只在源表为物理约束且 Stage 1 失败时执行；此时源表和目标表都**完全不过滤**语义角色 | 复合键匹配（Stage 2） |

| 源表逻辑主键包含 metric（单列） | ❌ 候选生成时重复过滤 | ✅ 元数据阶段已过滤，候选阶段不再重复过滤 | ✅ 逻辑清晰 |
| 目标表物理约束（PK/UK/索引）包含 metric | ⚠️ 被错误过滤 | ✅ **完全不过滤** | ⚠️ 可能新增关系（合理） |
| 目标表索引列包含 metric | ⚠️ 被错误过滤 | ✅ **完全不过滤** | ⚠️ 可能新增关系（合理） |
| 目标表逻辑约束包含 metric | ⚠️ 被过滤（但缺乏配置） | ✅ 按配置过滤（可配置） | ✅ 更灵活 |
| 动态同名对源表物理约束的处理 | ⚠️ 被错误过滤 | ✅ **完全不过滤** | ⚠️ 可能新增关系（合理） |
| LogicalKeyDetector 配置 | ❌ 硬编码 | ✅ 从 YAML 读取 | ✅ 配置统一 |
| CandidateGenerator 配置 | ❌ 硬编码 | ✅ 从 YAML 读取 | ✅ 配置统一 |

**影响评估**：
- ⚠️ 中等影响：可能新增一些之前被错误拒绝的关系（主要是物理约束关系）
- ✅ 正向变化：这些关系本应被识别（物理约束是 DBA 的明确意图）
- ✅ 可控调整：如果逻辑约束候选过多，可通过 `composite.exclude_semantic_roles` 收紧**逻辑约束**的过滤
- ⚠️ **设计原则**：物理约束关系无法通过配置关闭，这是不可妥协的原则

**关于"逻辑约束收紧配置"的重要说明**：
- ✅ `composite.exclude_semantic_roles` 只影响**逻辑主键生成**和**逻辑约束目标列**的过滤
- ❌ **不会影响物理约束**（PK/UK/索引）的匹配行为
- ❌ 无法通过配置"恢复"到物理约束也被过滤的状态（这是错误行为，不应恢复）

## 八、逻辑约束收紧建议（可选）

> ⚠️ **核心说明**：
> 1. 以下配置**只影响逻辑约束**（逻辑主键生成/逻辑约束目标列），**完全不影响物理约束**
> 2. **无法通过此配置"恢复"到物理约束也被过滤的状态**（那是错误行为，不应恢复）
> 3. 物理约束（PK/UK/索引）始终遵循"完全不过滤"原则，这是不可妥协的设计原则

### 使用场景

如果在实际使用中发现：
- 逻辑主键生成了过多的候选（质量不高）
- 关系候选中包含过多不相关的逻辑约束关系

可以通过收紧逻辑约束的过滤条件来优化：

```yaml
# 逻辑约束收紧配置（只影响逻辑主键生成和逻辑约束目标列过滤）
composite:
  exclude_semantic_roles:
    - metric        # 默认排除（推荐保留）
    - audit         # 可选：排除审计字段（如 created_at, updated_by）
    - description   # 可选：排除描述字段（如 remark, note）
    - attribute     # 可选：排除普通属性字段（谨慎使用，可能误伤）
```

### 效果说明

**影响范围**：
- ✅ **逻辑主键生成**：在元数据生成阶段，复合逻辑主键的候选字段会排除这些语义角色
- ✅ **逻辑约束目标列**：在关系匹配阶段，目标表的逻辑约束列会被过滤

**不影响范围**：
- ❌ **物理约束完全不受影响**：源表和目标表的物理约束（PK/UK/索引）始终完全不过滤
- ❌ **源列完全不受影响**：候选生成阶段的源列（物理约束+逻辑主键）完全不过滤

### 使用建议

- ✅ **默认配置足够**：`["metric"]` 已经能满足大多数场景
- ⚠️ **谨慎添加角色**：只在逻辑主键候选过多、质量明显不高时才考虑添加
- ❌ **不要过度排除**：不建议排除 `identifier`、`enum`、`datetime` 等常见关联字段类型
- 💡 **测试验证**：修改配置后应重新生成元数据和关系，验证效果

## 九、后续扩展

### 9.1 description 角色支持

当 `description` 角色实现后，可以直接在配置中添加：

```yaml
composite:
  exclude_semantic_roles:
    - metric
    - description  # 新增
```

### 9.2 更细粒度的控制（不推荐）

> ⚠️ **设计说明**：当前设计已经足够清晰和简洁，不建议引入过于复杂的配置。

如果**未来**确实有特殊需求，可以考虑更细粒度的配置（但需要充分论证必要性）：

```yaml
composite:
  # ❌ 不推荐：物理约束不应该有排除配置
  # source_physical_exclude_roles: []  # 物理约束应该始终完全不过滤
  
  # ✅ 可选：逻辑约束的细粒度控制
  source_logical_exclude_roles: [metric, description]    # 逻辑主键源表排除
  target_logical_exclude_roles: [metric, description]    # 逻辑约束目标表排除
```

**当前设计哲学**：
- ✅ **物理约束 = 完全不过滤**：这是核心原则，不应该有例外
- ✅ **配置简洁优先**：一个 `exclude_semantic_roles` 足够应对大多数场景
- ⚠️ **避免过度配置**：过多配置项会增加理解成本和出错风险

**建议**：除非有明确的业务需求，否则保持当前的单一配置项设计

### 9.3 物理约束质量检查

可以增加日志或警告，提示 DBA 检查不合理的物理约束：

```python
if has_physical_constraint and has_metric:
    logger.warning(
        "发现物理约束包含 metric 列: %s.%s，建议检查数据库设计",
        table_name, column_name
    )
```

## 十、总结

### 10.1 核心改进

1. ✅ **完全尊重物理约束**：
   - 源表物理约束**完全不过滤**任何语义角色（包括 metric）
   - 目标表物理约束**完全不过滤**任何语义角色（包括 metric）
   - 动态同名匹配**只对物理约束执行**，且**完全不过滤**

2. ✅ **配置统一管理**：
   - `LogicalKeyDetector` 和 `CandidateGenerator` 使用**相同的配置源**
   - 单列和复合键配置方式保持一致
   - 从 YAML 读取，不再硬编码

3. ✅ **逻辑约束可配置**：逻辑主键过滤规则可通过 YAML 配置

4. ✅ **清晰的设计哲学**：
   - 物理约束 = DBA 明确意图 = 完全尊重 = 不过滤
   - 逻辑约束 = 系统推断 = 可质疑 = 可配置过滤

### 10.2 预期收益

- ✅ 更准确识别物理约束定义的关系
- ✅ 更灵活的配置方式
- ✅ 更一致的代码逻辑
- ✅ 更清晰的设计哲学
