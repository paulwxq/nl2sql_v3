# 历史 SQL 检索功能改造方案（简化版）

## 需求与边界

- 在 SQL 生成流程中追加一步：针对原始自然语言问题检索 system.sql_embedding 中的相似历史问答对，并将结果以 question: ...、sql: ... 的纯文本形式拼接到提示词末尾。
- 历史 SQL 仅用于提示 LLM，**不参与候选表筛选、JOIN 规划或验证**；也不要求基于表名、质量分等附加过滤逻辑。
- 目标是在保持现有流程最小侵入的前提下，补充一个低成本的经验参考信号。

---

## 推荐落点：Generator 阶段

| 阶段 | 是否适合 | 原因 |
|------|----------|------|
| Parser | ❌ | 解析阶段聚焦结构化意图，过早注入示例会混淆职责 |
| Retriever | ❌ | 该阶段专注元数据检索，历史 SQL 不参与 Schema 构建 |
| Planner | ❌ | 仅负责 JOIN 规划，与历史案例无直接关联 |
| **Generator** | ✅ | 在生成提示词前补充上下文最自然；上下文已完整，便于直接输出 |

因此，本方案将历史 SQL 检索集成到 Generator 节点（或其前置工具函数）中：在构造提示词前触发一次向量检索，将返回结果拼入提示词上下文。

---

## 数据建模

system.sql_embedding 继续承担“问题-SQL”历史样本的存储，并保持最小字段集合：

`sql
CREATE TABLE system.sql_embedding (
    id SERIAL PRIMARY KEY,
    document JSONB NOT NULL,          -- {"question": "...", "sql": "..."}
    embedding vector(1536) NOT NULL,  -- 问题语义向量
    question_text TEXT NOT NULL,      -- 冗余字段，便于人工查看
    sql_text TEXT NOT NULL,           -- 冗余字段，便于人工查看
    source VARCHAR(32) DEFAULT 'user',
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);
`

> 说明：无需维护 	ables_used、quality_score 等附加字段；若线下已有该结构，可忽略或保留，但检索逻辑不依赖这些信息。

---

## 检索流程

1. **获取查询向量**：复用 Schema 检索阶段生成的向量，若不存在则调用同一 embedding 客户端重新计算。
2. **向量相似度查询**：
   `sql
   SELECT document,
          1 - (embedding <=> ::vector) AS similarity
   FROM system.sql_embedding
   ORDER BY embedding <=> ::vector
   LIMIT ;
   `
   - $1 为查询向量；$2 为返回条数（默认 2~3 条，可通过配置控制）。
   - 可选地设置最小相似度阈值以避免噪声，但不是必选项。
3. **解析结果**：
   - 将 document 解析成 { "question": str, "sql": str }。
   - 若解析失败或字段缺失，降级为将原始文本作为 sql 字段返回，question 置空。
4. **返回格式**：
   `json
   [
     {"question": "...", "sql": "...", "similarity": 0.87},
     ...
   ]
   `

---

## 与 Generator 节点的集成

1. **触发时机**：在构造提示词之前调用检索函数，当且仅当当前状态中尚未包含历史 SQL 结果时触发，避免重复查询。
2. **PGClient 接口**：
   `python
   def search_similar_sqls(
       self,
       embedding: List[float],
       top_k: int = 2,
       similarity_threshold: float = 0.0,
   ) -> List[Dict[str, Any]]:
       """返回 question/sql/similarity 三个字段"""
   `
3. **提示词拼接**：在 _format_similar_sqls（或等效函数）中按以下格式输出：
   `
   【示例 1】相似度: 0.87
   问题: ……
   SQL: ……
   `
   - 若 question 为空，可省略该行。
   - 若无命中结果，输出占位文本“（无相似案例）”。
4. **配置项**：在 sql_generation_subgraph.yaml 中暴露以下可调参数：
   `yaml
   sql_generation:
     prompt:
       include_history_examples: true
       max_similar_sqls: 2
       min_similarity: 0.6  # 可选，默认 0 表示不过滤
   `

## 测试与运维建议

- **单元测试**：
  - Mock PGClient 返回的 JSON，验证解析失败时的降级行为。
  - 断言生成的提示词片段包含示例编号与 SQL 文本。
- **集成测试**：
  - 在 SQL 生成子图的 e2e 测试中，确保当 include_history_examples=true 时提示词新增历史示例段落。
- **日志与监控**：
  - 记录检索耗时与返回条数，便于定位向量索引问题。
  - 可选地统计命中率，用于评估示例库质量。

---

## 总结

- 历史 SQL 检索仅作为提示词补充，不干扰 Schema/Planner 流程。
- 复用现有 embedding 能力与 PGClient，无需扩展数据库字段或复杂过滤。
- 通过配置开关可灵活启用/禁用示例段落，满足不同环境下的成本与隐私需求。
