# 33.0 NL2SQL 父图第一阶段（Fast Path）概要设计

## 文档信息
- **版本**: v1.6
- **创建日期**: 2025-11-13
- **目标**: 完成父图 Fast Path 的最小可用实现（MVP）
- **技术栈**: LangGraph v1.0.x, Python 3.10+
- **开发阶段**: Phase 1 - Fast Path Only

---

## 1. 设计目标与范围

### 1.1 第一阶段目标
实现父图左侧的 **Fast Path** 流程，为简单问题提供快速响应：

```
用户问题 → Router → SQL生成子图 → SQL执行 → Summarizer → 返回结果
```

### 1.2 设计原则
1. **复用优先**: SQL执行节点、Summarizer节点设计为两条路径（Fast Path / Complex Path）**共享**，真正做到统一响应构建
2. **清晰边界**: 通过 LangGraph 1.0.x 的 input/output schema 实现父子图清晰对接，避免状态污染
3. **面向扩展**: State 设计预留复杂路径所需字段（如多SQL、并发执行、依赖管理等）
4. **最小化开销**: Fast Path 仅包含必要节点，减少延迟和成本；失败时智能跳过不必要节点
5. **统一出口**: 所有路径（Simple成功/失败、Complex暂不支持）统一经过 Summarizer，确保返回格式一致
6. **可观测性**: 保留关键诊断信息，便于调试和监控

### 1.3 第一阶段测试约束
- 仅测试会被 Router 判定为 `simple` 的问题
- 验证端到端流程：问题输入 → SQL生成 → 执行 → 总结返回
- 暂不实现右侧复杂路径（Planner、Inject Params、Check完成度等）

## 2. 父图 State 设计

### 2.1 设计思路
1. **支持问题拆分**: `user_query` 保存原始问题，`query_id` 为会话级ID，`sub_queries` 支持复杂问题拆分为多个子查询
2. **与子图对齐**: 子查询的 `query` 字段与 SQL生成子图的 `query` 字段同名，利用 LangGraph 1.0.x 的自动映射特性
3. **精简原则**: 仅保留必要字段，子图内部细节（如 schema_context, validation_history）不传递到父图
4. **使用 Reducer**: 对列表类字段使用 `Annotated[List[...], add]` 实现增量累加
5. **不继承 MessagesState**: 父图不需要管理 LLM 消息历史，减少无关状态

### 2.2 State 定义

```python
from typing import Annotated, Any, Dict, List, Literal, Optional, TypedDict
from operator import add


class SubQueryInfo(TypedDict):
    """子查询信息（支持复杂问题拆分）"""
    sub_query_id: str                 # 子查询ID（如 "q1", "q2"）
    query: str                        # 子查询文本
    status: str                       # pending/in_progress/completed/failed
    dependencies: List[str]           # 依赖的 sub_query_id 列表
    validated_sql: Optional[str]      # 生成的SQL
    execution_result: Optional[Dict[str, Any]]  # 执行结果
    error: Optional[str]              # 错误信息
    iteration_count: int              # 迭代次数


class SQLExecutionResult(TypedDict):
    """单条SQL的执行结果"""
    sub_query_id: str                 # 所属子查询ID（用于精确定位，避免相同SQL互相覆盖）
    sql: str                          # 执行的SQL语句
    success: bool                     # 是否执行成功
    columns: Optional[List[str]]      # 列名列表
    rows: Optional[List[List[Any]]]   # 结果行（列表的列表）
    row_count: int                    # 结果行数
    execution_time_ms: float          # 执行耗时（毫秒）
    error: Optional[str]              # 错误信息（若失败）


class NL2SQLFatherState(TypedDict):
    """
    NL2SQL 父图 State（第一阶段 + 预留扩展）

    设计要点：
    1. user_query 保存原始问题，query_id 为会话级ID
    2. sub_queries 支持复杂问题拆分（Fast Path 只有1个子查询）
    3. 精简原则：不保留子图内部细节（schema_context, validation_history）
    4. 列表字段使用 reducer（add）支持增量累加
    """

    # ========== 输入与会话标识 ==========
    user_query: str                                         # 用户原始问题（始终不变）
    query_id: str                                           # 会话级查询ID（原始问题的ID）

    # ========== 子查询管理（支持拆分） ==========
    # Fast Path: sub_queries 只有1个元素（原始问题 = 唯一子查询）
    # Complex Path: sub_queries 包含多个子查询，按依赖顺序执行
    sub_queries: Annotated[List[SubQueryInfo], add]         # 子查询列表
    current_sub_query_id: Optional[str]                     # 当前处理的子查询ID

    # ========== Router 输出 ==========
    complexity: Optional[Literal["simple", "complex"]]      # 问题复杂度（字符串枚举：'simple' 或 'complex'，非布尔值）
    router_reason: Optional[str]                            # 路由理由（可选，用于调试）
    router_latency_ms: Optional[float]                      # Router 耗时

    # ========== 当前子查询的SQL生成输出 ==========
    # 注意：这些字段对应当前正在处理的子查询（由 current_sub_query_id 指定）
    validated_sql: Optional[str]                            # 子图生成的验证通过的SQL
    error: Optional[str]                                    # 子图错误信息
    error_type: Optional[str]                               # 错误类型
    iteration_count: Optional[int]                          # 子图迭代次数

    # ========== SQL执行结果 ==========
    execution_results: Annotated[List[SQLExecutionResult], add]  # 所有SQL的执行结果列表

    # ========== 总结与返回 ==========
    summary: Optional[str]                                  # 自然语言总结

    # ========== 可观测性与元数据 ==========
    total_execution_time_ms: Optional[float]                # 总耗时
    path_taken: Optional[Literal["fast", "complex"]]        # 实际走的路径（字符串枚举：'fast' 或 'complex'，非布尔值）
    metadata: Optional[Dict[str, Any]]                      # 其他元数据


# ========== 便捷函数 ==========

def create_initial_state(user_query: str, query_id: Optional[str] = None) -> NL2SQLFatherState:
    """创建初始State"""
    import uuid
    generated_query_id = query_id or uuid.uuid4().hex[:8]

    return {
        "user_query": user_query,
        "query_id": generated_query_id,
        "sub_queries": [],
        "execution_results": [],
    }


def extract_final_result(state: NL2SQLFatherState) -> Dict[str, Any]:
    """
    提取最终返回结果

    Returns:
        返回结果字典，包含：
        - complexity: "simple" | "complex" | None（字符串，非布尔值）
        - path_taken: "fast" | "complex" | None（字符串，非布尔值）
        - sql: 验证通过的 SQL（从第一个 sub_query 提取，快捷访问）
        - sub_queries: 完整子查询列表
        - execution_results: 完整执行结果列表
        - 其他执行结果字段
    """
    # 提取 SQL（快捷访问：从第一个 sub_query 提取 validated_sql）
    sql = None
    sub_queries = state.get("sub_queries", [])
    if sub_queries and len(sub_queries) > 0:
        first_sq = sub_queries[0]
        sql = first_sq.get("validated_sql")

    return {
        "user_query": state["user_query"],
        "query_id": state["query_id"],
        "complexity": state.get("complexity"),        # "simple" | "complex" | None
        "path_taken": state.get("path_taken"),        # "fast" | "complex" | None
        "summary": state.get("summary"),
        "error": state.get("error"),
        "sql": sql,                                   # 快捷访问：第一个 sub_query 的 SQL
        "sub_queries": sub_queries,                   # 完整子查询列表
        "execution_results": state.get("execution_results", []),  # 完整执行结果
        "metadata": {
            "total_execution_time_ms": state.get("total_execution_time_ms"),
            "router_latency_ms": state.get("router_latency_ms"),
        }
    }
```

### 2.4 关键字段详解

#### `SQLExecutionResult` - SQL执行结果类型

**完整字段定义**:
```python
class SQLExecutionResult(TypedDict):
    sub_query_id: str                 # 所属子查询ID（用于精确定位）
    sql: str                          # 执行的SQL语句
    success: bool                     # 是否执行成功
    columns: Optional[List[str]]      # 列名列表
    rows: Optional[List[List[Any]]]   # 结果行（列表的列表）
    row_count: int                    # 结果行数
    execution_time_ms: float          # 执行耗时（毫秒）
    error: Optional[str]              # 错误信息（若失败）
```

**关键设计点**:
- ✅ **sub_query_id**: 用于精确定位所属子查询，避免相同SQL互相覆盖
- ✅ **双向绑定**: 执行结果同时写入 `execution_results` 列表和对应 `sub_query.execution_result` 字段
- ✅ **错误处理**: `success=False` 时，`error` 字段包含错误信息，`rows/columns` 为 None

**示例**:
```python
{
    "sub_query_id": "q001_sq1",
    "sql": "SELECT COUNT(*) FROM sales",
    "success": True,
    "columns": ["count"],
    "rows": [[1234]],
    "row_count": 1,
    "execution_time_ms": 45.2,
    "error": None,
}
```

---

#### `complexity` - 问题复杂度

**类型**: `Optional[Literal["simple", "complex"]]`

**⚠️ 注意**: 这是**字符串枚举类型**，不是布尔值！

| 值 | 含义 | 路由目标 |
|----|------|---------|
| `"simple"` | 简单问题，可用一条SQL完成 | Fast Path |
| `"complex"` | 复杂问题，需要多步骤或多SQL | Complex Path |
| `None` | 未判定（初始状态） | - |

**示例**:
```python
# ✅ 正确
if state["complexity"] == "simple":
    print("走Fast Path")

# ❌ 错误（不要当布尔值用）
if state["complexity"]:  # 这会把 "complex" 也当作 True
    print("这样判断是错误的")
```

#### `path_taken` - 实际路径

**类型**: `Optional[Literal["fast", "complex"]]`

**⚠️ 注意**: 这也是**字符串枚举类型**，不是布尔值！

| 值 | 含义 |
|----|------|
| `"fast"` | 实际走了Fast Path |
| `"complex"` | 实际走了Complex Path |
| `None` | 未执行（初始状态） |

**与 complexity 的关系**:
- 通常 `complexity == "simple"` → `path_taken == "fast"`
- 通常 `complexity == "complex"` → `path_taken == "complex"`
- 但在异常情况下可能不一致（如Router失败后的fallback）

**示例输出**:
```json
{
  "user_query": "广州服务区的销售额是多少？",
  "complexity": "simple",
  "path_taken": "fast",
  "summary": "广州服务区的销售额为100万元。"
}
```

## 3. Router 节点设计

### 3.1 职责
- **唯一职责**: 快速判定问题复杂度（simple / complex）
- **不做**:
  - 问题重写、问题拆解（由 Planner 负责）
  - 参数准备、子查询创建（Simple: 由 Simple Planner 负责；Complex: 由 Complex Planner 负责）

### 3.2 判定标准

**核心原则**：能用一条SQL解决的问题，就不拆分成多条SQL

| 分类 | 标准 | 允许的SQL特性 | 示例 |
|------|------|--------------|------|
| **simple** | 单个独立问题，**一条SQL查询**可完成 | • 多表JOIN（INNER/LEFT/RIGHT JOIN）<br>• WITH子查询（CTE）<br>• 子查询、UNION<br>• 聚合函数、窗口函数 | "广州服务区的销售额是多少？"<br>"2024年总销售额是多少？"<br>"北京和上海服务区的销售额对比"<br>"销售额最高的服务区名称和所属公司"（可用JOIN） |
| **complex** | 必须拆分为**多条SQL**才能完成 | • 需要根据第一次结果决定后续查询<br>• 需要临时表存储中间结果<br>• 多个独立问题 | "哪个服务区销售最高？它的详细信息是？"（需要先查ID，再查详情）<br>"找出车流量前3的服务区，分别查询它们的营业额"（需要先查前3，再查各自营业额） |

**判定要点**：
1. ✅ **如果问题可以通过多表JOIN解决** → 判定为 `simple`
2. ✅ **如果问题可以用WITH子查询（CTE）解决** → 判定为 `simple`
3. ✅ **如果问题可以用子查询、UNION解决** → 判定为 `simple`
4. ❌ **只有当必须根据第一次查询结果动态决定第二次查询时** → 判定为 `complex`

### 3.3 实现方案

#### 方案选择
- **第一阶段（MVP）**: 使用轻量级 LLM（如 qwen-turbo, gpt-4o-mini）快速分类
- **后续优化**: 可考虑基于规则的预筛选（如问号数量、关键词等）+ LLM 兜底

#### 节点实现

```python
import time
from typing import Dict, Any
from langchain_community.chat_models import ChatTongyi  # 示例，可替换为其他LLM

# 加载配置（在模块级别加载一次）
from src.services.config_loader import load_config
import os

_config_cache = None

def _get_router_config():
    """获取Router配置（带缓存）"""
    global _config_cache
    if _config_cache is None:
        project_root = os.path.dirname(os.path.dirname(os.path.dirname(os.path.dirname(os.path.abspath(__file__)))))
        config_path = os.path.join(project_root, "src/modules/nl2sql_father/config/nl2sql_father_graph.yaml")
        _config_cache = load_config(config_path)
    return _config_cache["router"]

def router_node(state: NL2SQLFatherState) -> Dict[str, Any]:
    """
    Router 节点：判定问题复杂度

    职责：
    - 唯一职责：调用 LLM 判定问题是 simple 还是 complex
    - 不做任何数据准备工作（交给对应的 Planner）

    Args:
        state: 父图State

    Returns:
        更新的State字段
    """
    user_query = state["user_query"]

    # 加载配置
    config = _get_router_config()
    model_name = config["model"]
    temperature = config["temperature"]
    timeout = config["timeout"]
    default_on_error = config["default_on_error"]

    # 构造提示词
    prompt = f"""你是一个SQL查询复杂度分类器。判断用户问题需要执行几次SQL查询。

分类标准：
- simple: 问题可以通过**一条SQL查询**完成
  * 允许使用多表JOIN（如 INNER JOIN、LEFT JOIN等）
  * 允许使用WITH子查询（CTE，Common Table Expression）
  * 允许使用子查询、UNION、聚合函数等
  * 原则：能用一条SQL解决的，就不拆分成多条SQL

- complex: 问题必须拆分为多次SQL查询才能完成
  * 需要根据第一次查询结果决定后续查询内容
  * 包含多个独立的问题（如："A是多少？B是多少？"）
  * 需要使用临时表存储中间结果

用户问题：{user_query}

请只输出一个词：simple 或 complex"""

    # 调用LLM
    start_time = time.time()
    llm = ChatTongyi(
        model=model_name,              # 从配置读取
        temperature=temperature,       # 从配置读取
        timeout=timeout,               # 从配置读取
    )

    try:
        response = llm.invoke(prompt)
        content = response.content.strip().lower()

        # 归一化输出（注意：返回字符串 "simple" 或 "complex"，非布尔值）
        if "simple" in content:
            complexity = "simple"      # ← 字符串
        elif "complex" in content:
            complexity = "complex"     # ← 字符串
        else:
            # 异常情况：无法识别，使用配置的默认值
            complexity = default_on_error  # ← 从配置读取（默认 "complex"）

        latency_ms = (time.time() - start_time) * 1000

        return {
            "complexity": complexity,                                      # "simple" | "complex"（字符串）
            "router_reason": content[:200],                                # 保留前200字符（用于调试）
            "router_latency_ms": latency_ms,
            "path_taken": "fast" if complexity == "simple" else "complex", # "fast" | "complex"（字符串）
        }

    except Exception as e:
        # 失败时使用配置的默认值
        latency_ms = (time.time() - start_time) * 1000
        return {
            "complexity": default_on_error,  # 从配置读取（默认 "complex"）
            "router_reason": f"Router failed: {str(e)}",
            "router_latency_ms": latency_ms,
            "path_taken": default_on_error,
        }


def route_by_complexity(state: NL2SQLFatherState) -> str:
    """
    条件边：根据complexity路由

    注意：complexity 是字符串 "simple" 或 "complex"，不是布尔值

    Returns:
        "simple_planner": 走 Fast Path，进入 Simple Planner 准备参数（当 complexity == "simple"）
        "summarizer": 第一阶段暂不支持复杂问题，直接进入 Summarizer（当 complexity == "complex"）
    """
    complexity = state.get("complexity")

    # 字符串比较（不是布尔判断）
    if complexity == "simple":    # ← 比较字符串 "simple"
        return "simple_planner"   # ← 进入 Simple Planner
    else:
        # 第一阶段：复杂问题暂不支持，直接进入统一的 Summarizer 返回友好提示
        return "summarizer"
```

## 4. Simple Planner 节点设计

### 4.1 职责定位

Simple Planner 是 **Fast Path 的参数准备节点**，负责为 Simple 问题准备调用 SQL生成子图所需的参数。

**核心职责**：
1. **Phase 1（当前）**：创建单个子查询，包装参数
2. **Phase 2（未来扩展）**：
   - 历史对话记录裁剪
   - State 数据瘦身
   - 缓存检查
   - 问题标准化

**设计特点**：
- ✅ **纯函数**：不调用 LLM，不访问数据库
- ✅ **轻量级**：执行时间 < 1ms
- ✅ **对称设计**：与 Complex Planner 对应，保持架构一致性
- ✅ **易扩展**：为未来优化预留空间

---

### 4.2 节点实现

**Phase 1 实现**（基础功能）：

```python
def simple_planner_node(state: NL2SQLFatherState) -> Dict[str, Any]:
    """
    Simple Planner 节点：为 Simple 问题准备参数

    职责：
    - 创建单个子查询（query = user_query，不拆分）
    - 设置 current_sub_query_id
    - Phase 2 扩展：对话裁剪、缓存检查、State瘦身等

    特点：
    - 纯函数，不调用 LLM
    - 轻量级，延迟 < 1ms

    Args:
        state: 父图State

    Returns:
        更新的State字段
    """
    query_id = state["query_id"]
    user_query = state["user_query"]

    # ========== Phase 1: 创建单个子查询 ==========
    sub_query_id = f"{query_id}_sq1"
    sub_queries = [{
        "sub_query_id": sub_query_id,
        "query": user_query,          # 子查询 = 原始问题（Simple 问题不拆分）
        "status": "pending",
        "dependencies": [],           # Simple 问题无依赖
        "validated_sql": None,
        "execution_result": None,
        "error": None,
        "iteration_count": 0,
    }]

    return {
        "sub_queries": sub_queries,
        "current_sub_query_id": sub_query_id,
    }
```

## 5. SQL生成子图调用

### 5.1 子图接口
SQL生成子图已完成开发，接口定义参考 `docs/30.2.SQL 生成子图 State 定义.md`。

**子图输入**:
- `query`: str（子查询文本）
- `query_id`: str（查询ID）
- `user_query`: str（用户原始问题）
- `dependencies_results`: Dict[str, Any]（依赖结果，Fast Path 为空字典）
- `parse_hints`: Optional[Dict]（解析提示，可选）

**子图输出**:
- `validated_sql`: Optional[str]（成功标志）
- `error`: Optional[str]
- `error_type`: Optional[str]
- `iteration_count`: int
- `schema_context`: Dict
- `validation_history`: List[Dict]
- `execution_time`: float

### 4.2 与父图对接（Wrapper 方式）

#### 为什么选择 Wrapper？

虽然 LangGraph 1.0.x 支持 input/output schema 自动映射，但在我们的场景中需要使用 Wrapper，原因如下：

1. **数据结构不匹配**：
   - 父图使用 `sub_queries` 列表管理子查询（支持复杂路径的多子查询）
   - 子图需要单个 `query` 字符串作为输入
   - 需要从列表中提取当前子查询

2. **避免字段冗余**：
   - 如果用 input/output schema，父图State需要添加：`query`, `dependencies_results`, `parse_hints`
   - 这会导致 `query` 与 `sub_queries[].query` 重复
   - 违背"最小State"原则，造成状态污染

3. **数据转换逻辑**：
   - 需要根据 `current_sub_query_id` 从 `sub_queries` 中查找对应子查询
   - 调用子图后，需要更新子查询的状态（completed/failed）
   - 这些逻辑天然适合封装在 Wrapper 中

#### Wrapper 实现

```python
def sql_gen_wrapper(state: NL2SQLFatherState) -> Dict[str, Any]:
    """
    SQL生成子图的 Wrapper（Fast Path）

    职责：
    1. 从 sub_queries 中获取当前子查询
    2. 调用 SQL 生成子图
    3. 将子图输出映射回父图State
    4. 更新 sub_queries 中对应子查询的状态
    """
    from src.modules.sql_generation.subgraph.create_subgraph import run_sql_generation_subgraph
    from src.utils.logger import get_module_logger

    logger = get_module_logger("sql_gen_wrapper")

    current_sub_query_id = state.get("current_sub_query_id")
    if not current_sub_query_id:
        return {"error": "No current sub_query_id", "error_type": "internal_error"}

    # 从 sub_queries 中找到当前子查询
    sub_queries = state.get("sub_queries", [])
    current_sub_query = None
    for sq in sub_queries:
        if sq["sub_query_id"] == current_sub_query_id:
            current_sub_query = sq
            break

    if not current_sub_query:
        return {"error": f"Sub query {current_sub_query_id} not found", "error_type": "internal_error"}

    # 调用子图（使用便捷函数）+ 异常兜底
    try:
        subgraph_output = run_sql_generation_subgraph(
            query=current_sub_query["query"],  # 子查询文本
            query_id=state["query_id"],        # 会话级查询ID（用于日志关联与链路追踪）
            user_query=state["user_query"],    # 原始用户问题
            dependencies_results={},           # Fast Path 无依赖
            parse_hints=None,
        )

        # 更新当前子查询的状态
        if subgraph_output.get("validated_sql"):
            current_sub_query["status"] = "completed"
            current_sub_query["validated_sql"] = subgraph_output["validated_sql"]
            current_sub_query["iteration_count"] = subgraph_output.get("iteration_count", 0)
        else:
            current_sub_query["status"] = "failed"
            current_sub_query["error"] = subgraph_output.get("error")

        # 映射输出到父图State
        return {
            "validated_sql": subgraph_output.get("validated_sql"),
            "error": subgraph_output.get("error"),
            "error_type": subgraph_output.get("error_type"),
            "iteration_count": subgraph_output.get("iteration_count"),
        }

    except Exception as e:
        # 兜底：子图崩溃时记录错误，确保父图不会直接崩溃
        error_msg = f"SQL生成子图执行异常: {str(e)}"
        logger.error(error_msg, exc_info=True)

        # 更新子查询状态
        current_sub_query["status"] = "failed"
        current_sub_query["error"] = error_msg

        # 返回错误信息，流程将进入 Summarizer 输出友好提示
        return {
            "validated_sql": None,
            "error": error_msg,
            "error_type": "generation_failed",
            "iteration_count": 0,
        }

# 添加到父图
father_graph.add_node("sql_gen", sql_gen_wrapper)
```

**关键设计点**:
1. **数据提取**: 从 `sub_queries` 列表中提取当前子查询的 `query` 字段
2. **状态更新**: 根据子图执行结果更新子查询的 `status`（completed/failed）
3. **字段映射**: 将子图输出映射到父图State，无需修改父图State定义
4. **职责清晰**: Wrapper 专注于父子State的数据转换，保持父图State最小化

### 5.3 query_id 参数设计说明（重要）

#### Phase 1：传入会话级 query_id

**设计决策**:
- Wrapper 向子图传入的 `query_id` 参数使用**会话级 query_id**（`state["query_id"]`），而非子查询ID（`current_sub_query_id`）

**理由**:

1. **日志关联与链路追踪**
   - 子图所有节点使用 `with_query_id(logger, state["query_id"])` 在日志前加 `[query_id]` 前缀
   - 使用会话级ID可按会话过滤日志：`grep "test001" logs/`
   - 避免链路追踪断裂（不需要知道所有 sub_query_id 才能追踪完整会话）

2. **语义正确性**
   - 子图 State 定义：`query_id: str  # 查询ID`
   - 传入会话级ID符合字段语义（而非子查询级ID）

3. **Phase 1 单SQL场景**
   - Phase 1 只有单个子查询，不存在多子查询并发
   - 不需要子查询级别的日志区分

**日志示例**:
```
[test001] Schema retrieval started...
[test001] SQL generation completed in 1.2s
[test001] Validation passed
```

## 6. SQL执行节点设计

### 6.1 设计目标
- **两路复用**: Fast Path 和 Complex Path 共享同一个节点
- **单/多SQL统一**: 从 `sub_queries` 中获取待执行的SQL
- **支持并发**: 预留并发参数（第一阶段串行执行）
- **规范化输出**: 统一的 `SQLExecutionResult` 结构

### 6.2 节点实现

```python
import time
from typing import Dict, Any, List
from src.services.db.pg_client import PGClient
from src.utils.logger import get_module_logger, with_query_id
from src.services.config_loader import load_config
import os

logger = get_module_logger("sql_execution")

# 加载配置（在模块级别加载一次）
_sql_exec_config_cache = None

def _get_sql_execution_config():
    """获取SQL执行配置（带缓存）"""
    global _sql_exec_config_cache
    if _sql_exec_config_cache is None:
        project_root = os.path.dirname(os.path.dirname(os.path.dirname(os.path.dirname(os.path.abspath(__file__)))))
        config_path = os.path.join(project_root, "src/modules/nl2sql_father/config/nl2sql_father_graph.yaml")
        full_config = load_config(config_path)
        _sql_exec_config_cache = full_config["sql_execution"]
    return _sql_exec_config_cache


def sql_execution_node(state: NL2SQLFatherState, config: Dict[str, Any] = None) -> Dict[str, Any]:
    """
    SQL执行节点（两路复用）

    职责：
    1. 从 sub_queries 中收集所有状态为 completed 且有 validated_sql 的子查询
    2. 串行/并行执行SQL（第一阶段串行）
    3. 规范化输出为 SQLExecutionResult 列表
    4. 更新 sub_queries 中的 execution_result

    Args:
        state: 父图State
        config: 运行时配置（可选）
            - max_concurrency: 最大并发数（默认1）
            - timeout_per_sql: 单条SQL超时（默认30秒）
            - enable_explain: 是否执行EXPLAIN（默认False）

    Returns:
        更新的State字段
    """
    query_logger = with_query_id(logger, state.get("query_id", "unknown"))

    # 加载配置
    exec_config = _get_sql_execution_config()
    timeout_per_sql = exec_config["timeout_per_sql"]
    max_rows = exec_config["max_rows_per_query"]  # Phase 2 预留：当前版本暂未实现结果截断
    log_sql = exec_config["log_sql"]

    # 1. 从 sub_queries 中收集待执行SQL
    sub_queries = state.get("sub_queries", [])
    sqls_to_execute = []
    sub_query_map = {}  # 用于映射：sub_query_id -> sub_query

    for sq in sub_queries:
        if sq.get("status") == "completed" and sq.get("validated_sql"):
            sql_item = {
                "sub_query_id": sq["sub_query_id"],
                "sql": sq["validated_sql"]
            }
            sqls_to_execute.append(sql_item)
            sub_query_map[sq["sub_query_id"]] = sq

    if not sqls_to_execute:
        query_logger.warning("没有待执行的SQL")
        return {"execution_results": []}

    if log_sql:
        query_logger.info(f"准备执行 {len(sqls_to_execute)} 条SQL")

    # 2. 执行SQL（第一阶段：串行）

    results: List[SQLExecutionResult] = []
    pg_client = PGClient()

    for idx, sql_item in enumerate(sqls_to_execute, 1):
        sub_query_id = sql_item["sub_query_id"]
        sql = sql_item["sql"]

        if log_sql:
            query_logger.info(f"执行第 {idx}/{len(sqls_to_execute)} 条SQL (sub_query_id: {sub_query_id})")
        start_time = time.time()

        try:
            # 执行SQL
            result = pg_client.execute_query(sql, timeout=timeout_per_sql)
            execution_time_ms = (time.time() - start_time) * 1000

            exec_result = {
                "sub_query_id": sub_query_id,  # 携带 sub_query_id
                "sql": sql,
                "success": True,
                "columns": result.get("columns"),
                "rows": result.get("rows"),
                "row_count": len(result.get("rows", [])),
                "execution_time_ms": execution_time_ms,
                "error": None,
            }
            results.append(exec_result)

            # 更新 sub_query 的 execution_result（使用 sub_query_id 精确定位）
            if sub_query_id in sub_query_map:
                sub_query_map[sub_query_id]["execution_result"] = exec_result

            if log_sql:
                query_logger.info(f"SQL执行成功，返回 {len(result.get('rows', []))} 行，耗时 {execution_time_ms:.0f}ms")

        except Exception as e:
            execution_time_ms = (time.time() - start_time) * 1000
            error_msg = str(e)

            exec_result = {
                "sub_query_id": sub_query_id,  # 携带 sub_query_id
                "sql": sql,
                "success": False,
                "columns": None,
                "rows": None,
                "row_count": 0,
                "execution_time_ms": execution_time_ms,
                "error": error_msg,
            }
            results.append(exec_result)

            # 更新 sub_query 的 execution_result（使用 sub_query_id 精确定位）
            if sub_query_id in sub_query_map:
                sub_query_map[sub_query_id]["execution_result"] = exec_result

            query_logger.error(f"SQL执行失败: {error_msg}")

    return {"execution_results": results}
```

### 6.3 PGClient 扩展

需要在 `src/services/db/pg_client.py` 中添加通用的 `execute_query` 方法：

```python
def execute_query(
    self,
    sql: str,
    timeout: int = 30,
) -> Dict[str, Any]:
    """
    执行SQL查询（通用方法）

    Args:
        sql: SQL语句
        timeout: 超时时间（秒）

    Returns:
        {
            "columns": List[str],  # 列名
            "rows": List[List[Any]],  # 数据行
        }

    Raises:
        Exception: 执行失败时抛出异常
    """
    with self.pg_manager.get_connection() as conn:
        with conn.cursor() as cur:
            # 设置语句超时
            cur.execute(f"SET statement_timeout = {timeout * 1000}")

            # 执行SQL
            cur.execute(sql)

            # 获取列名
            columns = [desc[0] for desc in cur.description] if cur.description else []

            # 获取数据
            rows = cur.fetchall()
            rows = [list(row) for row in rows]  # 转为列表

            return {
                "columns": columns,
                "rows": rows,
            }
```

## 7. Summarizer 节点设计

### 7.1 设计目标与职责定位

#### 职责定位：响应构建器（自适应，两路共享）

Summarizer 不仅是"结果总结器"，更是"**响应构建器**"，负责将内部状态转换为用户友好的最终响应。

**核心设计理念**：Simple 和 Complex 路径共享同一个 Summarizer 节点，因为两者的核心逻辑完全一致：
- **输入**：用户问题 + SQL执行结果（1条或多条）
- **处理**：LLM理解问题意图，结合SQL结果生成自然语言回答
- **输出**：summary（自然语言总结）

**差异仅在于**：
- **Simple路径**（Phase 1）：1条SQL结果 → 直接总结
- **Complex路径**（Phase 2）：多条SQL结果 → 综合多个子查询结果进行总结

**处理场景**：
1. **SQL生成成功 + 执行成功**：将查询结果（单个或多个）转换为自然语言总结
2. **SQL生成失败**：将技术错误（`parsing_failed`, `schema_retrieval_failed`, `generation_failed`, `validation_failed`）转换为用户友好提示
3. **SQL执行失败**：给出明确的执行错误说明

#### 设计目标
- **统一出口**: 所有请求（成功/失败）都经过Summarizer，确保返回格式一致
- **两路复用**: Fast Path 和 Complex Path 共享同一个节点
- **自适应**: 根据执行结果数量（1条/多条）自动调整总结方式
- **友好提示**: 将技术错误转换为用户可理解的自然语言
- **可扩展**: Phase 1实现单SQL总结，Phase 2扩展为多SQL综合总结

### 7.2 节点实现

**⚠️ Phase 1 开发提示**：

本节代码包含两部分：
1. **Phase 1 必需**：`summarizer_node()`、`_build_error_summary()`、`_format_table()`
2. **Phase 2 预留**：`_format_multi_results()`（Phase 1 不会被调用，可跳过实现）

```python
    user_query = state["user_query"]
    complexity = state.get("complexity")
    execution_results = state.get("execution_results", [])
    error = state.get("error")
    error_type = state.get("error_type")

    # 加载配置
    config = _get_summarizer_config()
    model_name = config["model"]
    temperature = config["temperature"]
    timeout = config["timeout"]
    max_rows_in_prompt = config["max_rows_in_prompt"]
    use_template = config["use_template"]

    # ========== 场景0：第一阶段暂不支持的 complex 问题 ==========
    # 判断标准：complexity 为 complex 且没有执行结果（说明从 Router 直接跳转过来）
    if complexity == "complex" and not execution_results:
        return {
            "summary": "抱歉，当前版本暂不支持复杂问题。我们正在开发中，敬请期待！",
        }

    # ========== 场景1：SQL生成失败（从子图直接跳转过来，跳过了SQL执行） ==========
    if not execution_results and error:
        summary = _build_error_summary(error, error_type, user_query)
        return {"summary": summary}

    # ========== 场景2：SQL执行失败或无结果 ==========
    if not execution_results:
        return {
            "summary": "抱歉，系统未能生成有效的SQL查询。",
        }

    # 检查执行是否成功
    success_results = [r for r in execution_results if r.get("success")]
    failed_results = [r for r in execution_results if not r.get("success")]

    if not success_results:
        # 所有SQL都执行失败
        error_summary = "\n".join([r.get("error", "未知错误") for r in failed_results])
        return {
            "summary": f"抱歉，SQL执行失败。错误信息：\n{error_summary}",
        }

    # ========== 场景3：SQL执行成功，生成自然语言总结 ==========
    # 自适应处理：根据结果数量自动选择处理方式
    # - Phase 1 (Fast Path): 只有1条SQL结果，直接总结
    # - Phase 2 (Complex Path): 可能有多条SQL结果，需要综合总结

    if len(success_results) == 1:
        # 单SQL结果处理（Phase 1）
        result = success_results[0]
        rows = result.get("rows", [])
        columns = result.get("columns", [])

        # 格式化结果为表格形式（使用配置的最大行数）
        result_table = _format_table(columns, rows[:max_rows_in_prompt])
        context_info = f"查询结果（共{len(rows)}行）：\n{result_table}"
    else:
        # ⚠️ 注意：这个分支在 Phase 1 不会被执行
        # 原因：Phase 1 只处理 Simple 问题，最多只有 1 条 SQL
        # Phase 2 扩展：Complex 问题可能有多条 SQL 结果
        #
        # 多SQL结果处理（Phase 2 扩展，Phase 1 暂不实现）
        context_info = _format_multi_results(success_results, state.get("sub_queries", []), max_rows_in_prompt)

    # 生成总结
    if use_template:
        # 使用模板（不调用LLM）
        result_count = sum(len(r.get("rows", [])) for r in success_results)
        if result_count == 0:
            summary = "查询成功，但未找到相关数据。"
        elif result_count == 1:
            summary = f"查询成功，找到1条记录。"
        else:
            summary = f"查询成功，共找到{result_count}条记录。"
    else:
        # 调用LLM生成自然语言总结（自适应提示词）
        if len(success_results) == 1:
            # 单SQL结果提示词
            prompt = f"""你是一个数据分析助手。用户提出了一个问题，系统执行了SQL查询并获得了结果。请用自然语言总结结果。

用户问题：{user_query}

{context_info}

请用1-2句话总结结果，直接回答用户的问题。如果结果为空，请说明"未找到相关数据"。"""
        else:
            # 多SQL结果提示词（Phase 2）
            prompt = f"""你是一个数据分析助手。用户提出了一个问题，系统分步执行了多个SQL查询并获得了结果。请综合所有结果，用自然语言回答用户的问题。

用户问题：{user_query}

{context_info}

请综合以上所有查询结果，用2-3句话回答用户的问题。"""

        llm = ChatTongyi(
            model=model_name,          # 从配置读取
            temperature=temperature,   # 从配置读取
            timeout=timeout,           # 从配置读取
        )

        try:
            response = llm.invoke(prompt)
            summary = response.content.strip()
        except Exception as e:
            # LLM失败时，使用简单模板
            result_count = sum(len(r.get("rows", [])) for r in success_results)
            if result_count == 0:
                summary = "查询成功，但未找到相关数据。"
            elif result_count == 1:
                summary = f"查询成功，找到1条记录。"
            else:
                summary = f"查询成功，共找到{result_count}条记录。"

    return {
        "summary": summary,
    }


def _build_error_summary(error: str, error_type: str, user_query: str) -> str:
    """
    将技术错误转换为用户友好的提示

    Args:
        error: 错误信息
        error_type: 错误类型
        user_query: 用户问题

    Returns:
        用户友好的错误提示
    """
    # 错误类型到友好提示的映射
    error_templates = {
        "parsing_failed": "抱歉，系统无法理解您的问题。建议您换一种方式描述，或提供更多上下文信息。",
        "schema_retrieval_failed": "抱歉，系统暂时无法找到相关的数据表。请确认您的问题是否涉及系统已有的数据。",
        "generation_failed": "抱歉，系统在生成查询时遇到了问题。可能是问题过于复杂，建议您简化问题后重试。",
        "validation_failed": "抱歉，系统生成的查询存在问题，无法执行。建议您换一种方式提问。",
    }

    # 获取模板（如果error_type未知，使用通用模板）
    template = error_templates.get(error_type, "抱歉，系统遇到了问题，暂时无法处理您的请求。")

    # 第一阶段：直接返回模板
    # 第二阶段可扩展：使用轻量LLM生成更个性化的错误提示
    # if config.get("enable_llm_error_message"):
    #     return _generate_friendly_error_with_llm(error, error_type, user_query)

    return template


def _format_table(columns: List[str], rows: List[List[Any]]) -> str:
    """格式化结果为表格字符串"""
    if not rows:
        return "（无数据）"

    # 简单的表格格式
    lines = []
    lines.append(" | ".join(columns))
    lines.append("-" * 50)

    for row in rows:
        lines.append(" | ".join(str(v) for v in row))

    if len(rows) >= 10:
        lines.append("...")

    return "\n".join(lines)


def _format_multi_results(results: List[Dict], sub_queries: List[Dict], max_rows: int) -> str:
    """
    格式化多SQL结果为字符串

    ⚠️ 重要提示：本函数仅用于 Phase 2（Complex Path）

    Phase 1 说明：
    - Phase 1 只处理 Simple 问题，最多只有 1 条 SQL
    - 本函数在 Phase 1 不会被调用（代码永远走 if 分支）
    - 开发者在 Phase 1 阶段可以跳过本函数的实现

    Phase 2 说明：
    - Complex 问题会被拆分为多个子查询，生成多条 SQL
    - 本函数负责将多个查询结果格式化为易读的字符串
    - 每个结果会显示对应的子查询问题文本

    Args:
        results: SQL执行结果列表（Phase 2 可能有多个）
        sub_queries: 子查询列表
        max_rows: 每个结果最多显示的行数

    Returns:
        格式化后的多结果字符串
    """
    lines = []

    # 建立 SQL 到子查询的映射
    sql_to_subquery = {}
    for sq in sub_queries:
        if sq.get("validated_sql"):
            sql_to_subquery[sq["validated_sql"]] = sq

    for idx, result in enumerate(results, 1):
        sql = result.get("sql", "")

        # 获取对应的子查询信息
        sub_query = sql_to_subquery.get(sql)
        if sub_query:
            query_text = sub_query.get("query", f"子查询 {idx}")
            lines.append(f"\n【{query_text}】")
        else:
            lines.append(f"\n【子查询 {idx}】")

        # 格式化结果
        columns = result.get("columns", [])
        rows = result.get("rows", [])

        if rows:
            lines.append(_format_table(columns, rows[:max_rows]))
            if len(rows) > max_rows:
                lines.append(f"（还有 {len(rows) - max_rows} 行未显示）")
        else:
            lines.append("（无数据）")

        lines.append("")  # 空行分隔

    return "\n".join(lines)
```

## 8. 父图拓扑与连线

### 8.1 完整拓扑

```python
from langgraph.graph import StateGraph, END, START

def create_nl2sql_father_graph():
    """
    创建 NL2SQL 父图（第一阶段：Fast Path）

    拓扑：
    START → router → [simple_planner → sql_gen → [sql_exec → summarizer | summarizer] | summarizer] → END

    设计要点：
    1. Router 判定 simple → 进入 Simple Planner 准备参数
    2. Simple Planner → SQL生成子图（准备好参数后直接调用）
    3. Router 判定 complex → 直接进入 Summarizer（第一阶段返回"暂不支持"提示）
    4. SQL生成失败 → 跳过SQL执行，直接进入Summarizer
    5. 所有路径统一经过 Summarizer 节点，确保返回格式一致
    """
    # 创建状态图
    graph = StateGraph(NL2SQLFatherState)

    # ========== 添加节点 ==========
    graph.add_node("router", router_node)
    graph.add_node("simple_planner", simple_planner_node)  # 新增：Simple 问题参数准备
    graph.add_node("sql_gen", sql_gen_wrapper)
    graph.add_node("sql_exec", sql_execution_node)
    graph.add_node("summarizer", summarizer_node)  # 统一的响应构建器

    # ========== 添加边 ==========
    # 入口
    graph.add_edge(START, "router")

    # Router 条件边（共享 Summarizer 设计）
    graph.add_conditional_edges(
        "router",
        route_by_complexity,
        {
            "simple_planner": "simple_planner",  # simple → Simple Planner（参数准备）
            "summarizer": "summarizer",          # complex → 直接总结（暂不支持提示）
        }
    )

    # Simple Planner 固定边（参数准备完成后直接进入SQL生成）
    graph.add_edge("simple_planner", "sql_gen")

    # SQL生成后的条件边（优化：失败时跳过SQL执行）
    graph.add_conditional_edges(
        "sql_gen",
        route_after_sql_gen,
        {
            "sql_exec": "sql_exec",        # 生成成功 → 执行SQL
            "summarizer": "summarizer",    # 生成失败 → 直接总结错误
        }
    )

    # SQL执行后必定进入Summarizer
    graph.add_edge("sql_exec", "summarizer")
    graph.add_edge("summarizer", END)

    # ========== 编译 ==========
    app = graph.compile()

    return app


def route_after_sql_gen(state: NL2SQLFatherState) -> str:
    """
    SQL生成后的路由：判断是否成功

    成功标志：validated_sql 非空

    Returns:
        "sql_exec": SQL生成成功，进入执行节点
        "summarizer": SQL生成失败，跳过执行，直接总结错误
    """
    if state.get("validated_sql"):
        return "sql_exec"      # 生成成功 → 执行SQL
    else:
        return "summarizer"    # 生成失败 → 直接总结错误
```

```
┌──────────┐   ┌────────────────┐

说明：

2. Simple 问题 → Simple Planner 参数准备 → SQL生成失败 → Summarizer（转换技术错误为友好提示）
3. Complex 问题 → Summarizer（第一阶段返回"暂不支持"提示）
4. 所有路径统一经过同一个 Summarizer 节点，体现共享设计
5. Summarizer 内部自适应判断处理场景：
   - 场景0：complex 问题且无执行结果 → 暂不支持提示
   - 场景1：SQL生成失败 → 错误提示转换
   - 场景2：SQL执行失败 → 执行错误说明
   - 场景3：SQL执行成功 → 自然语言总结（单/多SQL自适应）
6. Simple Planner 是轻量级纯函数节点，不调用 LLM，执行时间可忽略不计
```

```python
import time
from typing import Dict, Any

def run_nl2sql_query(query: str, query_id: str = None) -> Dict[str, Any]:
    """
    执行 NL2SQL 查询（便捷函数）

    Args:
        query: 用户问题

    Returns:
        查询结果字典
    """
    # 创建父图
    app = create_nl2sql_father_graph()

    # 创建初始State
    initial_state = create_initial_state(query, query_id)

    # 记录开始时间
    start_time = time.time()

    # 执行
    final_state = app.invoke(initial_state)

    # 记录总耗时
    total_time_ms = (time.time() - start_time) * 1000
    final_state["total_execution_time_ms"] = total_time_ms

    result = extract_final_result(final_state)

    return result
```

```python
    # 基础信息

    query_id: str                      # 查询ID

    # 元数据
    metadata: Dict[str, Any]           # 性能指标等元信息
```

### 13.3 FAQ

**Q1: 为什么必须使用 Wrapper 而不能用 input/output schema？**

A: 因为父图State使用 `sub_queries` 列表管理子查询，而子图需要单个 `query` 字符串。如果用 input/output schema，父图State需要添加 `query`, `dependencies_results`, `parse_hints` 字段，会导致：
   - `query` 与 `sub_queries[].query` 重复（状态冗余）
   - 添加仅供子图使用的字段（状态污染）
   - 违背"最小State"设计原则

   Wrapper 负责从 `sub_queries` 中提取数据并转换，避免修改父图State，职责更清晰。

**Q2: SQL执行节点为什么要设计为列表形式？**

A: 为复杂路径预留。Fast Path 虽然只有单条SQL，但使用统一接口便于后续扩展。

**Q3: Router 失败时为什么判定为 complex？**

A: 保守策略。若判定为 simple 但实际是 complex，会导致错误；反之则只是走了慢路径，不影响正确性。

**Q4: Summarizer 为什么不使用模板而要调用 LLM？**

A: 提升用户体验。LLM 能生成更自然的答案。若成本敏感，可配置 `use_template: true`。

## 15. 设计调整说明（基于讨论）

本章节总结了基于设计评审讨论后的关键调整：

### 15.0 共享 Summarizer 节点优化（最新）

#### 15.0.1 第一次优化：SQL生成失败时跳过执行节点

**调整背景：**
最初设计中，无论SQL生成成功或失败，都会经过 `SQL执行节点 → Summarizer` 的完整链路，导致失败时也要经过SQL执行节点（虽然没有SQL可执行）。

**优化方案：**
添加条件边：SQL生成后判断是否成功

- **SQL生成成功** → 进入SQL执行节点 → Summarizer（总结结果）
- **SQL生成失败** → 跳过SQL执行节点 → 直接进入Summarizer（转换技术错误为友好提示）

#### 15.0.2 第二次优化：移除 summarizer_unsupported 节点，真正实现共享设计

**调整背景：**
第一次优化后，设计理念（第6节）已明确"Simple 和 Complex 路径共享同一个 Summarizer 节点"，但实现（第7节）中仍保留独立的 `summarizer_unsupported` 节点，存在以下问题：
1. 代码拓扑与设计理念不一致
2. 两个节点返回相同类型的响应，职责重叠
3. 未来扩展时需要同时维护两个节点

**优化方案：**
1. **移除独立节点**：删除 `summarizer_unsupported` 节点
2. **Router 路由调整**：
   - `complexity == "simple"` → `sql_gen`（不变）
   - `complexity == "complex"` → `summarizer`（原来是 `summarizer_unsupported`）
3. **Summarizer 内部判断**：在 `summarizer_node` 内部增加场景0的处理
   ```python
   # 场景0：第一阶段暂不支持的 complex 问题
   if complexity == "complex" and not execution_results:
       return {"summary": "抱歉，当前版本暂不支持复杂问题..."}
   ```

#### 设计要点

1. **Summarizer职责重新定位**
   - 从"结果总结器"升级为"**响应构建器**"
   - 负责将内部状态（成功/失败）转换为用户友好的最终响应
   - 作为所有路径的统一出口（Simple/Complex 共享）

2. **Router 路由优化**
   ```python
   def route_by_complexity(state: NL2SQLFatherState) -> str:
       if complexity == "simple":
           return "sql_gen"      # Simple → SQL生成子图
       else:
           return "summarizer"   # Complex → 直接进入Summarizer
   ```

3. **SQL生成后条件边**
   ```python
   def route_after_sql_gen(state: NL2SQLFatherState) -> str:
       if state.get("validated_sql"):
           return "sql_exec"      # 成功 → 执行SQL
       else:
           return "summarizer"    # 失败 → 直接总结
   ```

4. **Summarizer 内部自适应判断**
   根据 `complexity` 和 `execution_results` 自动识别处理场景：
   - **场景0**：complex 问题且无执行结果 → 暂不支持提示
   - **场景1**：SQL生成失败 → 技术错误转换为友好提示
   - **场景2**：SQL执行失败 → 执行错误说明
   - **场景3**：SQL执行成功 → 自然语言总结（单/多SQL自适应）

5. **错误提示转换**
   - 使用 `_build_error_summary()` 函数
   - 将技术错误类型映射为用户友好提示
   - 预留Phase 2扩展：使用LLM生成个性化错误提示

#### 优化收益
- ✅ **架构一致性**：代码实现与设计理念完全一致，真正实现共享设计
- ✅ **减少开销**：失败时跳过SQL执行节点
- ✅ **统一出口**：所有请求都经过唯一的Summarizer，返回格式一致
- ✅ **职责清晰**：单一节点处理所有响应构建逻辑，易于维护
- ✅ **语义清晰**：Summarizer作为"响应构建器"，自适应处理各种场景
- ✅ **友好体验**：将技术错误转换为用户可理解的自然语言
- ✅ **易于扩展**：
  - Phase 2 支持 Complex 路径时，只需修改场景0的逻辑
  - 未来可增强智能错误处理（建议用户重新提问、推荐相似问题等）
  - 单一节点扩展，无需维护多个节点


### 15.1 State 设计优化

#### 调整1: 支持复杂问题拆分
**原设计问题**:
- `query` 字段既是输入又是子查询，语义混淆
- `query_id` 作用不明确

**优化方案**:
- 新增 `user_query` 字段：明确保存用户原始问题（始终不变）
- `query_id` 明确为会话级ID（原始问题的ID）
- 新增 `sub_queries` 字段：支持问题拆分为多个子查询
  - Fast Path: 只有1个子查询（`query` = `user_query`）
  - Complex Path: 可包含多个子查询，每个有独立ID和依赖关系
- 新增 `current_sub_query_id` 字段：跟踪当前处理的子查询

**收益**:
- 清晰区分"用户原始问题"与"当前处理的子查询"
- 为复杂路径的问题拆分和依赖管理奠定基础
- 提升可追溯性和调试能力

#### 调整2: 精简State，移除子图内部细节
**原设计问题**:
- `schema_context` 和 `validation_history` 是子图内部细节，父图不需要

**优化方案**:
- 从父图State中完全删除这两个字段
- 子图内部细节通过日志系统观测，不传递到父图

**收益**:
- State更精简，减少序列化开销
- 避免状态污染，父图只关心结果（`validated_sql`）和错误（`error`）
- 真正实现关注点分离

#### 调整3: 删除冗余字段
**原设计问题**:
- `final_answer` 与 `summary` 语义重复
- `sql_texts` 字段与 `sub_queries` 中的 SQL 重复

**优化方案**:
- 删除 `final_answer`，统一使用 `summary`
- 删除 `sql_texts`，SQL统一从 `sub_queries` 中获取

**收益**:
- 避免数据冗余
- 单一数据源，减少不一致风险

### 15.2 节点实现优化

#### Router 判定原则强化
**判定标准**:
- **能用一条SQL解决，就不拆分**
- 允许的单SQL特性：多表JOIN、WITH子查询（CTE）、子查询、UNION、聚合、窗口函数
- 只有在必须根据第一次查询结果动态决定后续查询时，才判定为complex

**v1.6 更新**:
- Router 职责简化为**仅判定复杂度**（simple/complex）
- 参数准备职责移至 Simple Planner 节点（详见第4章）

#### SQL生成子图调用优化
**原设计**: 使用 input/output schema 直接从父图传递 `query`
**新设计**: 使用 Wrapper 从 `sub_queries` 中提取当前子查询，调用子图，并更新子查询状态

**选择Wrapper的原因**:
1. 父图使用 `sub_queries` 列表结构，子图需要单个 `query` 字符串
2. 避免在父图State中添加 `query`, `dependencies_results`, `parse_hints` 冗余字段
3. Wrapper 封装数据转换逻辑，保持父图State最小化

**状态管理**:
```python
# 调用前
sub_query["status"] = "pending"

# 调用后（成功）
sub_query["status"] = "completed"
sub_query["validated_sql"] = result

# 调用后（失败）
sub_query["status"] = "failed"
sub_query["error"] = error_msg
```

#### SQL执行节点优化
**原设计**: 从 `sql_texts` 列表获取SQL
**新设计**: 从 `sub_queries` 中筛选 `status == completed` 的子查询，获取 `validated_sql` 并执行

**双向绑定**:
- 执行结果写入 `execution_results`（全局汇总）
- 同时写入对应 `sub_query["execution_result"]`（便于追踪）

### 15.3 数据流优化对比

| 环节 | 原设计 | 新设计 | 优化点 |
|------|--------|--------|--------|
| 输入 | `query` | `user_query` + `sub_queries` | 语义明确，支持拆分 |
| 子图调用 | 直接传 `query` | 从 `sub_queries` 提取 | 统一数据源 |
| SQL获取 | `validated_sql` + `sql_texts` | `sub_queries[].validated_sql` | 消除冗余 |
| 执行结果 | 仅 `execution_results` | `execution_results` + `sub_query.execution_result` | 双向绑定 |
| 子图细节 | 传递 `schema_context`, `validation_history` | 不传递 | 精简State |
| 输出 | `summary` + `final_answer` | 仅 `summary` | 消除冗余 |

### 15.4 兼容性说明

**向前兼容（Phase 2 Complex Path）**:
- `sub_queries` 结构完整支持依赖关系（`dependencies` 字段）
- **v1.6 架构**：Simple Planner（Phase 1）创建单个子查询，Complex Planner（Phase 2）可创建多个
- Router 职责保持不变：仅判定复杂度，不创建子查询
- SQL执行节点天然支持多SQL执行（循环 `sub_queries`）

**向后兼容（与子图对接）**:
- 子图接口不变（仍接受 `query`, `query_id`, `user_query`）
- Wrapper 负责字段映射，子图无需感知父图变化

### 15.5 Phase 2 改造规划

#### 15.5.1 日志系统增强（支持子查询级追踪）

**Phase 1 现状**:
- Wrapper 向子图传入会话级 `query_id`（如 `"test001"`）
- 所有子图日志显示 `[test001] ...`
- 适用于单子查询场景，日志可按会话过滤

**Phase 2 改造需求**:
- Complex Path 支持多子查询并发执行（如 3 个子查询）
- 需要区分不同子查询的日志（当前无法区分）
- 需要保留会话级过滤能力（`grep "test001" logs/` 能匹配所有子查询）

**改造方案**：详见 **第 5.3 节 "query_id 参数设计说明"** 中的 Phase 2 部分，包括：

1. **子图 State 增加字段**：在 `SQLGenerationState` 中添加 `sub_query_id: Optional[str]` 字段
2. **Wrapper 传递两个ID**：同时传递会话级 `query_id` 和子查询级 `sub_query_id`
3. **子图节点日志改造**：所有节点使用组合ID格式（`[query_id:sub_query_id]`），实现细粒度追踪

**改造优势**:
- ✅ 向前兼容：Phase 1 不传 `sub_query_id` 时日志行为不变
- ✅ 会话追踪：可按会话级ID过滤所有子查询日志
- ✅ 细粒度追踪：可定位特定子查询的执行过程
- ✅ 并发场景支持：多子查询并发时日志清晰可辨

**实现时机**: Complex Planner 开发时同步改造（Phase 2 第一优先级）

## 16. 总结

本设计文档完成了 NL2SQL 父图第一阶段（Fast Path）的概要设计，并基于评审讨论进行了多轮重要优化。关键亮点：

1. **清晰的State设计**: 明确区分原始问题与子查询，支持问题拆分
2. **精简原则**: 移除子图内部细节和冗余字段，保持State最小化
3. **真正的节点共享**: SQL执行、Summarizer节点在 Simple/Complex 路径完全共享，无独立节点
4. **Wrapper集成方式**: 避免State冗余，使用Wrapper处理父子图数据转换
5. **配置驱动**: 统一使用 `nl2sql_father_graph.yaml` 模块配置文件，与子图保持一致架构
6. **智能路由优化**: 
   - SQL生成失败时跳过执行节点，直接进入Summarizer
   - Complex 问题直接路由到 Summarizer，无需独立节点
7. **响应构建器模式**: Summarizer作为唯一出口，内部自适应判断处理场景（4种场景）
8. **统一体验**: 所有路径返回格式一致，技术错误转换为用户友好提示
9. **LangGraph 1.0.x最佳实践**: 使用独立State，清晰的输入输出，合理使用条件边
10. **优秀的可扩展性**: 
    - `sub_queries` 结构完整支持 Phase 2 复杂路径
    - Phase 2 只需修改 Summarizer 场景0的逻辑，无需增加新节点
    - 预留智能错误处理能力
11. **完整的实施路径**: 从代码到测试到部署的完整清单

### 核心优化总结

| 优化点 | 问题 | 方案 | 收益 |
|--------|------|------|------|
| State设计 | 字段冗余、语义混淆 | 使用`user_query`+`sub_queries`结构 | 支持问题拆分，清晰可追溯 |
| 子图集成 | 使用input/output会导致State污染 | 使用Wrapper提取转换 | State最小化，职责清晰 |
| 配置管理 | 参数分散 | 独立模块配置文件 | 易于维护，与子图架构一致 |
| 失败处理 | 失败时也经过SQL执行节点 | 添加条件边，失败时跳过 | 减少开销，流程更高效 |
| 节点共享 | 独立的`summarizer_unsupported`节点与设计理念不符 | 移除独立节点，统一使用`summarizer` | 架构一致，易于维护和扩展 |
| 响应构建 | 技术错误直接暴露给用户 | Summarizer内部自适应判断，转换为友好提示 | 提升用户体验，统一返回格式 |

第一阶段完成后，将为简单问题提供快速响应（预期<1秒），并为后续复杂路径奠定坚实基础。
