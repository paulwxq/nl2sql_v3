# 56.1_父图graph_control遗留配置清理与术语统一方案

## 1. 文档目标

本文用于确认并固化父图 `graph_control` 遗留配置的清理方案，统一父图中的路径术语，并给出一份可执行的代码修改步骤清单。

本文聚焦以下三个结论：

1. `graph_control.enabled`、`graph_control.fast_path_enabled`、`graph_control.complex_path_enabled`、`graph_control.total_timeout` 当前都未被父图运行时代码实际消费，属于遗留未接线配置。
2. 文档中不新增任何 `complexity` 相关配置项；父图代码现有的运行时分类字段仍然是 `complexity`，其取值为 `"simple"` 或 `"complex"`。
3. `total_timeout` 涉及父图、子图、SQL 执行、LLM 调用的跨层预算传递，改造面较大，不应与本次配置清理混在同一阶段完成。

本文的目标不是讨论 Complex Path 的依赖注入问题，也不是引入新的调度机制，而是先把父图路径控制和术语体系收敛到一个更简单、真实、可维护的版本。

需要特别说明的是：

- `complexity` 不是本次要删除的配置参数
- `complexity` 是父图运行时已有的路由字段
- 本次要删除的是 `graph_control` 里的遗留配置项
- `simple / complex` 只是 `complexity` 的运行时取值，不是新的配置项
- 本文不要求新增 `complexity` 配置，也不要求在 YAML 中声明 `complexity == "simple"` 或 `complexity == "complex"`

---

## 2. 当前代码现状

### 2.1 `graph_control` 是遗留配置块

当前父图配置文件中定义了如下配置：

- `graph_control.enabled`
- `graph_control.total_timeout`
- `graph_control.fast_path_enabled`
- `graph_control.complex_path_enabled`

但父图实际运行过程中：

- 没有在入口处读取 `enabled`
- 没有在路由处读取 `fast_path_enabled`
- 没有在路由处读取 `complex_path_enabled`
- 没有在执行过程中落实 `total_timeout`

这意味着 `graph_control` 当前不是“有功能但未启用”，而是“配置存在但代码未接线”。

### 2.2 当前真实路由只依赖 `complexity`

父图真实路由逻辑是：

- Router 判定问题复杂度，输出 `complexity`
- 当 `complexity == "simple"` 时进入 `simple_planner`
- 否则进入 `planner`

因此，当前系统实际上只有两类问题分流语义：

- `simple`
- `complex`

不存在第三条独立的 `fast_path` 业务路径。所谓 `fast`，本质上只是历史上对简单路径的一种命名。

### 2.3 当前术语存在双轨并存问题

当前代码中同时存在两套术语：

- 分类术语：`simple / complex`
- 路径术语：`fast / complex`

这会带来以下问题：

- 新读代码的人容易误以为存在第三条 `fast_path`
- 配置项 `fast_path_enabled` 容易让人误判为“simple 之外还有单独路径”
- `path_taken == "fast"` 与 `complexity == "simple"` 之间存在不必要的概念映射
- 文档、日志、测试断言之间不够统一

## 3. 本次方案的决策

### 3.1 删除的配置项

本次方案确认删除以下 4 个参数：

1. `graph_control.enabled`
2. `graph_control.fast_path_enabled`
3. `graph_control.complex_path_enabled`
4. `graph_control.total_timeout`

删除理由如下。

#### 3.1.1 删除 `enabled`

- 当前没有任何运行时逻辑读取该参数。
- 该参数在现阶段没有灰度发布、熔断、降级切换等真实用途。
- 保留会让维护者误以为 `enabled=false` 可以关闭父图。

#### 3.1.2 删除 `fast_path_enabled`

- 当前系统没有独立的 `fast_path` 业务概念。
- 它实际对应的是 `complexity == "simple"` 这条简单路径。
- 如果关闭 simple 路径，系统也不会产生一个有独立业务意义的新模式，只会把所有请求都送去 complex 逻辑，配置收益很低。

#### 3.1.3 删除 `complex_path_enabled`

- 该配置当前没有接线。
- 它过去更像是一个测试/阶段性开关，而不是稳定的产品能力配置。
- 若后续仍需“临时禁用 Complex Path”能力，应改为测试代码控制、mock 控制、或单独的实验性开关，而不是长期保留在正式运行时配置中。

#### 3.1.4 删除 `total_timeout`

- 当前仅记录总耗时，并未形成真正的总超时控制。
- 该参数若保留，会让人误以为父图具备超时预算管理能力。
- 由于真正落地总超时需要跨父图、子图、SQL 执行、LLM 调用进行统一 deadline 传播，不适合在本阶段顺手补一个“伪超时”。

### 3.2 文档中不新增 `complexity` 配置

本次方案不新增以下内容：

- 不新增 `complexity == "simple"` 这种配置写法
- 不新增 `complexity == "complex"` 这种配置写法
- 不新增任何 `complexity` 相关 YAML 参数

原因是：

- `complexity` 已经是现有代码里的运行时字段
- 它由 Router 在执行过程中写入 state
- 它不是配置项，也不需要被包装成配置项写进方案里

### 3.3 代码层仍然保留现有 `complexity` 路由

本次方案确认：

- 父图唯一的运行时路径控制字段为 `complexity`
- 字段取值统一为 `"simple"` 或 `"complex"`
- 所有路由判断均基于该字段

这里的 `complexity` 需要明确区分：

- 它不是 YAML 配置参数
- 它不是人为手工维护的开关
- 它是 Router 在运行时写入 state 的分类结果

即：

- `complexity == "simple"` -> 进入简单路径
- `complexity == "complex"` -> 进入复杂路径

不再引入额外的路径开关配置。

也就是说，本次方案是“删除遗留配置，保留真实运行时分类字段”，而不是“删除 simple/complex 分类能力本身”。

### 3.4 统一术语为 `simple / complex`

本次方案确认术语统一规则如下：

1. 对外和对内的主术语统一为 `simple` 与 `complex`
2. `Fast Path`、`Complex Path` 这类旧术语可以作为历史说明逐步淡出，但不再作为主命名
3. `path_taken` 若继续保留，则其取值也应改为 `"simple" | "complex"`

这样可以消除下面这类双轨命名：

- `complexity = simple`
- `path_taken = fast`

统一后建议变为：

- `complexity = simple`
- `path_taken = simple`

这样能够保证分类、执行路径、日志、返回结果、测试断言完全一致。

## 4. 本阶段不做的事情

### 4.1 不在本阶段实现 `total_timeout`

本次改造明确不处理父图总超时。

原因不是总超时不重要，而是它不属于“小配置接线”问题，而属于“执行预算控制机制”问题。真实可用的总超时至少需要覆盖：

- 父图入口 deadline 计算
- 父图节点执行前剩余预算检查
- 子图调用时透传剩余预算
- SQL 执行层将单条 SQL 超时裁剪为剩余预算
- LLM 节点根据 provider 能力决定是否支持 per-request timeout
- 超时后的统一错误码和用户提示

当前系统中：

- 父图只是记录总耗时，不会中途打断
- 子图也只是记录总耗时，不会中途打断
- SQL 层已有单条 SQL 的 `statement_timeout`
- LLM 层对不同 provider 的 timeout 支持并不一致

因此，`total_timeout` 应作为下一阶段独立议题处理，不应与本次术语清理和配置删除混做。

### 4.2 不在本阶段改动 Complex Path 的执行能力

本次方案不涉及：

- 依赖注入闭环修复
- 动态重规划能力增强
- DeepAgents 引入
- Complex Path 成功链路重构

本次只做基础清理和命名统一。

## 9. 变更后的目标状态

本次方案落地后，父图路径控制的目标状态应为：

### 9.1 配置层

- 不再保留未接线的 `graph_control` 遗留参数
- 父图配置文件中只保留真正生效的参数

### 9.2 运行时语义层

- Router 输出 `complexity = "simple" | "complex"`
- 路由判断只依赖 `complexity`
- 返回结果中的 `path_taken` 与 `complexity` 术语保持一致

### 9.3 认知层

维护者对系统的理解应当变成：

- 当前父图只有两类路径语义：simple 和 complex
- simple 问题跳过父图问题分解 LLM，进入 `simple_planner`
- complex 问题进入 `planner`
- 不存在单独的 `fast_path_enabled` 这种产品级运行时概念

## 10. 下一阶段议题

本次文档确认将 `total_timeout` 明确延后到下一阶段。

下一阶段若要处理总超时，建议以独立设计文档展开，主题应当是：

- 父图/子图统一 deadline 机制设计
- 剩余预算在节点间的传递方式
- SQL timeout 与总预算的关系
- 不同 LLM provider 的 timeout 能力差异
- 超时错误码、日志与用户提示统一

只有在这些问题明确后，`total_timeout` 才应重新进入正式配置。

## 11. 最终结论

本次方案确认如下：

1. 删除 `graph_control.enabled`
2. 删除 `graph_control.fast_path_enabled`
3. 删除 `graph_control.complex_path_enabled`
4. 删除 `graph_control.total_timeout`
5. 父图路径控制统一收敛到 `complexity == "simple" | "complex"`
6. 项目中的路径术语统一收敛到 `simple / complex`
7. `path_taken` 若保留，也统一为 `simple / complex`
8. `total_timeout` 延后到下一阶段独立处理

这一步的目标不是增加能力，而是先把父图真实有效的控制逻辑、配置边界和术语体系整理干净，为后续 Complex Path 修复和 timeout 机制设计打下一个稳定基础。
