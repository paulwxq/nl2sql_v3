# 35.0 复杂查询依赖结果传递优化

## 文档信息

- **编号**: 35.0
- **创建日期**: 2025-11-16
- **版本**: v1.0
- **状态**: 待实施

## 一、问题背景

### 1.1 现状描述

在 Complex Path 的复杂查询拆解场景中，存在依赖关系的子查询需要使用前置查询的执行结果。当前实现存在以下问题：

**问题1：执行结果日志不完整**

SQL 执行成功后，日志只记录了行数和耗时，没有记录具体的返回数据：

```
2025-11-16 16:50:51 - INFO - SQL执行成功，返回 1 行，耗时 3ms
```

导致问题排查时无法快速了解执行结果的内容。

**问题2：依赖结果格式混乱**

`inject_params` 节点将完整的 `execution_result` 字典传递给 SQL 生成子图，格式化时简单地转换为字符串：

```python
def _format_dependencies(self, dependencies: Optional[Dict[str, Any]]) -> str:
    deps = []
    for query_id, result in dependencies.items():
        deps.append(f"- {query_id}: {result}")
    return "\n\n依赖查询结果：\n" + "\n".join(deps)
```

导致 LLM 收到的提示词包含大量冗余信息：

```
依赖查询结果：
- q_11e3134a_sq1: {'sub_query_id': 'q_11e3134a_sq1', 'sql': "SELECT...", 'success': True, 'columns': ['store_id', 'total_sales'], 'rows': [[303, Decimal('1662.00')]], 'row_count': 1, 'execution_time_ms': 3.244161605834961, 'error': None}
```

**问题3：语义不清晰**

LLM 收到完整字典后，需要自行理解：
- 哪个字段是真正需要使用的（应该是 `rows`）
- 依赖查询的业务含义是什么
- 如何将结果应用到当前查询

容易导致 LLM 误用 `sql` 字段或其他元数据。

**问题4：占位符被误识别为表名/列名**

当子查询文本中包含占位符（如 `{{sq1.result}}`）时，在问题解析或 Schema 检索阶段，这些占位符可能被误识别为：

- **表名**：Schema 检索尝试搜索名为 `sq1` 的表
- **列名**：问题解析尝试将 `result` 识别为列字段
- **关键词**：被当作无意义的噪声词处理

**示例**：

子查询文本：`"查询店铺 {{sq1.result}} 在9月份的总销售额"`

可能导致的错误：
```
问题解析阶段：
  dimensions: [
    {
      "text": "店铺",
      "role": "column"
    },
    {
      "text": "{{sq1.result}}",  ← 被识别为维度值，但无法匹配
      "role": "value"
    }
  ]

Schema检索阶段：
  向量检索关键词: ["查询", "店铺", "sq1", "result", "9月份", "总销售额"]
                                    ↑ 占位符被当作关键词
  可能检索到不相关的表/列
```

这不仅浪费了检索资源，还可能污染 Schema 上下文，降低 SQL 生成的准确性。

### 1.2 示例场景

用户查询：**"请问哪家店铺8月1日的销售额最高，请问它在9月份的总销售额是多少？"**

Planner 拆解为：
- **sq1**: 找出8月1日销售额最高的店铺ID
- **sq2**: 查询店铺 {{sq1.result}} 在9月份的总销售额（依赖 sq1）

执行流程：
1. sq1 执行成功，返回：`[[303, Decimal('1662.00')]]`
2. inject_params 将完整的 `execution_result` 注入到 sq2
3. sq2 的 SQL 生成收到混乱的依赖结果字符串

## 二、优化方案

### 2.1 优化范围

**本次优化（Phase 1）**：
- ✅ 问题1：增强 SQL 执行日志
- ✅ 问题2：优化依赖结果格式化
- ✅ 问题3：改进语义清晰度
- ⏸️ 问题4：保留占位符机制，**暂不处理**占位符误识别问题

**说明**：
- 本次采用**保守策略**，保留现有占位符机制不变
- 仅优化依赖结果的展示格式和日志输出
- 问题4（占位符误识别）留待后续阶段通过问题解析增强来解决

### 2.2 核心思路

**简化依赖传递逻辑，提供清晰的 question-result 语义结构**

- 保持子查询文本中的占位符不变（如 `{{sq1.result}}`）
- 在 SQL 生成提示词中，以结构化格式展示依赖问题和结果
- 让 LLM 根据占位符和清晰的依赖信息共同理解和使用

**步骤2 - 格式化输出（sql_generation）**:

重写 `_format_dependencies()` 方法，从新数据结构中提取关键信息：

```python
def _format_dependencies(self, dependencies: Optional[Dict[str, Any]]) -> str:
    """格式化依赖结果
    
    Args:
        dependencies: 依赖结果字典，格式：
            {
                "query_id": {
                    "question": "...",
                    "execution_result": {...}
                }
            }
    
    Returns:
        格式化后的依赖信息字符串
    """
    if not dependencies:
        return ""
    
    max_rows = self.config.get("dependencies_formatting", {}).get("max_display_rows", 5)
    include_columns = self.config.get("dependencies_formatting", {}).get("include_columns", True)
    
    deps = []
    for query_id, dep_data in dependencies.items():
        # 提取关键信息
        question = dep_data.get("question", f"子查询 {query_id}")
        exec_result = dep_data.get("execution_result", {})
        
        rows = exec_result.get("rows", [])
        columns = exec_result.get("columns", [])
        
        # 构建格式化输出
        deps.append(f"  - {query_id}:")
        deps.append(f"    question: {question}")
        
        if rows:
            deps.append(f"    result:")
            display_rows = rows[:max_rows]
            for row in display_rows:
                deps.append(f"      - {row}")
            
            if len(rows) > max_rows:
                deps.append(f"    ... 共 {len(rows)} 行，仅显示前 {max_rows} 行")
        
        if include_columns and columns:
            deps.append(f"    columns: {columns}")
    
    return "\n\n可能依赖的查询结果：\n" + "\n".join(deps)
```

**输出格式示例**：

```yaml
可能依赖的查询结果：
  - q_11e3134a_sq1:
    question: 找出8月1日销售额最高的店铺ID
    result: 
      - [303, 1662.00]
      - [205, 1580.50]
    columns: [store_id, total_sales]
```

**关键字段说明**:
- `question`: 依赖查询的问题描述（从 sub_query.query 获取）
- `result`: 查询结果数据行（从 execution_result.rows 提取）
- `columns`: 列名（从 execution_result.columns 获取）

**提取策略**:
- 显示前 N 行（可配置，默认 5 行）
- 超过限制时，添加提示：`... 共 M 行，仅显示前 N 行`
- 不显示 SQL、execution_time_ms 等元数据（减少 token 消耗）

#### 优化点3：保持占位符机制不变

**位置**: 无需修改

**说明**:
- 本次优化**保留现有占位符机制**（如 `{{sq1.result}}`）
- 子查询 `query` 字段继续包含占位符
- 不修改 Planner、inject_params 的占位符处理逻辑
- 不扩展 state schema

**原因**:
- 占位符是当前系统中唯一的依赖引用机制
- 移除占位符需要大量重构（state schema、测试用例、提示词）
- 采用保守策略，优先解决格式和日志问题

**未来优化方向**（Phase 2）:
- 在问题解析阶段增加占位符识别和过滤逻辑
- 避免占位符被误识别为表名/列名
- 在 Schema 检索阶段过滤占位符关键词

### 2.4 完整提示词示例

```
你是 PostgreSQL SQL 生成专家。根据以下上下文生成 SQL。

要求：
1. 仅输出 SQL，不附加说明。
2. 所有表必须包含 schema 前缀（例如 public.table）。
3. 日期时间过滤使用指定列：同时具备 start 与 end 时使用 >= start AND < end；仅有 start 时使用 >= start；仅有 end 时使用 < end。
4. JOIN 条件必须严格按照 ON 模板，用实际别名替换 SRC/DST。
5. 以下提供的表结构、JOIN 计划和时间列为参考，你可以选择性使用它们，但不得使用未列出的字段。

---

方言：postgresql
问题：查询店铺 {{sq1.result}} 在9月份的总销售额
日期时间粒度：month
日期时间条件：>= 2025-09-01 AND < 2025-10-01
维度过滤：column=店铺, value={{sq1.result}}
指标：总销售额

可能依赖的查询结果：
  - q_11e3134a_sq1:
    question: 找出8月1日销售额最高的店铺ID
    result: 
      - [303, 1662.00]
    columns: [store_id, total_sales]
    
---

表类型：
  - 事实表: public.fact_store_sales_day, public.fact_store_sales_month
  - 维度表: public.dim_company, public.dim_store

---

表结构：
- **public.dim_company**  公司维表（类型：维度表）。字段：company_id(integer) 公司ID（主键）[例:3]；company_name(character varying) 公司名称，唯一[例:喜士多]。
- **public.dim_store**  店铺维表（类型：维度表）。字段：store_id(integer) 店铺ID（主键）[例:101]；store_name(character varying) 店铺名称（同一公司下唯一）[例:全家苏州姑苏店]；company_id(integer) 所属公司ID（外键）[例:3]；region_id(integer) 所属区（县）ID（外键）[例:320508]。
- **public.fact_store_sales_day**  店铺销售日流水事实表（按天、店铺、商品类型汇总），类型：事实表。按日汇总数据，时间字段为 date_day。字段：store_id(integer) 店铺ID（外键）[例:203]；date_day(date) 交易日期（日粒度）[例:2025-08-15]；product_type_id(integer) 商品类型ID（外键）[例:1]；amount(numeric) 销售金额（当日）[例:326.00]。
- **public.fact_store_sales_month**  店铺销售月汇总事实表（按月、店铺、商品类型汇总），类型：事实表。按月汇总数据，时间字段为 date_month。字段：store_id(integer) 店铺ID（外键）[例:203]；date_month(date) 月份（建议为当月第一天）[例:2025-08-01]；product_type_id(integer) 商品类型ID（外键）[例:1]；amount(numeric) 销售金额（当月累计值）[例:666.00]。

---

JOIN 计划：
Base #1：**public.fact_store_sales_month**
- public.fact_store_sales_month --INNER JOIN (N:1)--> public.dim_store ON SRC.store_id = DST.store_id
- public.fact_store_sales_day --INNER JOIN (N:1)--> public.dim_store ON SRC.store_id = DST.store_id
- public.dim_store --INNER JOIN (N:1)--> public.dim_company ON SRC.company_id = DST.company_id

---

时间列：
- public.fact_store_sales_day.date_day
- public.fact_store_sales_month.date_month
```

- 识别 `{{...}}` 模式并标记为依赖引用
- 不将占位符内容用于维度值匹配
- 在维度过滤中标记"此值来自依赖"

**Schema 检索阶段**增强:
- 从关键词提取中过滤占位符模式
- 避免 `sq1`、`result` 等词汇污染检索

**预期效果对比**:

| 维度 | Phase 1（本次） | Phase 2（未来） |
|------|----------------|----------------|
| 问题文本 | `"查询店铺 {{sq1.result}} 在9月份的总销售额"` | 保持不变 |
| 关键词提取 | `["查询", "店铺", "sq1", "result", "9月份", "总销售额"]` | `["查询", "店铺", "9月份", "总销售额"]` |
| 维度识别 | 可能将 `{{sq1.result}}` 识别为无效维度值 | 识别为依赖引用 |
| Schema 检索 | 包含噪声关键词 | 纯净的业务关键词 |

## 五、数据格式规范

### 5.1 标准 execution_result 格式

所有 SQL 执行结果必须遵循以下标准格式：

```python
{
    "sub_query_id": "q_xxx_sq1",           # 子查询ID
    "sql": "SELECT ...",                   # 执行的SQL
    "success": True,                       # 是否成功
    "columns": ["col1", "col2"],           # 列名数组
    "rows": [[val1, val2], [...]],         # 行数据（二维数组）
    "row_count": 2,                        # 行数
    "execution_time_ms": 3.24,            # 执行耗时（毫秒）
    "error": None                          # 错误信息（失败时）
}
```

**字段说明**：

| 字段 | 类型 | 必需 | 说明 |
|------|------|------|------|
| `sub_query_id` | str | ✅ | 子查询唯一标识 |
| `sql` | str | ✅ | 执行的 SQL 语句 |
| `success` | bool | ✅ | 执行是否成功 |
| `columns` | List[str] | ✅ | 列名数组 |
| `rows` | List[List[Any]] | ✅ | 行数据（二维数组） |
| `row_count` | int | ✅ | 返回行数 |
| `execution_time_ms` | float | ✅ | 执行耗时（毫秒） |
| `error` | str \| None | ✅ | 错误信息（成功时为 None） |

**多行多列示例**：

```python
{
    "sub_query_id": "q_11e3134a_sq1",
    "sql": "SELECT store_id, store_name, total_sales FROM ...",
    "success": True,
    "columns": ["store_id", "store_name", "total_sales"],
    "rows": [
        [303, "苏州姑苏店", 1662.00],
        [205, "北京朝阳店", 1580.50],
        [108, "上海浦东店", 1520.30]
    ],
    "row_count": 3,
    "execution_time_ms": 5.42,
    "error": None
}
```

## 六、测试用例

### 6.1 单依赖场景

**用户查询**: "请问哪家店铺8月1日的销售额最高，请问它在9月份的总销售额是多少？"

**预期**:
- sq1 执行结果正确记录到日志
- sq2 收到清晰的依赖结果格式
- sq2 的 SQL 正确使用 store_id=303 进行过滤

### 6.2 多依赖场景

**用户查询**: "找出销售额最高的店铺和公司，查询它们的详细信息"

**预期**:
- 多个依赖结果独立展示
- 每个依赖的 question-result 结构清晰
- LLM 能够正确组合多个依赖结果

### 6.3 多行结果场景

**用户查询**: "找出销售额前10的店铺，分别统计它们的月度趋势"

**预期**:
- 依赖结果显示前 5 行（可配置）
- 提示"共 10 行，仅显示前 5 行"
- 后续查询正确处理所有 10 个店铺

## 七、后续优化方向

### 7.1 智能结果提取

根据后续查询的需求，智能提取依赖结果：
- 仅需要 ID → 提取第一列
- 需要多个字段 → 完整展示
- 结果过大 → 自动采样或聚合

### 7.2 依赖关系可视化

在日志或诊断信息中，可视化展示依赖关系：
```
sq1 (completed) ──→ sq2 (in_progress)
                  └→ sq3 (pending)
```

### 7.3 依赖结果缓存

对于重复依赖的结果，避免重复传递：
```yaml
dependencies:
  shared_results:
    company_list: [1, 2, 3, 5]
  sq2_deps:
    - ref: company_list
  sq3_deps:
    - ref: company_list
```
