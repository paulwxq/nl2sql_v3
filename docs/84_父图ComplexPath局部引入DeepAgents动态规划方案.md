# 84_父图ComplexPath局部引入DeepAgents动态规划方案

## 1. 文档目的

本文给出一份独立的重构建议：在 **不推翻现有 LangGraph 父图与 SQL 生成子图架构** 的前提下，仅对父图中的 **Complex Path** 引入 `langchain-ai/deepagents`，用于增强复杂问题的动态规划、反思和重规划能力。

本文强调三件事：

1. **Fast Path 保持不动**，继续追求低延迟和稳定性。
2. **SQL 子图保留**，继续承担问题重写、结构化解析、Schema 检索、SQL 生成、验证与重试。
3. `deepagents` **只接管复杂问题的规划编排层**，而不是接管整套 NL2SQL 系统。

---

## 2. 当前系统的真实结构

当前系统并不是一个“单一的父图拆解模块”，而是由多个父图节点协同完成复杂问题的分析、拆解、执行和回答：

- `Router`
  - 判断问题是 `simple` 还是 `complex`
- `Simple Planner`
  - Simple 问题直接生成一个子查询
- `Planner`
  - 对复杂问题一次性输出多个子查询和依赖关系
- `Inject Params`
  - 将已完成子查询的执行结果注入后续子查询
- `SQL Generation Subgraph`
  - 负责问题解析、问题重写、Schema 检索、SQL 生成、验证和重试
- `SQL Execution`
  - 执行验证通过的 SQL
- `Check Completion`
  - 控制复杂路径循环是否结束
- `Summarizer`
  - 生成最终自然语言回答

这套架构的核心特征是：

- 拓扑清晰
- 状态显式
- 错误点可定位
- 适合稳定生产

但它在复杂问题处理上仍然存在明显的静态规划特征。

---

## 3. 当前 Complex Path 的主要问题

### 3.1 一次性静态规划过强

当前 `Planner` 在问题开始阶段就一次性产出全部子查询和依赖关系。这种方式适合结构稳定的问题，但不适合中间结果会影响后续决策的问题。

典型问题：

- 第一步查出来为空，后续步骤本来就不该继续执行
- 第一步结果与预期不符，需要改写第二步问题
- 某一步 SQL 虽然成功执行，但结果不足以支持后续问题，需要补查

当前设计对这些情况的支持较弱。

### 3.2 手写调度逻辑较重

Complex Path 目前依赖多块手工逻辑共同维持：

- 子查询列表维护
- 依赖图维护
- 依赖结果注入
- 轮次推进
- 最大轮次保护
- 孤立节点和循环依赖处理

这些逻辑并非错误，但随着复杂场景增加，维护成本会上升。

### 3.3 “结果驱动重规划”能力不足

当前系统更像：

- 先定计划
- 再逐步执行

而不是：

- 先有初始计划
- 执行过程中根据结果持续修正计划

这正是 `deepagents` 相对更强的部分。

---

## 4. 为什么选择“局部引入”而不是“整体替换”

本项目不适合把父图和子图整体改造成通用 Agent Loop，原因如下。

### 4.1 现有 SQL 子图已经具备较强专业能力

当前 SQL 子图并不是简单的“生成 SQL”函数，它已经包含：

- 多轮对话指代消解
- 问题重写
- 结构化解析
- Schema 检索
- SQL 生成
- SQL 验证
- 失败重试

这是本项目最专业、最有业务价值的能力沉淀，不应轻易替换。

### 4.2 NL2SQL 场景需要受控编排，而不是完全开放式代理

`deepagents` 更适合开放式多步任务，比如：

- 研究类任务
- 文件整理类任务
- 通用代码操作类任务

而 NL2SQL 的生产目标是：

- 生成正确 SQL
- 可追踪
- 可解释
- 可回退
- 可测试

因此，最合理的方案是：

- 保留现有受控执行层
- 只在复杂问题规划层引入更强的动态能力

### 4.3 复杂路径当前本身还需要继续验证

当前复杂路径已有实现，但仍需要更多成功链路验证。此时直接整体替换，会同时引入两类风险：

- 原有复杂路径未完全验证
- 新框架能力边界未完全验证

这会让问题定位困难。

---

## 5. 目标方案总览

### 5.1 总体原则

新方案采用“**Hybrid Orchestration**”：

- 父图仍然是 LangGraph 主控图
- Fast Path 仍然是确定性工作流
- Complex Path 新增一个 DeepAgents 编排节点
- DeepAgents 只负责复杂问题的动态任务推进
- SQL 子图继续作为专业工具被调用

### 5.2 新的路径划分

#### Fast Path

保持原样：

`Router -> Simple Planner -> SQL 子图 -> SQL Execution -> Summarizer`

适用目标：

- 单条 SQL 可解决的问题
- 高频场景
- 对延迟敏感的场景

#### Complex Path

改造为：

`Router -> Deep Complex Orchestrator -> Summarizer`

其中 `Deep Complex Orchestrator` 内部使用 `deepagents` 实现动态规划与多轮工具调用。

---

## 6. 哪些能力被替换，哪些能力被保留

### 6.1 建议被替换的能力

以下能力建议由 DeepAgents 接管：

#### 1. 复杂问题的静态拆解

原本由 `Planner` 一次性输出 JSON 子查询列表。

替换后：

- DeepAgents 使用 `write_todos` 动态维护任务列表
- 可以先给出初始步骤
- 执行后根据结果再补充、修改、删除后续步骤

#### 2. 手工依赖推进逻辑

原本由 `Inject Params` 判断哪些子查询可以执行。

替换后：

- 由 Agent 根据当前已知结果决定下一步要做什么
- 不再依赖固定的“依赖满足即可进入下一批”逻辑

#### 3. 手工循环收敛逻辑

原本由 `Check Completion` 判断：

- 是否还有待执行子查询
- 是否达到最大轮次
- 是否出现无法推进的状态

替换后：

- DeepAgents 根据 Todo 状态和目标达成度自行推进
- 但必须保留外层硬约束，防止无限循环

### 6.2 必须保留的能力

以下能力不建议在第一阶段被替换：

#### 1. SQL 子图

保留现有 SQL 子图作为专业能力核心，继续负责：

- 问题重写
- 解析
- Schema 检索
- SQL 生成
- SQL 验证与重试

#### 2. SQL 执行模块

保留现有执行逻辑与结果规范化格式，不要让 DeepAgents 直接绕开现有执行层。

#### 3. Summarizer

最终回答生成仍建议复用现有 `Summarizer`，保持输出风格一致，降低回归风险。

#### 4. 父图总状态与链路信息

保留父图中的：

- `query_id`
- `thread_id`
- `conversation_history`
- `metadata`
- `execution_results`

这部分是可观测性和多轮能力的基础，不能丢。

---

## 7. 推荐的 DeepAgents 接入边界

这是整个方案最关键的部分。

### 7.1 不建议的接入方式

以下做法不建议在第一阶段采用：

- 让 DeepAgents 直接读写数据库
- 让 DeepAgents 直接生成最终 SQL 并跳过 SQL 子图
- 让 DeepAgents 使用文件系统、shell、subagents 等全部默认能力
- 让 Agent 完全依赖自然语言上下文来记忆前序结果

这些做法虽然灵活，但会显著降低可控性。

### 7.2 推荐的接入方式

推荐只给 DeepAgents 极少量、高抽象、强约束的工具。

第一阶段建议只提供以下工具：

#### 1. `solve_subproblem_tool`

职责：

- 输入一个子问题描述
- 可选输入依赖结果摘要
- 内部调用现有 SQL 子图和 SQL 执行模块
- 返回结构化结果

这个工具应当是第一阶段最核心的工具。

它的价值在于：

- DeepAgents 只负责“下一步做什么”
- 具体怎么把问题转成 SQL，仍由现有子图负责

#### 2. `finalize_answer_tool` 或等价的父图总结阶段

职责：

- 接收复杂路径执行产物
- 进入现有 `Summarizer`
- 统一最终输出风格

如果实现更简单，也可以不做这个工具，而是在 DeepAgents 结束后把产物映射回父图状态，再走现有 `Summarizer`。

#### 3. 可选工具：`inspect_schema_tool`

仅在后续需要时再加。

作用：

- 当 DeepAgents 多次调用 `solve_subproblem_tool` 仍无法推进时
- 允许它主动查看表、字段、业务别名等高层信息

但这不是第一阶段必需项。

---

## 8. Complex Path 的目标执行模型

### 8.1 执行方式

当 `Router` 判定为 `complex` 后：

1. 父图进入 `Deep Complex Orchestrator`
2. 该节点内部启动一个受控 DeepAgent
3. Agent 先建立 Todo 列表
4. Agent 逐步调用 `solve_subproblem_tool`
5. 每完成一步，根据结果决定：
   - 标记完成
   - 修改下一步
   - 新增一步
   - 提前收敛
6. DeepAgent 输出结构化执行产物
7. 父图将执行产物映射回统一状态
8. 进入现有 `Summarizer`

### 8.2 DeepAgent 的职责边界

DeepAgent 负责：

- 复杂问题拆解
- 动态任务推进
- 中间结果观察
- 失败后的轻量重规划

DeepAgent 不负责：

- 直接控制数据库连接
- 自由调用 shell
- 自由访问文件系统
- 替代 SQL 子图的专业判断

---

## 9. 为什么推荐使用单个高层工具，而不是拆成多个底层工具

表面上看，把能力拆成多个工具似乎更灵活，例如：

- `generate_sql_tool`
- `execute_query_tool`
- `validate_sql_tool`

但第一阶段不建议这样做，原因如下。

### 9.1 会破坏现有子图的能力闭环

现有 SQL 子图内部已经组织好了：

- 解析
- 检索
- 生成
- 验证
- 重试

如果把这些步骤拆成多个底层工具，DeepAgents 就要自己重新学会如何正确调用这些步骤，等于把子图内部编排重新暴露给 Agent。

这会降低稳定性。

### 9.2 会显著增加 Agent 决策负担

Agent 本来只需要决定：

- 下一步要解决什么问题

如果工具过细，Agent 还要决定：

- 先检索还是先生成
- 失败后是重试验证还是回到 Schema 检索
- 当前错误该用哪一种工具恢复

这会让 Agent loop 更长、成本更高、结果更不稳定。

### 9.3 不利于结果标准化

一个高层工具可以稳定返回统一结构：

- 子问题文本
- validated SQL
- execution result
- 成功或失败状态
- 失败原因

这更利于父图和 `Summarizer` 继续复用。

---

## 10. 依赖结果传递的建议

这是方案中最容易误判的一点。

### 10.1 不建议完全依赖自然语言上下文

虽然 DeepAgents 可以在多轮对话里保留前序结果，但对于 NL2SQL 来说，单靠自然语言上下文存在风险：

- 数值精度容易丢失
- 行列结构不稳定
- 多结果引用容易混淆
- 后续 SQL 生成难以形成稳定 prompt

### 10.2 建议保留结构化中间产物

即使去掉现有的 `Inject Params` 节点，也建议保留一个结构化中间结果仓：

- 每个 Todo 对应一个结果对象
- 每个结果对象至少包含：
  - 原问题
  - validated SQL
  - execution result
  - 结果摘要
  - 成功/失败状态

DeepAgents 在规划下一步时，可以读取这些结构化结果摘要。

同时，`solve_subproblem_tool` 接收的也不应只是纯文本，而应能拿到结构化依赖上下文。

这样做的好处是：

- 不再依赖现有 `Inject Params` 的固定节点逻辑
- 但仍然保留对中间结果的结构化控制

---

## 11. 对现有父图状态的建议

即使引入 DeepAgents，父图状态也不应被简化到只剩一个 `summary`。

建议保留以下状态概念：

- `complexity`
- `path_taken`
- `query_id`
- `thread_id`
- `conversation_history`
- `execution_results`
- `summary`
- `error`
- `metadata`

同时为 DeepAgents 新增一组复杂路径产物字段，例如：

- `complex_artifacts`
  - 保存每个 Todo 或子任务的产物
- `deep_agent_trace`
  - 保存 Todo 变化摘要
- `deep_agent_metrics`
  - 保存步骤数、工具调用数、修复次数等指标

这样做的目的不是“保留所有过程细节”，而是保证：

- 可以追踪为什么这样回答
- 可以排查为什么复杂问题失败
- 可以与旧复杂路径做效果对比

---

## 12. 特性开关设计

本方案必须使用显式 Feature Flag，不建议直接替换旧路径。

建议至少增加以下配置项：

- `complex_path_enabled`
- `complex_path_mode`
  - `static`
  - `deepagents`
- `deepagents_enabled`
- `deepagents_max_steps`
- `deepagents_max_replans`
- `deepagents_max_empty_result_repairs`

推荐行为：

- 默认仍为 `static`
- 仅在灰度环境启用 `deepagents`
- 允许按租户、按接口、按问题类型进行灰度

---

## 13. 运行时保护机制

DeepAgents 的价值在于动态性，但动态性必须被约束。

建议在第一阶段强制加入以下保护：

### 13.1 Step 上限

限制 Agent 最多进行多少轮工具调用与重规划。

### 13.2 重规划上限

限制因为结果异常导致的重写 Todo 次数。

### 13.3 空结果修复上限

避免 Agent 因为“查不到数据”而无限尝试更宽松条件。

### 13.4 单问题总时长上限

防止复杂问题长时间占用资源。

### 13.5 工具白名单

第一阶段仅允许：

- `write_todos`
- `solve_subproblem_tool`
- 可选的少量高层辅助工具

不要给默认文件系统工具、shell 工具或其他与 NL2SQL 无关的工具。

---

## 14. 与现有 Summarizer 的关系

推荐第一阶段继续使用现有 `Summarizer`。

原因如下：

### 14.1 输出风格统一

无论是 Fast Path 还是 Complex Path，最终都走统一回答模块，便于：

- API 对齐
- CLI 展示对齐
- 用户体验一致

### 14.2 回归风险更小

如果同时改规划层和最终回答层，问题定位会变得困难。

### 14.3 便于双轨对比

同一个 `Summarizer` 可以对比：

- 旧复杂路径产物
- 新 DeepAgents 产物

从而只评估规划层变化带来的收益。

---

## 15. 推荐的阶段化落地计划

### 阶段 0：先修现有复杂路径基础问题

在引入 DeepAgents 之前，建议先做两件事：

1. 让 `complex_path_enabled` 真正生效
2. 补齐 Complex Path 成功链路的端到端测试

原因：

- 先保证旧路径可以稳定作为 fallback
- 否则新旧路径都不稳，无法比较

### 阶段 1：引入受控 DeepAgent 原型

目标：

- 新增 `Deep Complex Orchestrator`
- 只接入 `write_todos`
- 只提供 `solve_subproblem_tool`
- 不引入文件系统、shell、subagents

产出：

- 能处理一类典型复杂问题
- 能根据中间结果动态调整后续步骤

### 阶段 2：补充结构化复杂产物沉淀

目标：

- 建立复杂路径的中间结果对象模型
- 把 Todo 执行产物稳定映射回父图状态
- 让 `Summarizer` 可以消费这些产物

### 阶段 3：灰度与效果评估

目标：

- 与旧静态 Planner 路线并行运行
- 比较准确率、成功率、时延、Token 成本
- 识别适合 DeepAgents 的复杂问题类型

### 阶段 4：按需增加辅助工具

只有在第一阶段证明有效之后，再考虑增加：

- `inspect_schema_tool`
- 结果摘要工具
- 复杂问题诊断工具

不建议过早增加工具种类。

---

## 16. 效果评估指标

引入 DeepAgents 后，不应只看“能不能跑通”，还应明确衡量。

建议追踪：

- Complex 问题成功率
- 动态重规划触发率
- 空结果后恢复成功率
- 平均工具调用次数
- 平均总耗时
- 平均 Token 成本
- 最终回答正确率
- 用户可接受率

还建议做问题分层：

- 结构化多跳查询
- 需要先定位对象再查详情的问题
- 需要结果驱动条件调整的问题
- 多指标复合问题

DeepAgents 未必对所有复杂问题都更优，应该按问题类型评估。

---

## 17. 主要风险与应对

### 风险 1：延迟上升

动态规划通常比静态 DAG 更慢。

应对：

- Fast Path 不变
- 只在 Complex Path 启用
- 设置严格步骤上限

### 风险 2：成本上升

Agent loop 天然更耗 Token。

应对：

- 使用高层工具减少不必要的 tool loop
- 限制重规划次数
- 限制中间结果注入大小

### 风险 3：行为不可控

如果给 Agent 过多自由，会出现偏航。

应对：

- 工具最小化
- 不开放通用 shell/filesystem
- 明确终止条件

### 风险 4：可观测性下降

如果只依赖自然语言过程，排查会很困难。

应对：

- 保存 Todo 变化摘要
- 保存每一步工具调用的结构化产物
- 继续映射回父图状态

### 风险 5：新框架稳定性风险

`deepagents` 相对较新。

应对：

- 通过 `complex_path_mode` 保留静态旧路径
- 任何时候允许回退
- 不在第一阶段侵入 SQL 子图核心

---

## 18. 最终建议

我的最终建议是：

**采用“保留 LangGraph 主体、仅在 Complex Path 局部引入 DeepAgents 动态规划”的混合架构。**

更具体地说：

- 不重写整个父图
- 不重写 SQL 子图
- 不让 DeepAgents 直接替代现有 SQL 生成能力
- 只让 DeepAgents 负责复杂问题的动态规划与步骤推进
- 通过少量高层工具与现有子图协同

这条路线的优势是：

- 最大化复用现有专业能力
- 最小化重构风险
- 真正利用 `write_todos` 的价值
- 为后续复杂场景扩展保留空间

一句话总结：

**DeepAgents 应该成为复杂问题的“动态规划层”，而不是成为整个 NL2SQL 系统的新内核。**
