📌 核心Tool（不调用LLM）：

  • 执行计划Tool：解析诊断Agent输出的JSON，初始化state
  • 查找当前批次Tool：根据依赖关系，找出可以并行执行的子查询
  • Schema检索Tool：向量检索+图检索，获取相关表、列、JOIN路径
  • 三层验证Tool：语法、安全、语义三层验证SQL

📋 完整执行流程

🔹 主流程：

  1. 用户输入查询 → 进入诊断和拆解Agent
  2. 诊断和拆解Agent（🆕 增强）：
    • 步骤1：判断数据库范围
      • 问题在DB范围内 → 输出 route_to="sql"，继续步骤2
      • 问题不在DB范围内 → 输出 route_to="chat"，直接路由到聊天Agent
    • 步骤2：分析并拆解查询（仅当route_to="sql"时）
      • 普通查询 → 1个子查询
      • 复杂查询 → 多个子查询（带依赖关系）
  3. 路由决策：
    • route_to="chat" → 聊天Agent（配备search_tool）
    • route_to="sql" → SQL流程
  4. SQL流程（route_to="sql"时）：
    • 解析执行计划 → 初始化state
    • 循环处理批次：
      • 查找当前批次（依赖已满足的子查询）
      • 批次SQL生成（调用SQL生成子图N次）
      • 验证：所有SQL都成功？
        • ❌ 任何一个失败（超过3次重试）→ 降级到聊天Agent
        • ✅ 全部成功 → 并行执行批次SQL
      • 🆕 SQL执行结果检查：
        • ✅ 成功 → 存储结果，继续下一批次
        • ❌ 失败 → 判断错误类型：
          • 🔵 可恢复错误（表不存在、字段不存在、数据超出范围）→ Fallback到聊天Agent
          • 🔴 不可恢复错误（连接失败、超时、权限）→ 返回系统错误（不降级）
    • 所有批次完成 → 结果聚合 → Summary Agent → 返回答案
  5. 🆕 聊天Agent流程：
    • 聊天Agent分析问题
    • 自主决定：
      • 需要搜索 → 调用search_tool → 基于结果回答
      • 不需要搜索 → 直接基于知识回答

💡 诊断和拆解Agent - Prompt示例

你是一个查询诊断和拆解Agent。

【第一步：判断数据库范围】

数据库覆盖范围：
- 电商订单数据（2020年1月 - 2025年1月）
- 涵盖内容：用户信息、商品信息、订单信息、销售数据、库存数据
- 不包括：天气信息、新闻资讯、概念解释、行业分析、宏观经济、未来预测

判断标准：
1. 如果问题明确涉及上述数据范围 → 在数据库范围内
2. 如果问题明确不涉及上述数据（如天气、新闻、概念解释）→ 不在数据库范围内
3. 如果边界模糊但可能相关（如"2030年销售预测"需要历史数据）→ 在数据库范围内（先尝试，失败会自动处理）

【第二步：分析并拆解查询】（仅当在数据库范围内时执行）

分析查询复杂度：
- 普通查询：单个聚合、简单筛选 → 输出1个子查询
- 复杂查询：多时间段对比、多维度分析 → 拆分为多个子查询

拆分原则：
- 每个子查询应该是独立的、可单独执行的SQL查询
- 明确依赖关系：如果子查询需要使用其他子查询的结果，在dependencies中标注
- 按批次执行：无依赖的可以并行执行

💡 聊天Agent - 配备search_tool（ReAct模式）

from langgraph.prebuilt import create_react_agent
from langchain_core.tools import tool
from duckduckgo_search import DDGS

@tool
def search_tool(query: str) -> str:
    """搜索网络获取最新信息"""
    with DDGS() as ddgs:
        results = list(ddgs.text(query, max_results=3))
    return format_results(results)

# 创建聊天Agent（配备search_tool）
chat_agent = create_react_agent(
    model=ChatOpenAI(model="deepseek-chat", base_url="https://api.deepseek.com"),
    tools=[search_tool],  # 给Agent配备工具
    state_modifier="""
    你是一个友好的助手，可以回答各种问题。
    
    当你需要最新信息、实时数据或不确定的信息时，可以使用search_tool搜索网络。
    
    自主决策示例：
    - "什么是机器学习？" → 不搜索（知识库有答案）
    - "今天北京天气？" → 调用search_tool
    - "最新AI新闻？" → 调用search_tool
    - "解释量子计算" → 不搜索（概念解释）
    - "SpaceX最新发射" → 调用search_tool（实时事件）
    """
)

# Agent自主决策，你只需要调用：
response = chat_agent.invoke({"messages": [user_query]})

⚠️ 完整错误处理策略

1️⃣ SQL生成/验证失败（在子图内）：

  • 单个子查询失败：
    • 子图内部最多重试3次
    • 超过3次 → 子图返回失败
  • 批次SQL生成失败：
    • 任何一个子查询的SQL生成失败 → 整个批次失败
    • 整个查询失败 → 降级到聊天Agent

2️⃣ SQL执行失败（🆕 区分错误类型）：

错误类型 具体情况 处理方式 原因
🔵 可恢复错误 • 表不存在
• 字段不存在
• 数据超出范围
• 语义错误（运行时才发现） 		✅ Fallback到聊天Agent 可能是诊断Agent判断错误
或SQL生成时信息不足
聊天Agent可以尝试其他方式回答
🔴 不可恢复错误 • 数据库连接失败
• 查询超时
• 权限不足（运行时）
• 资源不足 		❌ 返回系统错误信息
（不降级）		 这是系统/基础设施问题
聊天Agent无法解决
需要运维人员介入

3️⃣ 错误判断逻辑示例：

def classify_sql_execution_error(error):
    """分类SQL执行错误"""
    
    # 🔵 可恢复错误 - fallback到聊天Agent
    recoverable_errors = [
        "table does not exist",
        "column does not exist", 
        "no such table",
        "no such column",
        "unknown table",
        "unknown column",
        "out of range",
        "invalid date range"
    ]
    
    # 🔴 不可恢复错误 - 返回系统错误
    system_errors = [
        "connection refused",
        "connection timeout",
        "access denied",
        "permission denied",
        "out of memory",
        "too many connections",
        "database is locked",
        "execution timeout"
    ]
    
    error_message = str(error).lower()
    
    for pattern in recoverable_errors:
        if pattern in error_message:
            return "recoverable"  # Fallback到聊天Agent
    
    for pattern in system_errors:
        if pattern in error_message:
            return "system"  # 返回系统错误
    
    # 默认作为系统错误处理（保守策略）
    return "system"
