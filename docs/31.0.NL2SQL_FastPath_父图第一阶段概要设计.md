### 31.0 NL2SQL 父图（Fast Path）第一阶段概要设计

本设计覆盖父图左侧 Fast Path 的最小可用实现，并为后续复杂路径复用节点做好抽象。实现以 LangGraph v1.0.x 为基准，优先采用“独立 State + input/output schema”的官方推荐方式，避免不必要的 wrapper。

- **范围**：`Router → SQL生成子图 → SQL执行 → Summarizer → 返回结果`
- **复用性**：`SQL执行节点`与`Summarizer`在两条路径中共享；Fast Path 仅串行执行，但接口天然支持多 SQL 与并发，便于右侧复杂路径直接复用。
- **一致性**：为避免 wrapper，本父图将输入字段命名为 `query`，与“SQL生成子图”的 `SubState` 输入保持同名，以便直接用 `input=["query"]`。


### 31.1 父 State 设计

采用 TypedDict 风格并为列表/历史类字段使用 LangGraph v1 的 reducer（`add`）以便可在复杂路径中累积结果。父图不强制继承 `MessagesState`（减少无关状态），但 Summarizer 内部会临时构造提示词所需的消息。

```python
from typing import Annotated, Dict, List, Optional, TypedDict, Any, Literal
from langgraph.graph import add  # reducer

class SQLResult(TypedDict):
    sql: str
    columns: List[str]
    rows: List[List[Any]]
    row_count: int
    execution_ms: float
    error: Optional[str]

class ParentFastState(TypedDict):
    # ===== 输入与路由 =====
    query: str                                  # 与子图字段同名，避免 wrapper
    complexity: Optional[Literal["simple","complex"]]  # Router 输出

    # ===== SQL 生成输出（来自子图） =====
    validated_sql: Optional[str]                # 子图标准输出（同名直接映射）
    error: Optional[str]
    error_type: Optional[str]
    iteration_count: Optional[int]
    schema_context: Optional[Dict[str, Any]]
    validation_history: Annotated[List[Dict], add]     # 为复杂路径预留

    # ===== 统一执行输入/输出 =====
    sql_texts: Annotated[List[str], add]        # 统一入口（单/多 SQL 收敛为列表）
    sql_results: Annotated[List[SQLResult], add]

    # ===== 汇总与返回 =====
    summary: Optional[str]
    return_payload: Optional[Dict[str, Any]]    # API 返回的最终结构
```

设计要点：
- **字段对齐**：`query` 与子图一致，`validated_sql` 也与子图一致，减少粘合层代码。
- **多 SQL 兼容**：通过 `sql_texts` 统一承载单/多 SQL；Fast Path 中若只产出一个 `validated_sql`，在执行节点内转换为单元素列表并执行。
- **可共享**：`sql_results`、`summary` 为两条路径共用的汇总字段，右侧复杂路径直接沿用。


### 31.2 Router 节点

职责：判断是否走 Fast Path。第一阶段测试中仅选择会被判定为 `simple` 的问题。

接口约定：
- 输入：`query`
- 输出：`complexity: "simple" | "complex"`；`reason: Optional[str]`（日志用，不入 State，以减少状态污染）

路由：
- 当 `complexity == "simple"` → `sql_gen`
- 否则 → 预留到右侧路径（第一阶段可临时跳转到 `summarizer` 返回“暂不支持复杂问题”的友好提示，便于端到端联调）

伪代码（仅示意）：

```python
def router_node(state: ParentFastState) -> Dict:
    # 可实现为轻量 LLM 或规则；建议使用小模型，以毫秒级成本完成判定
    decision = "simple"  # 第一阶段：保证用例均为 simple
    return {"complexity": decision}

def route_by_complexity(state: ParentFastState) -> str:
    return "sql_gen" if state["complexity"] == "simple" else "summarizer"
```


### 31.3 调用“SQL生成子图”

- 子图状态与字段定义见《30.2.SQL 生成子图 State 定义.md》。子图关键输出为 `validated_sql`（成功标志）。
- 父图与子图通过 LangGraph v1 的 `input`/`output` 完成对接：
  - `input=["query"]`
  - `output=["validated_sql","error","error_type","iteration_count","schema_context","validation_history","execution_time"]`

说明：
- 父图字段名与子图保持一致，避免 wrapper；需要列表/历史字段时使用 reducer 聚合。
- Fast Path 仅依赖 `validated_sql` 即可继续；其余输出用于日志与可解释性（也便于后续 Summarizer 使用）。


### 31.4 SQL 执行节点（单/多 SQL，后续可并行）

职责：执行 `validated_sql` 或 `sql_texts` 中的多条 SQL，并规范化输出为 `sql_results: List[SQLResult]`。该节点在两条路径中复用。

输入：
- `validated_sql: Optional[str]`
- `sql_texts: Optional[List[str]]`
- 外部配置（通过 `config` 注入，不写入 State）：数据库连接、超时、并发度限制（Fast Path 默认为 1）、重试策略等。

输出：
- `sql_results: List[SQLResult]`（按输入 SQL 的顺序返回）

行为：
- 若存在 `validated_sql` 且 `sql_texts` 为空，则转为单元素列表执行。
- 第一阶段串行执行；为复杂路径预留 `max_concurrency` 参数（后续在复杂路径中开启并行）。

伪代码（仅示意）：

```python
def sql_exec_node(state: ParentFastState, config) -> Dict:
    sql_list = state.get("sql_texts") or []
    if not sql_list and state.get("validated_sql"):
        sql_list = [state["validated_sql"]]

    results: List[SQLResult] = []
    for sql in sql_list:  # Fast Path：串行；复杂路径可切换并发执行
        r = execute_sql(sql, config.db)  # 由基础设施层提供，返回统一结构
        results.append(r)

    return {"sql_results": results}
```


### 31.5 Summarizer（两路复用的统一总结节点）

职责：将 `query + sql_results (+ schema_context/validation_history 可选)` 生成面向业务用户的自然语言答案；失败时给出友好提示。

输入：
- `query`
- `sql_results`
- 可选：`schema_context`、`error`、`error_type`

输出：
- `summary`
- `return_payload`（建议结构：`{ query, sqls, results, summary, meta }`，便于 API 直接返回）

注意：
- 该节点同时服务于 Fast Path 与复杂路径；提示模版用能力检测的风格自适应单/多结果。


### 31.6 拓扑与连线（Fast Path）

```python
from langgraph.graph import StateGraph, END

graph = StateGraph(ParentFastState)

# 节点
graph.add_node("router", router_node)
graph.add_node("sql_gen", sql_generation_subgraph, input=["query"],
               output=["validated_sql", "error", "error_type",
                       "iteration_count", "schema_context", "validation_history"])
graph.add_node("sql_exec", sql_exec_node)      # 复用节点
graph.add_node("summarizer", summarizer_node)  # 复用节点

# 路由
graph.add_conditional_edges("router", route_by_complexity, {
    "sql_gen": "sql_gen",
    "summarizer": "summarizer"  # 第一阶段复杂问题临时回落到总结（提示“暂不支持复杂问题”）
})

# Fast Path 主干
graph.add_edge("sql_gen", "sql_exec")
graph.add_edge("sql_exec", "summarizer")
graph.add_edge("summarizer", END)

app = graph.compile()
```


### 31.7 接口契约与数据结构（对外）

- **执行输入**：
  - `query: str`
  - `config`（运行时注入）：`db`、`timeout_s`、`max_concurrency`（默认 1）、`enable_explain`（默认 false）等

- **执行输出（return_payload）**：
  - `query: str`
  - `sqls: List[str]`（从 `validated_sql`/`sql_texts` 归一化）
  - `results: List[SQLResult]`
  - `summary: str`
  - `meta: { iteration_count?, execution_time?, error?, error_type? }`


### 31.8 可扩展性与复用

- **SQL 执行节点**：已支持多 SQL 输入与并发参数，复杂路径可直接复用并开启并行；也可作为“Check 完成度”前的执行器。
- **Summarizer**：两路共享；在复杂路径中可多轮调用做阶段性/最终总结，提示词加入`中间结果`与`依赖图`。
- **状态隔离**：通过 input/output 显式映射，子图内部字段不会污染父图。
- **观测性**：保留 `validation_history` 与 `schema_context`（可选裁剪），便于后续 Debug 与质量分析。


### 31.9 测试建议（第一阶段）

- 仅选择 `simple` 用例：单表聚合、简单筛选、单 SQL 对比。
- 验证链路：`validated_sql` 非空 → `sql_results[0].rows` 有值 → `summary` 语义正确。
- 失败兜底：`validated_sql` 为空或执行报错时由 Summarizer 友好提示“问题超出快速路径能力或数据异常”。


### 31.10 与既有文档/子图的一致性说明

- 父图字段 `query` 与《30.2》子图输入 `query` 保持同名，`validated_sql` 同名映射，遵循《30.1》推荐的“独立 State + input/output schema”。
- 无需修改“SQL生成子图”的 State/接口；Fast Path 只消费其 `validated_sql` 与若干诊断字段。
- 为后续复杂路径预留：`sql_texts`、`sql_results`、`validation_history`、并发参数等，避免将来重构。
