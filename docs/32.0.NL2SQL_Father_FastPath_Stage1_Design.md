### 32.0 NL2SQL 父图 Fast Path 第一阶段概要设计
- 范围：对齐 docs/30.0 中左侧线路，覆盖节点 0~4（父 State、Router、SQL 生成子图、SQL 执行、Summarizer 返回）。
- 目标：在 LangGraph v1.0.x 下完成可运行的最小链路，同时为右侧复杂路径复用 SQL 执行与 Summarizer 节点做好抽象。
- 约束：第一阶段测试仅挑选简单问题，确保 Router 会将样例路由到 Fast Path；但设计需预留未来扩展（多 SQL、复杂路径复用、同 Summarizer 节点等）。

### 32.1 父 State 设计（节点0）
结合 docs/30.1.State 技术参考.md 的推荐做法，采用独立 TypedDict 状态，并通过 Annotated[..., add] reducer 聚合列表字段，避免子图污染父图。

```python
from typing import Annotated, Any, Dict, List, Literal, Optional, TypedDict
from langgraph.graph import add

class SQLResult(TypedDict):
    sql: str
    columns: List[str]
    rows: List[List[Any]]
    row_count: int
    execution_ms: float
    error: Optional[str]

class FatherFastPathState(TypedDict):
    # ===== 输入 & Router =====
    query: str
    query_id: str                           # 由父图生成的短 ID，传递给子图
    complexity: Optional[Literal["simple", "complex"]]
    router_model: Optional[str]
    router_confidence: Optional[float]
    router_reason: Optional[str]
    router_latency_ms: Optional[int]

    # ===== SQL 生成子图输出（字段名与子图保持一致，直接 input/output 接线） =====
    validated_sql: Optional[str]
    error: Optional[str]
    error_type: Optional[str]
    iteration_count: Optional[int]
    schema_context: Optional[Dict[str, Any]]
    validation_history: Annotated[List[Dict[str, Any]], add]
    execution_time: Optional[float]

    # ===== SQL 执行（单/多 SQL 统一入口，供双路径公用） =====
    sql_texts: Annotated[List[str], add]
    sql_results: Annotated[List[SQLResult], add]

    # ===== 汇总与返回 =====
    summary: Optional[str]
    return_payload: Optional[Dict[str, Any]]
```

设计要点：
- `query`、`query_id` 与 SQL 生成子图 SQLGenerationState 同名字段对齐，直接通过 `input=["query", "query_id"]` 传入。
- `sql_texts`/`sql_results` 统一收口，Fast Path 默认只有 `validated_sql`，但复杂路径可写入多条 SQL 或中间结果。
- Summarizer 只依赖 `query`、`sql_results`、`schema_context` 等公共字段，天然可被右侧路径复用。

### 32.2 节点职责总览
1. **Router（节点1）**：轻量 LLM/规则将问题二分类为 `simple` / `complex`，写入观测字段。
2. **SQL 生成子图（节点2）**：引用 docs/30.2 中现有子图；父图仅负责输入输出映射。
3. **SQL 执行（节点3）**：接收单条 `validated_sql` 或多条 `sql_texts`，执行并结构化结果；该节点在 Fast Path 与复杂路径中共享。
4. **Summarizer（节点4）**：统一总结 LLM，Fast Path 与复杂路径共用，用于汇总 SQL 结果并形成 `return_payload`。

### 32.3 Router 设计（节点1）
- **输入**：`query`
- **输出**：`complexity`（必填）、`router_model`、`router_confidence`、`router_reason`、`router_latency_ms`。
- **实现建议**：
  - 采用低成本模型（如 gpt-4o-mini / claude-haiku），提示仅允许输出 `simple` 或 `complex`；超时或解析失败默认判为 `complex`。
  - 使用 `time.perf_counter()` 记录节点耗时，便于监控第一阶段快速路径命中率。
  - 在 LangGraph 中通过 `graph.add_conditional_edges("router", route_by_complexity, {...})` 走向 Fast Path；第一阶段所有测试样例提前挑选为 `simple`，但逻辑仍保留 `complex` 分支。
- **失败处理**：当 `complexity != "simple"` 或 Router 异常时，暂时回退到 Summarizer，Summarizer 输出“暂不支持复杂问题”提示，避免流程中断。

### 32.4 SQL 生成子图集成（节点2）
父图直接挂载现有子图，并使用 LangGraph v1.0 的 input/output schema 完成字段映射：

```python
graph.add_node(
    "sql_gen",
    sql_generation_subgraph,
    input=["query", "query_id"],
    output=[
        "validated_sql", "error", "error_type",
        "iteration_count", "schema_context",
        "validation_history", "execution_time",
    ],
)
```

- 子图最终节点通过 `Command(update=...)` 返回字段，父图自动接收。
- 若未来需要注入解析提示，可在父 State 中新增 `parse_hints` 并与子图字段同名映射。

### 32.5 SQL 执行节点（节点3）
- **输入**：`validated_sql`、`sql_texts`、数据库运行配置（通过 `config` 注入，不写入 State）。
- **行为**：
  1. 构造执行列表：`sql_list = sql_texts or ([validated_sql] if validated_sql else [])`。
  2. Fast Path 串行执行；保留 `max_concurrency` 参数，复杂路径可开启并行执行或批量异步模式。
  3. 结构化返回 `SQLResult`，包含列名、行数据、耗时、错误信息。
  4. 写回 `sql_results`，供 Summarizer 和下游复用。
- **共享性**：节点实现不绑定 Fast Path，可直接被复杂路径多次调用（如执行多个分解 SQL）。
- **错误处理**：
  - 数据库错误时，将 `error` 写入对应 `SQLResult`，Summarizer 根据 `error` 生成友好提示。
  - 若 `sql_list` 为空，直接返回空结果并交由 Summarizer 告知“暂无可执行 SQL”。

### 32.6 Summarizer 节点（节点4，双路径共享）
- **输入**：`query`、`sql_results`、`schema_context`（可选）、`error`/`error_type`。
- **实现**：统一调用“Summarizer LLM”模板，通过 few-shot 提示生成自然语言回答；Fast Path 与复杂路径公用同一节点与提示。
- **输出**：
  - `summary`: 面向终端用户的描述。
  - `return_payload`: `{ "query": query, "sqls": [r.sql for r in sql_results], "results": sql_results, "summary": summary, "meta": {...} }`，供 API 返回。
- **兼容性**：当 Router 判定为 `complex` 或 SQL 执行失败时，Summarizer 根据 `error` 字段生成兜底文案，实现统一出参。

### 32.7 LangGraph 拓扑
```python
from langgraph.graph import END, StateGraph

graph = StateGraph(FatherFastPathState)

graph.add_node("router", router_node)
graph.add_node(
    "sql_gen",
    sql_generation_subgraph,
    input=["query", "query_id"],
    output=[
        "validated_sql", "error", "error_type",
        "iteration_count", "schema_context",
        "validation_history", "execution_time",
    ],
)
graph.add_node("sql_exec", sql_exec_node)
graph.add_node("summarizer", summarizer_node)


def route_by_complexity(state: FatherFastPathState) -> str:
    return "sql_gen" if state.get("complexity") == "simple" else "summarizer"


graph.add_conditional_edges(
    "router",
    route_by_complexity,
    {"sql_gen": "sql_gen", "summarizer": "summarizer"},
)

graph.add_edge("sql_gen", "sql_exec")
graph.add_edge("sql_exec", "summarizer")
graph.add_edge("summarizer", END)

app = graph.compile()
```
- 通过条件边保持 Fast Path 与未来复杂路径的一致结构；复杂路径接入后只需在 `route_by_complexity` 中新增分支即可。
- SQL 执行和 Summarizer 节点可被多个上游节点复用，LangGraph 会负责状态合并。

### 32.8 错误与回退策略
- **Router**：失败或超时 → `complexity="complex"`，直接进入 Summarizer，由 Summarizer 输出“当前仅支持简单问题”。
- **SQL 生成子图**：若 `validated_sql` 为空，将 `error`/`error_type` 写回；SQL 执行节点发现无 SQL 时返回空结果，Summarizer 告知失败原因。
- **SQL 执行**：数据库报错写入 `SQLResult.error`；Summarizer 将错误透传为易读文案。
- **Summarizer**：任何异常都返回结构化 `return_payload`，并在 `meta` 中标记 `status: "failed"`，保证接口稳定。

### 32.9 扩展与复用考虑
- SQL 执行节点暴露 `max_concurrency`、`retry`、`enable_explain` 等配置参数，为复杂路径批量执行做准备。
- Summarizer 模板包含 `path_type`（Fast / Complex）标志位，后续在复杂路径中可插入中间结果、计划说明，但复用同一 LLM。
- State 仅包含必要字段；右侧复杂路径新增字段（如 `plan_graph`、`intermediate_results`）时，可通过 `Annotated[..., add]` 扩展，不影响 Fast Path。
- LangGraph 1.0 的 `input`/`output` 映射、`Command(update=...)` 等机制确保父子图隔离，避免状态污染。

### 32.10 测试与验收要点（阶段性）
1. **路由准确性**：使用 5~10 个简单样例，验证 Router 输出均为 `simple` 且延迟 < 200ms。
2. **端到端**：对每个样例验证 `validated_sql` 非空、`sql_results[0].rows` 返回数据、`summary` 描述正确。
3. **兜底场景**：
   - 手动将 Router 输出改为 `complex`，确认流程直接跳至 Summarizer 并输出“暂不支持复杂问题”。
   - 模拟 SQL 执行异常，确认 Summarizer 告知错误并返回结构化 `return_payload`。
4. **共享节点验证**：通过单元测试直接调用 SQL 执行与 Summarizer，确保它们不依赖 Fast Path 特定字段，可为复杂路径重用。
