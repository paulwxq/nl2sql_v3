# 关联关系缺失的真正原因分析

## 用户质疑

用户指出：**YAML 配置中 `composite.exclude_semantic_roles` 并没有包含 `audit`，但为什么 audit 角色的列还是被排除了？**

## 真正的根本原因（已确认）

经过深入分析，**`audit` 角色过滤不是主要问题**。真正的根本原因是：

---

## 🔴 根本原因 1: 逻辑主键检测缺失关键组合

### equipment_config → maintenance_work_order 关系

**问题**：`maintenance_work_order` 的逻辑主键候选中**没有** `[equipment_id, config_version]` 组合

#### 验证结果：

```python
# maintenance_work_order 的 18 个逻辑主键候选
Total logical key candidates: 18

# 包含 equipment_id 或 config_version 的候选：
✅ ['wo_id', 'equipment_id']       - confidence: 0.88
✅ ['wo_id', 'config_version']     - confidence: 0.88
✅ ['equipment_id', 'downtime_minutes'] - confidence: 0.88
✅ ['fault_ts', 'equipment_id']    - confidence: 0.83

# 恰好 [equipment_id, config_version] 的候选：
❌ NOT FOUND  # ← 这就是问题所在！
```

**影响**：
- 源表 `equipment_config` 有逻辑主键候选：`[equipment_id, config_version]`
- 目标表 `maintenance_work_order` **没有**这个组合的逻辑主键候选
- 在 Stage 1 (特权模式) 中无法匹配（因为目标表约束列表中没有这个组合）
- Stage 2 (动态同名匹配) **被跳过**（因为源表是 `logical` 类型，不是 `physical`）

---

## 🔴 根本原因 2: Stage 2 动态同名匹配的限制

### 代码证据：

**文件**: `src/metaweave/core/relationships/candidate_generator.py:290-314`

```python
# Stage 2: 动态同名匹配（Dynamic Same-Name）
# ⚠️ 重要修改：只对源表物理约束（PK/UK）执行 Stage 2
# 原因：动态同名匹配是为源表物理约束设计的"兜底策略"
if combo_type == "physical":
    logger.debug(
        "[find_target_columns] Stage 2: 动态同名匹配（仅源表物理约束 PK/UK）"
    )

    matched = self._find_dynamic_same_name(
        source_columns,
        source_table,
        target_table,
        is_physical=True
    )
    # ...
else:
    logger.debug(
        "[find_target_columns] 跳过 Stage 2: combo_type=%s（只对源表 physical 执行）",
        combo_type
    )
```

**影响**：
- `equipment_config` 没有物理约束（没有 PK/UK），只有逻辑主键 `[equipment_id, config_version]`
- 因此 `combo_type = "logical"`
- **Stage 2 被跳过**，无法使用动态同名匹配来找到 `maintenance_work_order` 的 `[equipment_id, config_version]` 列

### 日志证据：


**关键日志解读**：
- 系统尝试在目标表的约束组合中匹配 `['equipment_id', 'config_version']`
- 但目标表的逻辑主键候选中**没有这个组合**
- Stage 2 被跳过，无法使用动态同名匹配
- 最终：**未找到满足阈值的匹配**

## 🔴 根本原因 3: fault_catalog 逻辑主键检测不准确（与之前分析一致）

**问题**：`fault_catalog` 的逻辑主键检测结果与业务键不符

| 维度 | 检测结果 | 真实业务键 |
|------|---------|-----------|
| 列组合 | `[product_line_code, fault_name]` | `[product_line_code, subsystem_code, fault_code]` |
| 列数 | 2列 | 3列 |

**结果**：无法匹配到 `maintenance_work_order` 的正确列组合

## ⚠️ 关于 `audit` 角色过滤的澄清

### YAML 配置（用户说的对）：


## 解决方案

### 方案 1: 修复逻辑主键检测（治本）

**问题根源**：逻辑主键检测器没有生成 `[equipment_id, config_version]` 组合

**解决方法**：手动添加到 `maintenance_work_order` 的逻辑主键候选

编辑 `output/metaweave/metadata/json/public.maintenance_work_order.json`:

```json
"candidate_primary_keys": [
  // ... 现有的 18 个候选 ...

  // 新增：真实的业务外键组合
  {
    "columns": ["equipment_id", "config_version"],
    "confidence_score": 0.88,
    "uniqueness": 1.0,
    "null_rate": 0.0,
    "comment": "手动添加：与 equipment_config 的关联键"
  }
]
```

**执行后重新生成关系**：
```bash
python -m src.metaweave.cli.main metadata --step relationships
```

---

### 方案 2: 扩展 Stage 2 到逻辑主键（代码修改）

**修改**: `src/metaweave/core/relationships/candidate_generator.py:290`

```python
# 当前逻辑（Line 290）
if combo_type == "physical":
    # 只对物理约束执行 Stage 2

# 改为
if combo_type in ["physical", "logical"]:  # ← 扩展到逻辑主键
    # 对物理约束和逻辑主键都执行 Stage 2
```

**优点**：
- 逻辑主键也可以使用动态同名匹配
- `equipment_config[equipment_id, config_version]` 可以匹配到 `maintenance_work_order` 的同名列

**缺点**：
- 可能引入更多候选（需要评估）
- 需要测试验证

---

### 方案 3: 优化逻辑主键检测算法（长期）

**问题**：逻辑主键检测器偏好选择更少的列，而不是业务上更合理的组合

**改进方向**：
1. 增加外键列组合的识别（通过列名模式识别，如 `*_id`, `*_code`, `*_version` 等）
2. 增加业务键评分（多个 identifier 列的组合）
3. 对于事实表，优先生成多维度列的组合

## 总结

### ✅ 用户质疑是正确的

- `composite.exclude_semantic_roles` 确实**不包含** `audit`
- `audit` 角色在复合键匹配时**不会被过滤**

### ❌ 真正的问题是

1. **逻辑主键检测缺失**：`maintenance_work_order` 没有生成 `[equipment_id, config_version]` 逻辑主键候选
2. **Stage 2 限制**：动态同名匹配只对物理约束执行，逻辑主键无法使用这个"兜底策略"
3. **fault_catalog 逻辑主键不准确**：检测结果与业务键不符

### 🎯 核心结论

**audit 角色过滤不是主要问题**。主要问题是：
- 逻辑主键检测算法没有生成正确的候选组合
- Stage 2 动态同名匹配对逻辑主键不起作用
