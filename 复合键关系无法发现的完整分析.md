## 三、根本原因分析

### 3.1 代码执行流程

#### Step 1: 收集源表组合（✅ 成功）
**代码位置**: `candidate_generator.py:161-207`
```python
def _collect_source_combinations(table, include_indexes=False):
    # 4. 逻辑主键（总是收集）
    for lk in logical_keys.get("candidate_primary_keys", []):
        if 2 <= len(lk_cols) <= max_columns and lk_conf >= 0.8:
            combinations.append({"columns": lk_cols, "type": "logical"})
```
**结果**: ✅ 收集到 `['product_line_code', 'subsystem_code', 'fault_code']`，类型为 `logical`

---

#### Step 2: 查找目标列（❌ 失败）
**代码位置**: `candidate_generator.py:209-316`

##### Stage 1: 特权模式（Privilege Mode）
**适用范围**: `combo_type in ["physical", "logical"]`
**匹配策略**: 源表约束 → 目标表约束

```python
# 收集目标表的约束组合
target_combinations = self._collect_source_combinations(
    target_table,
    include_indexes=include_target_indexes
)

# 遍历目标表的所有约束组合
for target_combo in target_combinations:
    if 匹配成功:
        return matched
```

**执行情况**:
- 源表: `fault_catalog.[product_line_code, subsystem_code, fault_code]` (type=logical, uniqueness=1.0)
- 目标表: `maintenance_work_order` 的逻辑主键候选列表中**没有**这个 3 列组合
  - ❌ 原因: 这 3 列在目标表中是**外键**，uniqueness < 1.0（一对多关系）
  - ❌ 逻辑主键检测器要求 uniqueness = 1.0，所以不会将其识别为逻辑主键候选

**结果**: ❌ Stage 1 匹配失败

---

##### Stage 2: 动态同名匹配（Dynamic Same-Name）
**代码位置**: `candidate_generator.py:285-314`

```python
# ⚠️ 重要修改：只对源表物理约束（PK/UK）执行 Stage 2
if combo_type == "physical":
    matched = self._find_dynamic_same_name(...)
    if matched:
        return matched
else:
    logger.debug(
        "[find_target_columns] 跳过 Stage 2: combo_type=%s（只对源表 physical 执行）",
        combo_type
    )
```

**执行情况**:
- 源表 combo_type = `"logical"` (不是 "physical")
- ❌ Stage 2 被跳过

**结果**: ❌ Stage 2 未执行

---

#### Step 3: 创建候选（❌ 失败）
**代码位置**: `candidate_generator.py:133-138`

```python
target_columns = self._find_target_columns(
    source_columns, source_table, target_table, combo_type
)

if not target_columns:
    continue  # ← 跳过此目标表
```

**结果**: ❌ target_columns = None，未创建任何候选

---

### 3.2 根本原因总结

**当前架构的匹配规则**:

| 源表约束类型 | Stage 1 (特权模式) | Stage 2 (动态同名) |
|-------------|-------------------|-------------------|
| **physical** (PK/UK) | ✅ 匹配目标表的 PK/UK/逻辑键/索引 | ✅ 精确同名 + 类型兼容 |
| **logical** (逻辑主键) | ✅ 匹配目标表的 PK/UK/逻辑键/索引 | ❌ **跳过** |

**本案例的关系特征**:
- 源表: `fault_catalog.[product_line_code, subsystem_code, fault_code]`
  - 类型: **逻辑主键** (uniqueness = 1.0)
- 目标表: `maintenance_work_order.[product_line_code, subsystem_code, fault_code]`
  - 类型: **外键** (uniqueness < 1.0，一对多关系)
  - **不是**逻辑主键候选（因为 uniqueness < 1.0）

**匹配失败原因**:
1. ❌ **Stage 1 失败**: 目标表没有这个组合的逻辑主键候选（因为是外键，uniqueness < 1.0）
2. ❌ **Stage 2 跳过**: 源表是 logical 类型，不执行 Stage 2

**架构限制**:
- **当前架构不支持** "逻辑主键 → 外键" 的关系发现
- 只支持 "逻辑主键 → 逻辑主键" 或 "物理约束 → 任意列" 的关系发现

## 四、为什么日志显示 "生成复合键候选: 0 个"

### 4.1 日志来源
**代码位置**: `candidate_generator.py:82-84`
```python
composite_candidates = self._generate_composite_candidates(tables)
candidates.extend(composite_candidates)
logger.info(f"生成复合键候选: {len(composite_candidates)} 个")
```

### 4.2 执行流程
```
1. _collect_source_combinations(fault_catalog)
   → 返回: [{'columns': [...], 'type': 'logical'}]  ✅ 1 个组合

2. 对于每个组合，遍历所有目标表:
   - target = maintenance_work_order
   - _find_target_columns(...)
     → Stage 1: 未找到匹配  ❌
     → Stage 2: 跳过  ❌
     → 返回: None  ❌

   - if not target_columns:
       continue  # 跳过，不创建候选

3. 最终 composite_candidates = []  ❌

4. logger.info(f"生成复合键候选: {len(composite_candidates)} 个")
   → "生成复合键候选: 0 个"  ❌
```

### 4.3 结论
- "收集到的组合" = 1 个 ✅ (源表层面)
- "生成的候选" = 0 个 ❌ (匹配失败，未创建候选)

## 七、解决方案

### 7.1 短期解决方案（配置层面）

#### 方案 A: 为 fault_catalog 添加物理主键约束
如果你可以修改数据库 DDL:

```sql
ALTER TABLE public.fault_catalog
ADD CONSTRAINT pk_fault_catalog
PRIMARY KEY (product_line_code, subsystem_code, fault_code);
```

**效果**:
- 源表组合类型变为 `"physical"` (而不是 "logical")
- ✅ Stage 1: 仍会失败（目标表没有对应约束）
- ✅ Stage 2: **会执行**（因为 combo_type = "physical"）
- ✅ Stage 2 会通过精确同名匹配找到目标列
- ✅ 关系会被发现

**优点**: 无需修改代码
**缺点**: 需要 DBA 权限修改数据库结构

---

#### 方案 B: 配置 target_sources（如果支持）
检查配置文件中是否支持:
```yaml
composite:
  target_sources:
    - composite_pk
    - composite_uk
    - composite_indexes
    - composite_dynamic_same_name  # ← 如果支持此选项
```

**注意**: 根据当前代码，target_sources 只控制 Stage 1 是否包含索引，不影响 Stage 2

---

### 7.2 中期解决方案（代码修改）

#### 修改点: 允许逻辑主键执行 Stage 2
**文件**: `src/metaweave/core/relationships/candidate_generator.py`
**位置**: 第 285-314 行

**当前代码**:
```python
# Stage 2: 动态同名匹配（Dynamic Same-Name）
# ⚠️ 重要修改：只对源表物理约束（PK/UK）执行 Stage 2
if combo_type == "physical":
    matched = self._find_dynamic_same_name(...)
else:
    logger.debug("跳过 Stage 2: combo_type=%s", combo_type)
```

**修改建议**:
```python
# Stage 2: 动态同名匹配（Dynamic Same-Name）
# 对物理约束和逻辑主键都执行 Stage 2
if combo_type in ["physical", "logical"]:
    is_physical = (combo_type == "physical")
    matched = self._find_dynamic_same_name(
        source_columns,
        source_table,
        target_table,
        is_physical=is_physical
    )
    if matched:
        logger.debug("[Stage 2 成功] %s -> %s", source_columns, matched)
        return matched
```

**影响评估**:
- ✅ 可以发现 "逻辑主键 → 外键" 关系
- ⚠️ 可能增加误匹配风险（逻辑主键的置信度低于物理约束）
- ⚠️ 需要充分测试，确保不会引入假阳性

---

### 7.3 长期解决方案（架构重构）

#### 方案: 引入 "Foreign Key Inference" 机制
在目标表层面推断潜在的外键组合:

1. **检测条件**:
   - 目标表有 N 列与源表逻辑主键完全同名
   - 这 N 列的类型与源表兼容
   - 这 N 列在目标表中的 uniqueness < 1.0（符合外键特征）

2. **推断规则**:
   - 如果目标表的组合 uniqueness 在 [0.1, 0.9] 范围内
   - 且源表是维度表（dim），目标表是事实表（fact）
   - 则推断为外键关系

3. **评分调整**:
   - 推断的外键关系评分应低于物理约束
   - 需要额外的数据验证（如采样检查参照完整性）

## 九、总结

### 9.1 问题本质
- **不是**数据质量问题 ✅
- **不是**逻辑主键检测问题（已通过配置修复）✅
- **是**关系发现架构的设计限制 ❌

### 9.2 关键发现
- 当前架构只支持 "逻辑主键 → 逻辑主键" 匹配
- 不支持 "逻辑主键 → 外键" 匹配
- 原因: Stage 2 (动态同名匹配) 只对 physical 类型执行，logical 类型被跳过

### 9.3 最简单的解决方案
**为维度表添加物理主键约束**，使组合类型从 "logical" 变为 "physical"，从而启用 Stage 2 动态同名匹配。
